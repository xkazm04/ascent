## Surface model notes (affordance → backing file:line)

**Governance gate (the "one bar" + drift crux)**
- Active policy text + CI snippet assembled by `buildGovernanceOverview` (`org/governance.ts:99-176`) from the **persisted** policy `getOrgGatePolicy(slug)` ?? `defaultGatePolicy("org")` (`governance.ts:104`).
- Owner edits via `GatePolicyEditor` (`GatePolicyEditor.tsx`) → `POST /api/org/gate-policy` → `setOrgGatePolicy` → `sanitizeGatePolicy` (`db/org-gate.ts:19,28`). Editor exposes: minLevel, minOverall, minDimension, **Security floor (D9≥50)**, forbid-ungoverned (`GatePolicyEditor.tsx:24-32`).
- Three consumers of that same persisted policy:
  1. Dashboard fleet eval — `evaluateGateLite` (`governance.ts:118`) over the rollup.
  2. App PR Check Run — `getOrgGatePolicy(owner)` → `evaluateGate(headReport, policy)` (`app/webhook/route.ts:224-225`).
  3. CI snippet / `action.yml` / Gate API — `policyFromParams` (`gate.ts:229-263`) from URL params.
- `evaluateGate` and `evaluateGateLite` share identical rules (`gate.ts:112-223`) — dashboard ↔ App check **agree**. Archetype-aware defaults (`gate.ts:58-68`: org=L3/dim≥40/no-ungoverned, team=L3/≥35, solo=L2/≥25). Fail-closed on unscored dims (`gate.ts:45-47`). **This is genuinely one policy across dashboard + App check.**
- **The drift:** `ciWith`/`gateQuery` (`governance.ts:81-97`) emit only `min_level/min_overall/min_dimension/no_ungoverned` — they **drop `minDimensionFor` (the D9 security floor)**. `action.yml` has no `min-security`/per-dimension input (`action.yml:20-51`), and `policyFromParams` only reconstructs D9 via a *separate* `?security=`/`?min_security=` param (`gate.ts:237-244`) that the generated snippet never sets. So a Security-floor policy enforced on the dashboard + App check is **silently absent from the copied CI snippet**. See F-1.

**Path to green (earned adoption)**
- `closestToGreen` worklist (`governance.ts:133-159`), rendered as deep links to the matching practice (`governance/page.tsx:150-160`, `practices#practice-<id>`). 1:1 D1–D9 → practice map (`practices.ts:17-123`, consumed `governance.ts:113`).
- Per-gap starter PR: `PracticeApply` preview + single + batch ("Roll out to the fleet") (`PracticeApply.tsx:78-116,179-244`) → `buildArtifact` (`practice-artifact.ts:102-311`).

**The `.ai/` standard + SKILL.md (senior-quality crux)**
- `buildFoundation` (`standard/index.ts:28-37`): manifest + doctor + CI wiring + maintain + memory + CONTEXT scaffold.
- Manifest is repo-shaped: `commandsFor(report.repo.primaryLanguage)` for real build/test/lint commands (`manifest.ts:30`, `practice-artifact.ts:49-63`), language→typecheck map (`manifest.ts:20-27`), `generatedFrom` = the real language manifest the doctor drift-checks (`manifest.ts:12-18,44`), purpose from `report.repo.description` (`manifest.ts:48`), `archetype` from the scan. **But** `neverTouch`, `secretsFrom`, `agents`, per-module CONTEXT, and `purpose` (when description is blank) ship as explicit **`TODO`s** for the running agent to fill (`manifest.ts:60-64`, `context.ts:38-67`).
- Doctor **proves** claims, not just asserts: runs capability commands under `--run` (`doctor.ts:80-86`), checks pre-push controls are actually wired into the real local hook (lefthook/husky/pre-commit, with command-alias matching) and FAILs if pre-push controls exist with no hook (`doctor.ts:88-107`), flags leftover TODOs (`doctor.ts:124`), drift, stale CONTEXT. Genuinely executable conformance.
- Control model is correct-by-construction: pre-push primary / CI thin backstop, stated in the manifest, the skill body, and the tracks matrix (`manifest.ts:69-72,138-141`; `skill.ts:115-132`; `tracks.ts:5-11,72-90`); one doctor script wired into both layers (`wiring.ts`).
- Guardrails forbid fabrication ("Never fabricate architecture, commands, or constraints you can't verify… Leave a clearly-marked TODO instead of inventing" — `skill.ts:272-281`).

**Scoring grounding (what the gate/score actually rests on)** — audited `source.ts` / `prompt.ts` / `engine.ts` / `recommendations.ts`:
- Ingestion is **curated, not thin**: ≤32 files (`source.ts:36-41`), deliberately the high-signal ones — AI-guidance files, ~40 manifests/configs, CI workflows, MCP configs, docs/ADRs, test + source samples (`source.ts:530-625`).
- Score is a **deterministic-signal × LLM blend with a ±guardband**, blend scaled by coverage (`engine.ts:71,99-102`); the LLM can nuance but not contradict the evidence, and a thin/truncated scan leans on the robust signal layer.
- **Provenance carried through**: every dimension keeps signalScore + llmScore + evidence strings, plus reconciliation warnings and an LLM-as-auditor discrepancy track (`engine.ts:40-42,104-156,192`); `contributions()` decomposes the headline into per-dimension weighted points (`engine.ts:396-419`). A leader can defend the number.
- **Soft spot:** on the **default mock path** (which the gate, `action.yml` `live:false`, the App check, and the dashboard all use), the roadmap prose comes from a static 9-template `CATALOG` (`recommendations.ts:20-120`), repo-personalized only by injecting the dimension's numeric score (`recommendations.ts:156`). The *number* is grounded; the *remediation prose on the path the gate uses* is a generic per-dimension template. See F-5.

## Character feedback (Priya, first person)

Okay — on paper, this is closer to shippable than anything I've evaluated since I killed our Backstage. The thing I came in skeptical about, I'm warm on: the gate is archetype-aware (an org held to L3/no-dim-below-40/no-ungoverned, a solo repo to L2 — `gate.ts:58-68`), so I'm not stuck arguing a flat global threshold per team. And it's the *same evaluator* behind the dashboard and the App PR check — one policy object, edited once. That's the drift-killer I always demand, and they actually built it.

Governance reads the way I'd want to present it upward: pass-rate, where the fleet fails, worst offenders, and — the part that matters most to my no-mandate-war strategy — a **cheapest path to green** where every failing dimension is a *link to the practice that fixes it*, plus a one-click and a fleet-batch starter PR. "Don't show me a red dashboard, show me the PR that turns it green" — they did. The roadmap is phrased as questions to explore, not orders. That's DX-Core-4-shaped friction-to-remove, not a stick. Good.

Now where I go quiet and start poking. **First, the drift I'd catch in five minutes.** The page literally says "no drift… enforce it everywhere," and for the level/overall/dimension/posture bars that's true. But the moment I tick the **Security floor** box, the dashboard and the App check enforce D9≥50 and the **CI snippet I copy does not** — `ciWith` drops it and `action.yml` has no input for it. That's two sources of truth wearing a one-source-of-truth label, on the exact surface I trust to *not* do that. That's the finding that keeps this at conditional. Fix it and the "one bar" claim is honest.

**Second — and this is my whole quarter — the `.ai/` standard.** It's genuinely good: capability-not-tool, pointers-not-embeds, a doctor that *runs the commands and FAILs if a pre-push control isn't wired into our real hook*, control placement pre-push-primary with CI as a thin backstop. That's a senior platform lead's reasoning, not a paste-anywhere template — when I can find it. But from /practices, where I'm *authoring the standard*, there's no link to it; it's hidden on an individual repo's report header. The artifact at the center of my job is the one I'd get lost looking for. And the generated manifest hands me real commands but leaves `neverTouch`/`secretsFrom`/`agents` as TODOs — honest (it won't fabricate, which I respect more than invention), and the skill is meant to be *run* so an agent fills them, but I can't fully sign off on "adapted to this repo" until I see the post-run result.

**Third, the soft underbelly I always assume a demo is hiding:** the score is credibly grounded — curated 32-file sample of the files that actually matter, a deterministic-signal blend with a guardband so the LLM can't run away, full provenance I could defend to my VP. That part I'd stake on. But the *remediation prose* on the default mock path the gate uses is a generic 9-template catalog with my score stamped on it. "Agent guidance is thin" reading identically across forty repos is exactly what my staff engineers will eat me alive over. The links are repo-specific; the words aren't, until a live scan runs.

Would I adopt it? On this read — yes, conditionally, and it would beat hand-rolling the doc plus a day-per-repo audit by a mile **if** L2 confirms the live `.ai/` standard and live roadmap prose are as repo-specific as the scaffold promises and the D9 drift is closed. Author once, see fleet conformance without re-reading repos, ship remediation as PRs not tickets — that's the collapse I need. I'm rooting for it, which is why the drift label and the generic mock prose sting: an honest "not yet" I can work with; a "no drift" that has drift, my staff engineers will find first.
