## 1. /api/history silently ignores the `limit` param — the 06-09 "aligned trend ranges" fix is inert, and CSV export silently truncates to 30 scans
- **Type**: bug
- **Severity**: High
- **Category**: silent-failure / data-inconsistency
- **File**: src/app/api/history/route.ts:84 (consumer: src/components/report/DimensionTrends.tsx:40-43; DB support: src/lib/db/scans-read.ts:117,125)
- **Scenario**: Commit `5cb8ded` fixed prior finding 06-09 #4 by making `DimensionTrends.loadDimensions` request `/api/history?repo=X&limit=${history.scans.length}` so the per-dimension charts match the overall series (server-fetched with `limit: 60`, trends/page.tsx:92). But the route's GET never reads `searchParams.get("limit")` — line 84 builds `{ orgSlug, includeDimensions }` only — so `getRepositoryHistory` falls back to its default `limit: 30` (scans-read.ts:125). Reproduce with any repo having >30 scans: /trends plots up to 60 overall points, the lazy dim fetch sends `&limit=60`, gets 30 back, and the "By dimension — N scans" header (DimensionTrends.tsx:139, derived from the overall series) overstates what the small-multiples actually drew — the exact mislabel the fix claimed to close. Worse: the CSV export (`?format=csv`, the "show my boss progress" artifact) flows through the same call and silently exports only the newest 30 rows of a possibly years-long history, with no truncation indicator in the file.
- **Root cause / Opportunity**: The fix was split across a client and a server file and only the client half landed (route.ts's recent commits — `bf2abea`, `23160d4` — touched ETag/CSV, never `limit`). The DB layer already supports and safely clamps `limit` to 1..200, so the route is the single missing link.
- **Impact**: Dimension trends silently desync from the overall chart for any actively-scanned repo (>30 scans — exactly the engaged users); the CSV a lead drops into a QBR deck is missing the oldest two-thirds of the L2→L4 story with zero signal that it's truncated.
- **Fix sketch**: In `route.ts` GET, parse `const limitRaw = Number(searchParams.get("limit"))` and pass `limit: Number.isFinite(limitRaw) ? limitRaw : (wantCsv ? 200 : undefined)` into `getRepositoryHistory` (the DB clamp at scans-read.ts:125 already bounds it 1..200 and coerces junk to 30; CSV defaults to the 200 cap so exports cover the full retained window). The ETag already folds `payload.scans.length`, so cached validators stay correct across different limits. ~5 LOC + a vitest on the param coercion if extracted.
- **Effort**: 1/10 · **Impact score**: 7/10

## 2. Free weekly scan slots are burned on failed scans, invalid URLs, and in-stream cache hits — the free funnel meters on attempt while the paid path meters on commit
- **Type**: bug
- **Severity**: High
- **Category**: edge-case / monetization-integrity (uncommitted WIP)
- **File**: src/app/api/scan/stream/route.ts:59-68 (also src/app/api/scan/route.ts:99-108)
- **Scenario**: The new weekly quota consumes a slot (`consumePublicScanQuota`) before any scan work. Everything downstream can then fail without delivering a report, with the slot already spent: (a) a nonexistent/typo'd repo — `scanRepository` throws `GitHubError NOT_FOUND` at stream/route.ts:173 → `error` frame; (b) an unparseable `url` — `parseRepoUrl` returns null at line 42 yet quota is still consumed at line 61 before `runScan(url)` throws `INVALID_URL` (the request is provably invalid *before* the slot is taken); (c) GitHub rate-limit/upstream 502 mid-ingest; (d) even an in-route **cache hit** — `lookupCachedScan` at line 112 can return `cached` (peek-miss → another scanner populates the cache in the race window, or a direct API caller who never peeked) and the "free" cached response still cost a slot. With the default anonymous limit of **3/week**, three typos lock a first-time visitor out of the conversion funnel for seven days having received nothing. Meanwhile the same function debits private prepaid credits only "after the scan actually produces real inference… a dedup or a degrade-to-mock run is free" (scan/route.ts:177-181) — and the repo's own harness convention says "Meter on commit, not on attempt" (wave 6). The free tier got the opposite policy.
- **Root cause / Opportunity**: The quota check was placed on the request path (correct for *deciding*) but also *consumes* there (wrong for *metering*); no refund path exists, and URL validity isn't checked before consumption in the stream route.
- **Impact**: Anonymous/new users — the top of the funnel the quota exists to nudge toward sign-up — get silently penalized for errors that cost the operator nothing. A 429 "come back next week" after three failed attempts is the single worst first impression the product can produce.
- **Fix sketch**: (1) In stream/route.ts, return 400 when `!parsed` *before* the quota block (mirrors the JSON route's `INVALID_URL` → 400). (2) Have `consumePublicScanQuota` include the recorded `hitAt` timestamp in its result, and add `refundPublicScanQuota(req, identity, hitAt)` in public-scan-quota.ts: read the row, remove one entry `=== hitAt`, write back (best-effort, fail-open — same posture as consume; add a pure `removeHit(prior, ts)` next to `decideQuota` with vitest cases in public-scan-quota.test.ts). (3) Call the refund in the stream route's catch (non-Abort errors that emit the `error` frame) and on the `lookup.cached` early-return; in scan/route.ts, refund before `handleError` maps a thrown scan failure. Degrade-to-mock still delivers a report, so it stays counted. ~100 LOC across three files; verifiable with vitest (pure window math) + tsc + next build.
- **Effort**: 4/10 · **Impact score**: 8/10

## 3. A non-OK /api/history response renders "Baseline established" — the histError flag only fires on network throw, so a 500 masquerades as first-scan success
- **Type**: bug
- **Severity**: Medium
- **Category**: silent-failure
- **File**: src/components/report/ReportView.tsx:41-47 (copy: src/components/report/ScoringTab.tsx:73-77)
- **Scenario**: `histError` exists precisely to distinguish "history fetch failed" from "no history yet" (ReportView.tsx:28-30) — but it is set only in the `catch`, which fires solely on a *rejected* fetch (offline). An HTTP error response takes the other path: `if (active && h.ok) setHistory(…)` does nothing when `h.ok` is false, `histError` stays false, `scans` stays `[]`, `trendPoints` collapses to the single live point, and ScoringTab confidently prints "Baseline established — re-scan later to track progress" for a repo that may have months of history. The trigger is not hypothetical: the harness's own deferred backlog notes the read-path `getPrisma()` sites (including `getRepositoryHistory`) "still 500 transiently on a DSQL token expiry" — every such transient turns a long trend history into a fake first-scan baseline, and the user's mental model of their data resets.
- **Root cause / Opportunity**: Asymmetric failure handling at one boundary: transport failure → flagged; HTTP failure → silently conflated with the empty-history success case. (401 on an auth-gated deploy and 503 on a DB-less one are *legitimate* "no trends available" modes and must keep the quiet path — which is probably why the blanket `!h.ok` was never flagged.)
- **Impact**: Misleading headline state on the report page's trend panel after any transient backend error; users can't tell a data-loss-looking event from a real baseline, eroding exactly the trust a maturity-tracking product sells.
- **Fix sketch**: In the ReportView effect, branch on status: `if (h.ok) setHistory(parseRepositoryHistory(await h.json())); else if (h.status !== 503 && h.status !== 401) setHistError(true);` (503 = persistence off → genuine no-history; 401 = signed-out viewer → optionally a follow-up "sign in to see trends" hint, but quiet is acceptable today). Apply the same status check to the sibling recommendations fetch only if desired (it degrades to the static roadmap, which is honest). ~5 LOC; verifiable with tsc + next build.
- **Effort**: 1/10 · **Impact score**: 5/10

## 4. Close the investigation loop: DimLine deep-links + GitHub commit jump from trend points (sanctioned RPT-2 deferral)
- **Type**: business
- **Severity**: Medium
- **Category**: retention / differentiation
- **File**: src/components/report/DimLine.tsx:19 (no href/router — confirmed dead-end), src/components/report/chartHover.tsx:90-135, src/components/report/DimensionTrends.tsx:93
- **Scenario**: The user job is *diagnosis*: "D2 Testing dropped 12 points between these two scans — what landed?" The overall TrendChart already deep-links each dot to its pinned report (`href` via `reportPermalink`, TrendChart.tsx:113-141) — but the per-dimension charts, where movements are actually *localized*, link nowhere: `DimLine` has no router, no `href`, and its tooltip omits the commit sha that `PointTooltip` already supports (chartHover.tsx:105-107). And nothing anywhere links out to the GitHub commit itself, though `headSha` rides on every `HistoryPoint` and the harness notes the URL is simply `https://github.com/{fullName}/commit/{headSha}`. This is the explicitly-deferred remainder of RPT-2 (feature-scout 2026-06-08 #2), still absent in current source.
- **Root cause / Opportunity**: All the data is already fetched and validated (`headSha` survives `parseRepositoryHistory`, validate.ts:132); only ~3 small wiring steps separate "chart as record" from "chart as investigation tool". Tools that let an engineer go from anomaly → exact commit diff in two clicks are what they return to weekly; that loop is the trends page's retention engine and a differentiator versus static scorecards.
- **Impact**: Turns the per-dimension grid — the page's main real estate — from read-only into the start of every "what changed" investigation; drives repeat visits (retention) and carries users onto pinned permalinks (the shareable, OG-imaged growth surface).
- **Fix sketch**: (1) Extend `ScanMeta` (DimLine.tsx:8) with `sha?: string`, populate it in DimensionTrends.tsx:93 from `s.headSha`, and pass `fullName` (or a prebuilt `href` via the existing `reportPermalink` from `@/lib/ui`) down to `DimLine`. (2) Mirror TrendChart's pattern in DimLine: `useRouter`, `activeHref` from the hovered point, `cursor: pointer` + `onClick` → pinned report; pass `sha`/`linked` into its `PointTooltip` (props already exist). (3) External commit jump: in both charts' `onClick`, open `https://github.com/{fullName}/commit/{sha}` on shift-click (`window.open(commitUrl, "_blank", "noopener")`) with a one-line tooltip hint ("shift-click for the GitHub commit") appended in `PointTooltip`. ~70 LOC, no new deps, stays inside the chartScale/tooltip conventions; verifiable with tsc + next build.
- **Effort**: 3/10 · **Impact score**: 6/10
