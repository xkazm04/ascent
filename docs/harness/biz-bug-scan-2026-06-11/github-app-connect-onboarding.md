> Total: 4 findings (2 bug / 2 business)

## 1. Installation-lifecycle webhooks have no redelivery-retry net — a transient failure permanently loses the install/uninstall
- **Type**: bug
- **Severity**: High
- **Category**: silent-failure
- **File**: src/app/api/app/webhook/route.ts:306 (mark-seen), :324-332 (created/unsuspend), :333-343 (deleted/suspend), :350-355 (installation_repositories)
- **Scenario**: A user installs the App. GitHub POSTs `installation` `created`. The handler marks the delivery seen at the top of `POST` (`deliveryAlreadySeen`, line 306), then runs `getInstallation(id)` + `upsertInstallation(...)` inside the inner try at 324-332. A transient failure there (GitHub 5xx / rate-limit, DB connection blip) is **swallowed by the `catch` with only a `console.warn`** — the installation is never persisted. GitHub redelivers the same `X-GitHub-Delivery` id; `deliveryAlreadySeen` now returns `true`, so the route 200s as a duplicate (line 307) and the redelivery is dropped. The org's installation is never recorded: `/connect` shows "no installation," `getInstallationIdForOwner` returns null, and every future private scan resolves to the public/tokenless path. The same gap loses a `deleted`/`suspend` teardown (org keeps minting tokens + running dead rescans) and an `installation_repositories` removal.
- **Root cause / Opportunity**: The 2026-06-09 fix added `forgetDelivery(id)` to release the dedup slot on failure, but wired it **only** into the deferred `runPrGate` (line 244) and `runPushRescan` (line 291) `after()` paths. The synchronous `installation` and `installation_repositories` branches — the most consequential events, since they establish/remove the org mapping — never receive `delivery` and never call `forgetDelivery`, so their transient failures are deduped into permanent loss. Dedup conflates "HTTP acknowledged" with "successfully processed."
- **Impact**: Silent activation failure (user installs the App but Ascent records nothing → zero private repos, broken connect page) and silent stale-install (uninstall lost → continued token mints / private access past revocation intent, perpetual 401 rescans). No self-heal — GitHub's redelivery, the only retry, is defeated.
- **Fix sketch**: In `POST`, thread the `delivery` id into the `installation` and `installation_repositories` branches. Call `forgetDelivery(delivery)` whenever their work throws/skips: in the created/unsuspend inner `catch` (332), in the deleted/suspend path if `removeInstallation` throws, and around the `await unwatchReposForInstallation(...)` fast-path (354). Equivalent: wrap each branch in try/catch that releases the slot on any failure. ~20 LOC, all in webhook/route.ts. Add a vitest case (no existing test for this route) asserting a thrown `upsertInstallation` leaves the delivery re-deliverable.
- **Effort**: 3/10 · **Impact score**: 8/10

## 2. No cost/quota context anywhere in connect or onboarding — private scans are billable but invisible at the moment of commitment
- **Type**: business
- **Severity**: High
- **Category**: monetization
- **File**: src/components/connect/InstallationRepos.tsx:159-173 (the "N of M watched" header); src/components/onboarding/importScan.ts:54-61 (auto `watch:true, schedule:"weekly"`)
- **Scenario**: Private scans are metered billable units (`consumeScanCredit`/`checkScanEntitlement` in the import route; `getUsageSummary` already computes period tokens + `estimatedCostUsd` + a per-repo top-10). Yet the connect panel and onboarding flow surface **zero** usage, quota, count, or cost. A user can flip every private repo to `daily` autoscan (RepoRow select), multiplying billable cron rescans, with no visibility into consumption or remaining allowance; onboarding silently enables `watch:true, schedule:"weekly"` for up to 10 repos. The user's job-to-be-done — "decide how much to scan without a bill surprise" — is unsupported precisely where the spend decision is made.
- **Root cause / Opportunity**: The whole metering backend exists (`GET /api/usage?org=` is live and `canReadOrg`-gated; `getUsageSummary(orgSlug, periodDays)`), but no connect/onboarding surface consumes it. This is a high-leverage in-product upsell location: usage anxiety appears exactly when the user is choosing cadence.
- **Impact**: Leaves the natural upgrade trigger ("you've used 80% of this month's private scans — upgrade") unbuilt, and risks churn from bill surprise on daily-cadence private fleets. A small header converts an invisible cost into a conversion/retention lever.
- **Fix sketch**: Add a compact usage strip to `InstallationRepos` (near line 163) fed by `GET /api/usage?org={org}` — "{used}/{quota} private scans this month · est ${cost}" with an upgrade link when over a threshold, and a one-line confirmation hint when a private repo is set to `daily`. The connect page already gates on `isDbConfigured`, so the data path is present. ~80-120 LOC (one client fetch + the strip + a hint in RepoRow). No new deps; verifiable with tsc + next build.
- **Effort**: 6/10 · **Impact score**: 8/10

## 3. "Add or manage repositories" and the empty-state both deep-link to the generic install page, not the installation's Configure page
- **Type**: business
- **Severity**: Medium
- **Category**: retention
- **File**: src/lib/github/app.ts:36 (`appInstallUrl`); src/app/connect/page.tsx:278-283; src/components/connect/InstallationRepos.tsx:151-157 (empty-state)
- **Scenario**: When an installation grants only a few (or zero) repos, the user's job is "grant Ascent the one repo I actually want." The connect header offers "+ Add or manage repositories on GitHub →" and the empty-state says "Adjust access on GitHub, then refresh" — but both route to `appInstallUrl()` = `.../installations/new` (the **install** page) rather than `https://github.com/settings/installations/{id}` (the per-installation **Configure** page where repos are added). The `installation_id` is already in hand (`InstallationRepos`'s `installationId` prop, the connect page's `inst.id`), so the precise destination is known but unused.
- **Root cause / Opportunity**: Only a generic install URL helper exists (`appInstallUrl`); there is no `appConfigureUrl(installationId)`. Selected-repo installs ("I don't see my repo") are the most common onboarding dead-end, and one extra manual GitHub-navigation step is exactly where users drop.
- **Impact**: Removes friction on the highest-drop selected-repo path — a one-click jump straight to the screen that grants the repo, lifting activation/repo-add completion. Tiny effort for a real funnel gain.
- **Fix sketch**: Add `appConfigureUrl(installationId)` next to `appInstallUrl` in app.ts (return `https://github.com/settings/installations/{id}`). Use it in the connect "Add or manage" link and the `InstallationRepos` empty-state whenever `installationId` is known, falling back to `appInstallUrl()` otherwise. ~20 LOC; verifiable with tsc + next build.
- **Effort**: 2/10 · **Impact score**: 5/10

## 4. `installation_repositories` fast-path unwatches repos from the payload's `repositories_removed` with no GitHub confirmation — asymmetric with the hardened delete/suspend path
- **Type**: bug
- **Severity**: Medium
- **Category**: webhook-integrity
- **File**: src/app/api/app/webhook/route.ts:350-355 (fast-path) vs :154-171/:338 (`confirmRevocationWithGitHub`-gated teardown); src/lib/db/installations.ts:95-110 (`unwatchReposForInstallation`)
- **Scenario**: A validly-signed but forged/misrouted `installation_repositories` delivery names a **victim's** installation id with `repositories_removed: [<the victim's actively-watched repos>]`. The handler immediately `await unwatchReposForInstallation(id, removed)` — setting `watched:false, scanSchedule:"off", nextScanAt:null` for those repos under the victim's org — with **no confirmation that GitHub actually revoked access**. The deferred `reconcileInstallationRepos` (line 361) re-lists from GitHub, sees those repos are still accessible, and does nothing to restore them (reconcile only ever *unwatches* stale repos; "Added repos are deliberately NOT auto-watched"). The forced unwatch is therefore permanent until the victim manually re-watches.
- **Root cause / Opportunity**: The prior Critical fix (github-app #1) established the convention that *destructive* webhook actions must be GitHub-confirmed — `confirmRevocationWithGitHub` gates `installation deleted/suspend`. But the equally-destructive `installation_repositories` removal teardown was left trusting the payload's delta array verbatim, an inconsistency under the same "secret-leak / cold-instance replay" threat model the team already accepts. (Distinct from the deferred github-app #4, which is the *opposite* direction — repos that should be unwatched but aren't; that reconcile is now implemented.)
- **Impact**: A single forged signed delivery silently disables a victim's watch + autoscan for named repos (targeted DoS on scan freshness), with no self-heal.
- **Fix sketch**: Drop the blind fast-path and rely solely on the GitHub-authoritative `reconcileInstallationRepos(id)` (it only unwatches repos GitHub confirms are gone) — the reconcile runs in the same request's `after()`, so legitimate quiescing is barely delayed. Or, if the latency matters, intersect `repositories_removed` against a fresh `listInstallationRepos(id)` before unwatching. ~10-15 LOC in webhook/route.ts; vitest-testable by asserting a removed-payload for a still-accessible repo does not unwatch it.
- **Effort**: 3/10 · **Impact score**: 5/10
