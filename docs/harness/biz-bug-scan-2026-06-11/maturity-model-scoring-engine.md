## 1. PROCESS SIGNALS block feeds the LLM 100×-inflated rates ("merge rate 8500%") on every tokened scan
- **Type**: bug
- **Severity**: High
- **Category**: silent-failure / unit-mismatch
- **File**: src/lib/scoring/prompt.ts:8 (definition), src/lib/scoring/prompt.ts:23-27 (all five usages)
- **Scenario**: Any scan with a token computes `PrStats` whose rates are **already 0..100 integers** (`pct = (n, d) => Math.round((n / d) * 100)`, pulls.ts:44; type contract "All rates are 0..100", types.ts:249). `buildAssessmentPrompt → processBlock` then renders them through a *second* scaler, `const pct = (n) => \`${Math.round(n * 100)}%\`` (prompt.ts:8), which assumes a 0..1 fraction. A repo with mergeRate 85, reviewedRate 92, smallPrRate 60, aiInvolvedRate 40, aiGovernedRate 67 is presented to the model as: "merge rate **8500%**, reviewed rate **9200%** … small-PR rate **6000%** … AI-involved rate **4000%**; of those, governed (reviewed) rate **6700%**." Reproduce: call `buildAssessmentPrompt` with any non-null `prStats` and inspect the PROCESS SIGNALS lines.
- **Root cause / Opportunity**: When wave 5 (feature-scout MAT-1) threaded `prStats` into the prompt, the new local `pct` helper was written for fractions while `PrStats` carries whole percentages — two helpers with the same name and opposite unit conventions in pulls.ts vs prompt.ts. Nothing validates the rendered text, so the absurdity ships silently on every tokened scan.
- **Impact**: The evidence block whose entire purpose is to anchor the LLM's D3/D6/D7/D8 judgment ("calibrate those dimensions to this too", prompt.ts:111) is nonsense on arrival. Best case the model silently ignores it (the feature's value is zero); worst case it skews dimension scores within the ±25 guardband and ~0.6 blend (up to ~15 points/dimension) or emits spurious "discrepancies" against detectors. Hours-to-merge/first-review lines sit beside the garbage and lose credibility too.
- **Fix sketch**: In src/lib/scoring/prompt.ts change the helper to render the value as-is (`const pct = (n: number) => \`${Math.round(n)}%\``) — `aiGovernedRate`'s null branch already handles "n/a". Add a vitest (src/lib/scoring/prompt.test.ts) that builds a prompt from a fixture `PrStats { mergeRate: 85, reviewedRate: 92, … }` and asserts the output contains "merge rate 85%" and not "8500%". Calibration-safe: the bench (docs/CALIBRATION.md) runs mock-mode and token-less, so `prStats` is null there — prompt text never reaches it.
- **Effort**: 1/10 · **Impact score**: 8/10

## 2. Recommendation carry-forward keys on the exact LLM title — a rephrased roadmap silently wipes backlog status, owner, and due date
- **Type**: bug
- **Severity**: High
- **Category**: silent-failure / data-loss
- **File**: src/lib/db/scans-persist.ts:142-153 (carry map), src/lib/db/scans-persist.ts:202 (lookup); same key in src/lib/report/compare.ts:90 (`recKey`)
- **Scenario**: A team tracks a recommendation via the backlog (PATCH /api/recommendations/[id]: status `in_progress`, assignee `@alice`, due date set). A new commit lands and the repo is re-scanned with a live LLM provider. The model re-emits the same gap with slightly different wording — e.g. "Agent guidance is thin — agents have little to go on **here**" vs "…to go on" — because the prompt explicitly asks for free-form invitational titles (prompt.ts:128-133) and the model never sees the previous scan's titles. `carry.get(\`${r.dimension}::${r.title}\`)` misses, so the new row is created `status: "open", assigneeLogin: null, targetDate: null` (scans-persist.ts:211-213). GET /api/recommendations returns only the latest scan's rows, so the tracked item — with its owner, due date, and in-progress state — vanishes from the backlog with no warning, no event, no audit row.
- **Root cause / Opportunity**: Identity is the raw generated string. The code comment admits the assumption ("stable for mock + low-temp LLM", scans-persist.ts:144), but stability is not guaranteed for any live provider (temperature is env-tunable via `LLM_TEMPERATURE`, evidence changes between commits change the wording, and provider failover — gemini→bedrock→openai — switches models mid-history). There is no fallback identity even though every item carries a stable `dimId` and each scan has at most ~6 roadmap rows.
- **Impact**: The recommendations product's core promise — "neither progress nor the backlog's planning state is lost on re-scan" — silently breaks exactly when customers use the paid path (live LLM + scheduled rescans). Lost assignments/due dates erode trust in the backlog as a system of record; `recsMovedToDone` in the scan diff (compare.ts:199) also under-reports completed work for the same reason.
- **Fix sketch**: Extract a pure matcher into src/lib/db/scans-shared.ts (e.g. `matchCarryForward(prev: {dimId,title,…}[], next: {dimension,title}[])`): tier 1 exact `dimId::title`; tier 2 normalized title (lowercase, collapse whitespace, strip punctuation — reuse compare.ts's `norm` approach); tier 3 unambiguous dimension match (exactly one unmatched prior row and one unmatched new item share a `dimId` → pair them, since a dimension's gap statement is the same gap restated). Use it at scans-persist.ts:202; mark tier-3 carries in the rec's event timeline if desired. Unit-test the matcher in vitest (exact, rephrased, two-items-same-dim ambiguity stays unmatched). ~60-90 LOC, no schema change, no score movement.
- **Effort**: 3/10 · **Impact score**: 8/10

## 3. `reviewedRate` fabricates "0% reviewed" when zero human-authored PRs merged, dragging D6 for bot/agent-run repos
- **Type**: bug
- **Severity**: Medium
- **Category**: edge-case (measured-zero vs. no-sample)
- **File**: src/lib/analyze/pulls.ts:128 (rate), src/lib/analyze/pulls.ts:162-179 (fold into D6)
- **Scenario**: A repo whose merged PRs in the 40-PR window are all bot-authored — a Renovate/Dependabot auto-merge fleet, or precisely the agent-run AI-native repo the product celebrates (PRs opened by Copilot/Devin/Claude as Bot authors) — has `mergedHuman === 0`, because bot merges are deliberately excluded from review-discipline accounting (pulls.ts:51, 89-92). `pct(reviewedHumanMerged, mergedHuman)` then returns **0** (the `d ? … : 0` branch, pulls.ts:44), not "no sample". `applyPrSignals` folds it with full force: `prRigor = clamp(0.5*0 + 0.3*smallPrRate + 0.2*(100-revertRate*6))`, and D6 becomes `0.65*signal + 0.35*prRigor` — e.g. a deterministic D6 of 80 with smallPrRate 70 drops to `0.65*80 + 0.35*35 ≈ 64`, a ~16-point penalty for review behavior that was never measurable. The evidence line renders "PR review coverage 0%" and the prompt repeats "reviewed rate 0%" to the LLM auditor — a stated falsehood it is told to calibrate to.
- **Root cause / Opportunity**: The same conflation class this codebase has already fixed twice ("couldn't measure ≠ measured zero": failed detectors, `aiGovernedRate`'s null-below-5 floor) survives in `reviewedRate`: the zero-denominator default of the `pct` helper turns an empty sample into a confident worst-possible measurement. `aiGovernedRate` got a null guard; `reviewedRate` — feeding a 50%-dominant term of `prRigor` — did not.
- **Impact**: Systematically penalizes D6 (rigor axis) for the most automated repos — near the posture threshold this misclassifies an AI-native fleet toward "Fast & Ungoverned", the exact judgment buyers screenshot. Also poisons the LLM evidence block and the persisted `prStats` shown on the org dashboard.
- **Fix sketch**: Make `reviewedRate: number | null` (types.ts): `mergedHuman > 0 ? pct(reviewedHumanMerged, mergedHuman) : null`. In `applyPrSignals` (pulls.ts:166-179) when null, drop the review term and renormalize `prRigor` over the remaining terms (or skip the D6 fold entirely, matching the D8 `aiGovernedRate == null` pattern) and label the evidence "no human-merged PRs in window". In prompt.ts render "n/a (no human-merged PRs)". Vitest: `summarizePullRequests` with all-bot merged nodes yields null; `applyPrSignals` leaves D6's signalScore unchanged. Calibration-safe: the bench is mock-mode and token-less, so `prStats` is always null there and bench scores cannot move. (Distinct from the DEFERRED maturity #5 triple-counting: this fixes the *value* being wrong, not how many stages consume it.)
- **Effort**: 2/10 · **Impact score**: 6/10

## 4. Quantify backlog items with engine-true ROI ("+6 pts · unlocks L3") instead of fuzzy impact/effort words
- **Type**: business
- **Severity**: Medium
- **Category**: retention / differentiation
- **File**: src/components/org/BacklogItemRow.tsx:56-58 (shows only `impact high · effort low`), src/lib/db/org-insights.ts:303 (org backlog query), src/app/api/recommendations/route.ts (per-repo list), with the math already in src/lib/scoring/engine.ts:229 (`projectDimensionClose`)
- **Scenario**: The job-to-be-done of the backlog (the org product's recurring-engagement surface: status, owner, due date, timeline) is *prioritization* — "which of these 14 items across 9 repos actually moves a level?" Today each row offers only the qualitative `impact high/medium/low` strings (LLM- or catalog-asserted), while the live single-repo report already proves the better answer exists: RoadmapPanel.tsx:75-82 computes "↑ up to +N pts · L2→L3" per item via `projectDimensionClose` — deterministic, glass-box, archetype-weighted. That quantification never reaches the persisted backlog, the org rollup, or the GET /api/recommendations payload, so the place where teams plan work is the place with the weakest signal.
- **Root cause / Opportunity**: All inputs are already persisted per scan (dimension scores + lens weights + archetype), and the projection math is a pure exported function. This is a wiring gap, not a feature build. Quantified, engine-derived ROI per backlog item is also the product's core differentiator made tangible: the score isn't an opinion, so the *roadmap* shouldn't be either — competitors' maturity assessments end at a static grade.
- **Impact**: Turns the backlog into a defensible prioritization tool ("closing D2 on repo X gains 6 points and crosses the L3 floor; the other 5 items gain 1-2") — the artifact an engineering manager pastes into planning, which is the retention loop for the org dashboard. Sorting by projected points also surfaces cross-repo leverage the per-repo report can't show.
- **Fix sketch**: Add a pure helper (src/lib/scoring/engine.ts or org-insights) `projectedGain(dims: {id,score}[], archetype, dimId): { points: number; unlocks: LevelId | null }` = `overallScoreFor(with dimId→100) − overallScoreFor(current)` + `levelForScore` band comparison — reusing the exact headline math, display-only (zero score movement, calibration-neutral). In `getOrgBacklog` (org-insights.ts:303) select each latest scan's `dimensions {dimId, score}` alongside its recommendations and stamp `projectedPoints`/`unlocks` on `BacklogItem`; same enrichment in the GET /api/recommendations handler via the scan it already resolves. Render a `+N pts · L3` chip in BacklogItemRow next to impact/effort and add a "by points" sort to BacklogPanel. Vitest the helper (gain matches `projectDimensionClose` on an assembled report; zero-weight dim → 0; already-100 dim → 0). ~120 LOC across 4 files, no schema change, no new deps.
- **Effort**: 3/10 · **Impact score**: 6/10
