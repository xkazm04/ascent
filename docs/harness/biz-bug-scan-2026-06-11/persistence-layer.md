## 1. `withDb`'s proactive refresh turns a transient IAM-mint hiccup into a failed write — while the current token is still valid
- **Type**: bug
- **Severity**: High
- **Category**: race-window / availability (token refresh)
- **File**: src/lib/db/client.ts:376 (withDb getClient; cold-start seed at client.ts:314-326)
- **Scenario**: Production DSQL. `persistScanReport` — deliberately the ONLY path routed through `withDb` — runs during the token's refresh margin (last 120s of a 900s TTL, when `tokenIsStale` is true but the token is still VALID). `getClient` does `await refresh(cfg)`; `mintDsqlToken` hits a transient STS/IAM throttle ("Rate exceeded") or a momentary signer/credentials blip; `refresh` rejects → `getClient` rejects → `runWithReconnect` re-throws it (a mint error is not an auth-expiry error, so the reconnect branch never engages). The scan was computed, the response says `x-ascent-persisted: false`, and a cron lane records the repo as failed and backs off 6h — even though the cached client could have completed the write on its still-valid token. Meanwhile every raw `getPrisma()` READ in the same window sails through (background refresh failure is only `console.error`'d). Cold start widens the window: the 06-09 fix seeds `expiresAt: 0`, so from boot until the FIRST successful mint, every `withDb` write hard-depends on minting succeeding right now, even though the deploy-time token in DATABASE_URL may be seconds old.
- **Root cause / Opportunity**: `getClient` treats a failed PROACTIVE refresh as fatal, but inside the margin the existing client is usable by definition, and the genuinely-dead-token case is already covered by the REACTIVE path (op throws auth-expiry → `reconnectDb`, which correctly propagates mint errors). The protected write path is therefore strictly *more* fragile than the unprotected read path during margin windows — the inverse of `withDb`'s contract ("the recommended entry point for production DSQL queries").
- **Impact**: Lost scan persists and spurious cron failure-backoffs exactly during IAM control-plane blips — the "2 AM outage" class the module header promises to prevent. One mint hiccup per 15-minute token cycle is enough; there is no retry/backoff on the proactive mint.
- **Fix sketch**: In `withDb`'s `getClient` (client.ts:376-379), wrap `await refresh(cfg)` in try/catch: on failure, if `g.__ascentPrisma` exists, `console.warn` and fall through to `getPrisma()` (the op then either succeeds on the still-valid token, or throws an auth-expiry that `runWithReconnect` recovers via `reconnectDb` — where a second mint failure is rightly fatal); re-throw only when no client exists at all. ~10 LOC. Add a `client.test.ts` case via the injectable `runWithReconnect` deps: getClient whose refresh fails while a working client is cached → op still runs once and succeeds.
- **Effort**: 2/10 · **Impact score**: 8/10

## 2. `consumeScanCredit` stamps `balanceAfter` from a pre-decrement read — concurrent debits write a credit ledger that can never reconcile
- **Type**: bug
- **Severity**: High
- **Category**: race-window / data-integrity (billing audit)
- **File**: src/lib/db/credits.ts:112 (stale read at credits.ts:101-104; conditional decrement at credits.ts:107-110)
- **Scenario**: Concurrent same-org debits are the NORM, not an edge case: `POST /api/org/scan` reserves a credit inside `mapPool(scanList, SCAN_CONCURRENCY=4, …)` (src/app/api/org/scan/route.ts:110-127), so 4 lanes call `consumeScanCredit(org)` simultaneously (cron/rescan and import do the same). On a static-Postgres deployment (READ COMMITTED, no OCC): each tx `findUnique`s `scanCredits = 10`; the atomic conditional decrements serialize correctly 10→9→8→7→6; but each tx then computes `balanceAfter = org.scanCredits - 1` from its own STALE read — four CreditLedger rows all stamped `balanceAfter: 9` while the real balance is 6. The returned `balance` (the `x-ascent-credits-remaining` header and SSE skip decisions) misreports identically. On DSQL the commit-time OCC conflict + `withRetry` re-run happens to mask it, so the corruption is exclusive to the plain-Postgres path — and silent.
- **Root cause / Opportunity**: The decrement itself is race-safe (`updateMany WHERE scanCredits > 0`), but the ledger stamp and return value are derived from the pre-decrement snapshot instead of the post-decrement row. `grantCredits` does NOT have the bug — its `update` returns the post-increment row — which makes the asymmetry easy to miss in review.
- **Impact**: schema.prisma:54 and docs/BILLING.md §Reconciliation define CreditLedger as "the trustworthy record behind Organization.scanCredits … reconciled against Stripe top-ups." Duplicate/wrong `balanceAfter` values corrupt exactly the column reconciliation walks, on every concurrent batch, and history can't be repaired after the fact. When Stripe lands, finance inherits a money trail that doesn't add up — the most expensive kind of silent failure for a prepaid-credits business.
- **Fix sketch**: After a successful decrement, re-read inside the same tx — `const after = await tx.organization.findUniqueOrThrow({ where: { id: org.id }, select: { scanCredits: true } })` — and stamp/return `after.scanCredits`. Under READ COMMITTED the post-update read sees this tx's own write (which was applied against the latest committed version because the row update serializes); under DSQL it's identical post-retry. ~6 LOC. Add a credits unit test with a mocked `$transaction` whose shared fake row is decremented by a simulated concurrent debit between the initial read and the re-read, asserting ledger stamps 9, 8 — not 9, 9.
- **Effort**: 2/10 · **Impact score**: 7/10

## 3. Running out of credits is a silent lifecycle event — no low-balance warning, no zero-crossing push, no digest line, so the fleet quietly stops updating at the exact moment the user is ready to pay
- **Type**: business
- **Severity**: High
- **Category**: monetization / retention
- **File**: src/lib/db/credits.ts:93 (every debit computes the crossing; no consumer acts on it); src/lib/scan-alerts.ts; src/lib/alerts.ts:201; src/app/api/cron/digest/route.ts
- **Scenario**: An org lead prepays N credits and schedules a watched fleet — the job-to-be-done is "my maturity trends keep flowing without me babysitting." Credits drain via the weekly cron with nobody in the war-room; the only zero-balance signals today are a `skipped: insufficient_credits` SSE event in a stream no one is watching and a 402 on the next MANUAL scan, possibly weeks later. The trend lines they paid for silently flatline (the churn trigger for a habit product), and the single highest-intent purchase moment a prepaid model has — "balance just hit zero, mid-useful-work" — passes with zero touch. docs/BILLING.md explicitly defers "automatic low-balance top-up" to future scope, but the ALERT half needs no Stripe and all of its infrastructure already exists: per-debit `balanceAfter` (credits.ts), a Slack-compatible sink (`dispatchAlert`, graceful no-op when unset), and a per-org weekly digest cron.
- **Root cause / Opportunity**: Prepaid models live on the refill loop; threshold and zero crossings are the product's only built-in upsell moments and its strongest churn guards. The 06-11 siblings fix how a skip *displays* (org-dashboard #1) and price *visibility at install* (github-app #2) — nothing fixes the *proactive* lifecycle push, which is what actually reaches a user who isn't looking at the app.
- **Impact**: Protects the data continuity retention depends on (a flatlined chart is the #1 reason to stop visiting), and manufactures the top-up conversion touchpoint the design-stage Stripe flow will monetize. Zero new dependencies; inert without `ALERT_WEBHOOK_URL`, matching the digest's clean no-op convention.
- **Fix sketch**: (1) Pure `buildLowCreditsMessage(org, balance, threshold)` in src/lib/alerts.ts beside `buildFleetDigestMessage`, unit-tested like its siblings. (2) `maybeAlertLowCredits(orgSlug, balanceAfter)` in src/lib/scan-alerts.ts (sibling of `checkAndAlertRegression`, never throws), called after each successful debit at the four debit sites — src/app/api/scan/route.ts:182, src/app/api/org/scan/route.ts:119, src/app/api/org/import/route.ts:163, src/app/api/cron/rescan/route.ts:105 — firing when `balanceAfter === CREDITS_ALERT_THRESHOLD` (env, default 5) or `=== 0`; debits are unit-sized, so each crossing fires exactly once with no dedupe table. (3) Append a "Credits remaining: N — top up" line to the weekly digest when the org is metered and balance ≤ 2× threshold. ~100 LOC + vitest for the builder and crossing logic.
- **Effort**: 3/10 · **Impact score**: 7/10

## 4. prisma/init.sql is six tables and two columns behind schema.prisma — the documented psql bootstrap builds a database that 500s the org dashboard
- **Type**: bug
- **Severity**: Medium
- **Category**: schema-drift / silent-failure (onboarding)
- **File**: prisma/init.sql:12 (Organization without `scanCredits`; whole file) vs prisma/schema.prisma:34, 82 and prisma/migrations/**
- **Scenario**: A self-hoster follows init.sql's own header ("Source of truth is prisma/schema.prisma; this file mirrors it for a plain `psql -f` bootstrap" — also referenced from docs/ARCHITECTURE.md:230). The resulting database is missing `Goal` and `Initiative` (present since the 0_init migration BASELINE — prisma/migrations/0_init/migration.sql:219,233), `CreditLedger`, `Playbook`, `PlaybookApplication`, `PublicScanQuota`, plus columns `Organization.scanCredits` and `User.githubLogin`. First visit to any org dashboard: `src/app/org/[slug]/page.tsx:97` awaits `listGoals(slug)` → Prisma P2021 "table does not exist" → 500, including for `/org/public`. Private orgs die one frame earlier in `layout.tsx:100` (`getCreditState` selects the missing `scanCredits`). Worse, the billing half fails SILENT: bulk-scan's `consumeScanCredit(...).catch(() => null)` reads the column error as "out of credits" and skips every repo with `insufficient_credits` — on an org that never had a credit system in its schema.
- **Root cause / Opportunity**: Three waves (billing/RBAC 20260608, playbooks 20260609 ×2, quota 20260610) — and the pre-baseline Goal/Initiative work — updated schema.prisma + migrations but never regenerated init.sql, even though the in-repo convention ("schema.prisma + init.sql" updated together) was followed by the earlier token-usage and scan-outcome waves, whose columns ARE present (init.sql:64-66, 145-147). The file now actively misdeclares itself as a mirror.
- **Impact**: A broken-by-construction first run on the documented manual bootstrap path — the worst onboarding bug shape: most of the app works, then the dashboard 500s and metering misbehaves in a way that looks like a billing problem, not a setup problem. Also a standing trap for the next contributor who trusts the header.
- **Fix sketch**: Regenerate with the file's own documented, OFFLINE command (no live DB needed): `npx prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script > prisma/init.sql`; re-apply the explanatory header and the public-org seed block (current init.sql:339-347); diff the output against the concatenated prisma/migrations/** to confirm parity. To make the drift impossible to repeat silently, add a small vitest that asserts every `model X` name in schema.prisma has a matching `CREATE TABLE "X"` in init.sql (pure file parse, runs in CI with no DB).
- **Effort**: 1/10 · **Impact score**: 6/10
