## 1. `providerAvailable("bedrock")` tests the wrong env var — a correctly-configured Bedrock deploy silently scores everything with mock
- **Type**: bug
- **Severity**: High
- **Category**: silent-failure
- **File**: src/lib/llm/index.ts:56-57 (the `bedrock` case), src/lib/llm/bedrock.ts:36-40 (region resolution)
- **Scenario**: An operator follows bedrock.ts's own header docs ("Override with BEDROCK_MODEL_ID / BEDROCK_REGION") and deploys with `LLM_PROVIDER=bedrock`, `BEDROCK_REGION=eu-central-1`, and `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY` — a completely working configuration (`BedrockProvider` reads `BEDROCK_REGION` first, and even falls back to `DEFAULT_BEDROCK_REGION`, so region is *never* actually a missing prerequisite). But `providerAvailable("bedrock")` checks only `AWS_REGION || AWS_DEFAULT_REGION`, returns false, and `getProvider()` pre-degrades every scan to `MockProvider` via `orMockIf`. Because the degrade happens in the *picker*, `intendedProvider === "mock"` in scan.ts, so `llmFailed` is never set: no "AI analysis was unavailable" report warning, no `fallback: true` SSE event — just a quiet "Demo · deterministic rubric" badge and a one-line server `console.warn`. The same false negative makes `providerByName("bedrock")` return null, so `LLM_FALLBACK_PROVIDER=bedrock` is silently never tried. (Credential-only setups relying on the `us-east-1` default, and ECS/EC2 role-credentialed hosts without `AWS_REGION`, hit the same wall.)
- **Root cause / Opportunity**: The W9-5 fix (commit `0f62121`, 2026-06-09) introduced the prerequisite pre-check but keyed Bedrock's availability on a variable that (a) is not the provider's documented config knob and (b) isn't a prerequisite at all — `BedrockProvider` always resolves a region. The check can't see `BEDROCK_REGION` or any AWS credential signal, so it produces false negatives for the exact deployments the enterprise provider exists for. (This is a new defect introduced *by* the W9-5 fix, not a re-flag of it: W9-5 fixed false positives — doomed attempts; this is the inverse — a healthy provider gated off.)
- **Impact**: The enterprise/privacy provider — the Phase-2 revenue path — silently delivers deterministic-floor reports on affected deploys. The customer configured and expects (and may be paying for) AI-audited scans; nothing in the report or `/usage` (0 tokens) says why they're not getting them. Trust-destroying in precisely the segment with the highest willingness to pay.
- **Fix sketch**: In `src/lib/llm/index.ts`:
  1. Broaden the `bedrock` case to any AWS-ish signal: `Boolean(BEDROCK_REGION || AWS_REGION || AWS_DEFAULT_REGION || AWS_ACCESS_KEY_ID || AWS_PROFILE || AWS_ROLE_ARN || AWS_WEB_IDENTITY_TOKEN_FILE || AWS_CONTAINER_CREDENTIALS_RELATIVE_URI)`.
  2. For the *picker* path (`getProvider`, where the operator explicitly chose `LLM_PROVIDER=bedrock`), mirror the claude-cli precedent: trust the explicit selection (return the provider) — a genuinely broken config still fails fast at `assess()` and the existing retry → failover → mock chain handles it with the honest `llmFailed` warning. Keep the env check only for the failover skip in `providerByName`.
  Add vitest cases with `vi.stubEnv` covering `BEDROCK_REGION`-only and credential-only configs (the false negatives) plus the all-unset case. ~10-15 LOC + tests; verifiable with tsc + vitest + next build.
- **Effort**: 1/10 · **Impact score**: 8/10

## 2. Ship a built-in per-model price table so the `/usage` cost panel works out-of-the-box (and prices mixed-provider fleets correctly)
- **Type**: business
- **Severity**: High
- **Category**: monetization
- **File**: src/lib/llm/config.ts (new table), src/lib/db/usage.ts:126-131 (`estimateLlmCostUsd` call site), src/app/usage/page.tsx:165-169
- **Scenario**: The job-to-be-done is "show me what my fleet's AI scanning costs" — the COGS visibility that wave 1 built token metering for and that the deferred Stripe/plan-quota work (PERS-2) will bill against. Today `estimatedCostUsd` is null unless the operator hand-sets BOTH `LLM_INPUT_COST_PER_MTOK` and `LLM_OUTPUT_COST_PER_MTOK`, so every default deployment renders "— · set LLM_*_COST_PER_MTOK to estimate". The flagship monetization surface ships dark. Worse, a single global rate can't be *correct* once `LLM_FALLBACK_PROVIDER` exists: one 30-day window can legitimately contain Gemini Flash scans (~cents/MTok) and Claude Sonnet scans (~dollars/MTok) — `engineModel` is already persisted per Scan row (scans-persist.ts:174), but the aggregate prices all tokens at one number.
- **Root cause / Opportunity**: Every input for accurate per-model pricing already exists (per-scan `engineModel`, `inputTokens`, `outputTokens`); only the rate lookup is missing. A small static table keyed by model-id prefix (`gemini-3-flash`, `anthropic.claude-sonnet-4-6` — matching the `us.`/`eu.`/`global.` geo prefixes, `gpt-4o-mini`) turns the existing panel into a real spend dashboard with zero configuration, which is the prerequisite for "you've used $X this month — upgrade" messaging and usage-based pricing.
- **Impact**: Activates the cost/billing funnel for 100% of deployments instead of the few that configure rates; correct attribution when failover mixes models; gives the operator the per-org COGS number that pricing the SaaS tiers depends on. Clear retention lever (operators check a live spend number; they don't check a "—").
- **Fix sketch**: In `src/lib/llm/config.ts`, add `MODEL_PRICES: { prefix: string; inPerMTok: number; outPerMTok: number }[]` (rates for the three default models, sourced from current price sheets, with an "approximate, env overrides win" doc comment) and a pure `priceForModel(model: string)` longest-prefix lookup. In `src/lib/db/usage.ts`, replace the single `_sum` token aggregate with a Prisma `groupBy(['engineModel'])` over the same where-clause and sum `Σ priceForModel(model) × tokens`; keep the env vars as a global override (precedence: env > table > null) so existing configured deploys are unchanged. Update the `/usage` sub-copy to say "built-in rates" vs "configured rates". Unit-test `priceForModel` + the fold (pure functions — no DB, no live LLM). ~80-110 LOC total.
- **Effort**: 4/10 · **Impact score**: 7/10

## 3. Bedrock is the only provider with no per-call timeout — one hung Converse call eats the entire 90s LLM budget, structurally disabling retry and failover for the enterprise path
- **Type**: bug
- **Severity**: Medium
- **Category**: race-window / recovery-gap
- **File**: src/lib/llm/bedrock.ts:43-77 (no timer), vs src/lib/llm/gemini.ts:35-42 and src/lib/llm/openai.ts:38-41 (the established pattern); interplay at src/lib/scan.ts:236-261
- **Scenario**: Gemini and OpenAI abort themselves at `LLM_TIMEOUT_MS` (60s) and claude-cli at 150s, so when the primary hangs, scan.ts's resilience plan still has ~30s of its 90s `LLM_TOTAL_BUDGET_MS` left to run the retry (and failover steps abort fast on the spent budget). `BedrockProvider.assess` passes only `{ abortSignal: opts.signal }` — no per-call timer — and `BedrockRuntimeClient`'s default node handler has no request timeout. So when Bedrock hangs (throttled endpoint, cross-region inference profile blip), the first attempt runs until the scan-wide deadline aborts it at 90s; the loop then hits `if (llmDeadline.signal.aborted) break` and goes straight to mock. Net effect: for `LLM_PROVIDER=bedrock`, the retry step and a configured `LLM_FALLBACK_PROVIDER` can *never* execute on a hang — the resilience plan (W9-1/LLM-2) exists but is unreachable, and the hung request also stays open (billing) for 90s instead of 60s.
- **Root cause / Opportunity**: The W6-2 timeout-AbortController pattern was applied to gemini, and its tracked follow-up closed openai — but Bedrock never had a per-call timeout to fix, so it was skipped by both passes. The scan-wide budget masks the gap (no 500s, mock still arrives), which is why it survived: the failure mode is "silently worse degradation," not a crash. (Distinct from the tracked openai follow-up — different file, and the consequence here is failover starvation under the budget design, not a missing mirror.)
- **Impact**: Paying enterprise scans degrade to the deterministic floor on any single Bedrock hang, even when the operator explicitly configured a fallback model to prevent exactly that; plus ~30s of extra wasted wall-clock and open-socket billing per incident.
- **Fix sketch**: Mirror gemini.ts in `BedrockProvider.assess`: create a `timeoutCtrl = new AbortController()`, `setTimeout(() => timeoutCtrl.abort(new Error("Bedrock request timed out.")), LLM_TIMEOUT_MS)` (read `Number(process.env.LLM_TIMEOUT_MS) || 60_000` like the siblings), combine with `opts.signal` via `AbortSignal.any`, pass as `abortSignal` to `client.send`, `clearTimeout` in `finally`. ~12 LOC. Verify with a vitest that `vi.mock`s `@aws-sdk/client-bedrock-runtime` with a never-resolving `send` + fake timers and asserts the abort reason — no live AWS call.
- **Effort**: 2/10 · **Impact score**: 6/10

## 4. `providerByName("gemini")` returns a MockProvider when keyless — the mock then masquerades as a *successful* failover and suppresses the "AI analysis was unavailable" warning
- **Type**: bug
- **Severity**: Medium
- **Category**: silent-failure
- **File**: src/lib/llm/index.ts:104-106, consumed at src/lib/scan.ts:243-292
- **Scenario**: `LLM_PROVIDER=bedrock` + `LLM_FALLBACK_PROVIDER=gemini`, but no `GEMINI_API_KEY` in this environment (key rotated out, or staging copied prod's `LLM_FALLBACK_PROVIDER` without the key). `providerByName("gemini")` returns `geminiOrMock()` → a `MockProvider`, not null. scan.ts pushes it as a real plan step (`fallback.name !== intendedProvider` — "mock" !== "bedrock"). When Bedrock fails twice, the mock "failover" runs and *succeeds* (`attemptAssess` skips the usability gate for `p.name === "mock"`), so `resolved` is set and the `llmFailed` branch never executes: no `report.warnings` entry ("AI analysis was unavailable, so scores reflect detected signals only…"), no `fallback: true` SSE event, no `console.error` diagnostic. The route's mock-degrade cache guard still works (it keys off `engine.provider === "mock"`), and the header still shows the "Demo" badge — but the explicit honesty warning the degrade path exists to deliver, and the operator's log signal, are silently lost.
- **Root cause / Opportunity**: `providerByName`'s own contract (its doc comment, index.ts:100-101) says it "returns null for 'mock'/unknown/empty: those mean no real fallback, and the caller degrades to MockProvider itself" — but the `gemini` case predates `providerAvailable` and short-circuits through `geminiOrMock()`, whose keyless branch *is* mock. The other three cases were converted to the `providerAvailable(...) ? provider : null` shape in W9-5; gemini was left inconsistent (the inline comment even acknowledges it: "already returns mock without a key").
- **Impact**: In the misconfigured-fallback scenario, every degraded scan renders deterministic-floor scores without the standard caveat warning, and operators lose the `[scan] LLM provider failed…` log line they'd grep for — the exact "degradation must be flagged" invariant the rest of scan.ts enforces.
- **Fix sketch**: In `src/lib/llm/index.ts`, make the gemini case match its siblings: `case "gemini": return providerAvailable("gemini") ? geminiOrMock() : null;` (1 line — `geminiOrMock` then always constructs the real provider since the key is present). Add a vitest with `vi.stubEnv` asserting `providerByName("gemini")` is null without a key and a `GeminiProvider` with one. Verifiable with tsc + vitest + next build.
- **Effort**: 1/10 · **Impact score**: 4/10
