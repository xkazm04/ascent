## 1. Weekly quota slot is burned on scans that deliver nothing — typos, upstream failures, aborts, mock-degrades, and stream-path cache hits all consume one of 3 anonymous slots
- **Type**: bug
- **Severity**: High
- **Category**: silent-failure / edge-case (billing-fairness)
- **File**: src/app/api/scan/stream/route.ts:59-68 (consume) vs :112-121 (cache hit) and :173-185 (error unwind); src/app/api/scan/route.ts:99-108 (consume) vs :210-227 (handleError — no refund)
- **Scenario**: `consumePublicScanQuota` records the hit BEFORE the scan's outcome is known, and there is no refund path anywhere (grep: none). Concretely, each of these burns one of the default **3** weekly anonymous slots while delivering zero (or already-free) value: (a) a typo'd repo — `owner/reop` passes `normalizeRepo`/`parseRepoUrl`, scan throws `NOT_FOUND`; (b) `EMPTY` repo, GitHub `RATE_LIMITED`, or `UPSTREAM` 502; (c) an unparseable `url` POSTed to either route (`INVALID_URL` is thrown by `scanRepository` *after* the quota line); (d) the user refreshes / navigates away mid-scan (AbortError — scan intentionally cancelled, report never delivered); (e) an LLM outage degrade-to-mock — the user gets the deterministic floor, yet pays a slot; (f) on the STREAM route only, quota is consumed before `lookupCachedScan` runs inside `start()`, so a direct API caller (or a peek-miss race where another scan just populated the cache, or two tabs whose second `coalesceScan` join shares one computation) is charged for a cache hit the JSON route explicitly serves for free. Three mishaps = anonymous funnel locked out for 7 days.
- **Root cause / Opportunity**: Consume-then-run with no compensating action. The same file pair already states the correct policy twice and violates it: route.ts:99 — "Only real, public scans count: a cache hit / peek above already returned for free"; route.ts:112/177 — credit metering charges only when `report.engine.provider !== "mock"` ("a dedup or a degrade-to-mock run is free"). The free tier got neither rule.
- **Impact**: With limit=3/week, the most common first-touch failure modes (typos, mid-scan refresh, a Gemini blip) eat the entire free allowance of the product's primary acquisition funnel and present a week-long `QuotaBlocked` wall — to users who never received a single successful scan. Conversion-killing and invisible in monitoring (the scan "failed normally").
- **Fix sketch**: In `src/lib/public-scan-quota.ts` add `refundPublicScanQuota(req, identity)`: recompute the bucket key exactly as `consumePublicScanQuota` does (`u:`/`ip:` + `hashKey`), read the row, drop the single newest timestamp, write back — best-effort, fail-open, wrapped in the existing `withDb`/`withRetry`, plus a pure exported `removeNewestHit(hits: number[]): number[]` for vitest (sibling of `decideQuota` in public-scan-quota.test.ts). Wire it: (1) both routes — remember `quotaCharged` (already tracked as `quotaScope !== null`), wrap the `coalesceScan/runScan` + report-handling block in try/catch and refund on `GitHubError`/`AbortError` before rethrowing; (2) stream route — refund when `lookup.cached` returns (the JSON route's ordering needs no change); (3) both routes — refund when `degradedToMock` (mirrors the credit rule). Headers already emitted stay correct enough (soft gate); document the 1-slot staleness. ~70 LOC + ~30 LOC tests; tsc + vitest + next build verifiable.
- **Effort**: 3/10 · **Impact score**: 8/10

- **Severity**: High
- **Category**: monetization / retention (trust + sign-up conversion)
- **File**: src/app/page.tsx:243-248 (Public pricing card: "Unlimited public-repo scans", "No signup. Free forever…"), :62-68 (hero: "Free for public repos · No signup"); src/lib/public-scan-quota.ts:27-41 (the real numbers)
- **Scenario**: The in-flight weekly quota (3 anonymous / 20 signed-in, enforced with a 429 + `QuotaBlocked` wall) ships against a landing page whose pricing card literally says "Unlimited public-repo scans" and "Free forever". A user who trusted the headline discovers the real terms only at the moment of denial — the single worst place to learn the marketing was false. Meanwhile the elevated signed-in tier (the first rung of the conversion funnel the quota system was built to create) is advertised NOWHERE on the landing page; it's discoverable only after exhaustion, inside `QuotaBlocked`/`QuotaBanner`.

The repo already has the convention for this ("user-facing counts derive from the source array" — layout.tsx derives `DIMENSIONS.length`); the quota limits are env-derived functions a server component can call directly.
- **Impact**: Today: a trust break at peak intent, and a free sign-up lever (3 → 20 scans, ~7x) left invisible until users are already annoyed. Fixed: the pricing card becomes the top-of-funnel pitch for accounts — the exact monetization story (free taste → free account → prepaid credits) the quota and Supabase work exist to serve.

Free for public repositories.", and soften the hero microcopy "Free for public repos" → keep (still true) but drop nothing else; optionally add a small "Sign in for {N}/week" line under the ScanForm. Render the numeric features only when the gate can enforce (`isDbConfigured() && !publicScanQuotaDisabled()`), else keep today's copy — so a DB-less/local deploy doesn't advertise a gate that never trips. ~25 LOC, no new components; tsc + next build verifiable.
- **Effort**: 2/10 · **Impact score**: 7/10

## 3. Quota read-modify-write is not transactional — the claimed DSQL OCC serialization can never fire, so a trivially parallel client bypasses the weekly gate entirely
- **Type**: bug
- **Severity**: Medium
- **Category**: race-window
- **File**: src/lib/public-scan-quota.ts:147-179 (findUnique → decide → upsert as separate statements; the :128-130 doc-comment claims they "serialize cleanly on DSQL's optimistic concurrency")
- **Scenario**: `consumePublicScanQuota` runs `db.publicScanQuota.findUnique(...)` and `db.publicScanQuota.upsert(...)` as two independent implicit transactions. N concurrent requests from the same IP all read the same row state, all pass `decideQuota` (each sees the pre-existing hits only), and the last upsert wins — at most ONE new hit is recorded per concurrent batch. A script firing requests in parallel under the burst limiter's ceiling (20/min/IP) gets ~20 LLM scans per minute indefinitely while its weekly bucket barely accumulates: the long-horizon cost gate — built precisely to stop "a cheap script… grazing indefinitely" (its own header comment) — is a near-no-op against parallelism. Benign concurrency (two tabs) undercounts too, which currently masks finding #1's over-charging.
- **Root cause / Opportunity**: The comment assumes DSQL's OCC will abort one of two conflicting *transactions*, but conflict detection needs both the read and the write inside ONE transaction. As written, neither DSQL nor vanilla Postgres (default READ COMMITTED) ever raises a serialization error, so the carefully-built `withRetry`/`isSerializationConflictError` machinery around it never triggers — classic dead resilience primitive (same class as the wave-3 "withDb was on no call path" learning).
- **Impact**: The quota's only adversary-relevant guarantee is fictional; real LLM spend leaks scale with attacker parallelism. Also makes any future per-user paid quota built on this table silently lossy.
- **Fix sketch**: In `consumePublicScanQuota`, wrap read+decide+write in an interactive transaction so the existing retry loop becomes live: `withDb((db) => withRetry(() => db.$transaction(async (tx) => { const row = await tx.publicScanQuota.findUnique({ where: { ipHash } }); …decideQuota…; await tx.publicScanQuota.upsert(…); return result; }, { isolationLevel: "Serializable" }), { label: "public-scan-quota" }))`. On Postgres, Serializable makes the concurrent read-write dependency abort one txn (retried by `withRetry`); on DSQL pass no isolationLevel (snapshot OCC already aborts on the write-write conflict against the same `ipHash` row) — gate the option on `readDsqlConfig()` or just omit it and accept write-write detection on both engines (sufficient here: both racers upsert the same row). ~12 LOC; existing pure tests unchanged; tsc + vitest + next build verifiable (no migration — same schema).
- **Effort**: 2/10 · **Impact score**: 6/10

## 4. Quota-blocked users are shown a dead-end wall even when a persisted report of the repo already exists — serve the last saved scan instead of nothing
- **Type**: business
- **Severity**: Medium
- **Category**: retention / differentiation
- **File**: src/components/report/ReportClient.tsx:120-125 (429 → `QuotaBlocked` dead end); src/app/api/scan/route.ts:67-81 (head-pinned peek); src/lib/db/scans-read.ts:592-616 (`getScanReportByCommit` already returns the LATEST scan when `headSha` is omitted)
- **Scenario**: The peek fast-path is head-pinned: the moment a repo gets one new commit, the peek 204-misses and the client opens a stream — which can 429 on `weekly_quota`. The user then gets the `QuotaBlocked` empty-state ("come back on Jun 18") even though the product may hold a perfectly useful report of that repo from yesterday in the DB/gallery. The job-to-be-done — "show me this repo's maturity" — is 95% answerable with data already paid for, but the current flow discards it at the exact moment of peak interest. (Popular gallery/chip repos are the most likely to have a saved scan AND the most likely destinations for a user's last free slots.)
- **Root cause / Opportunity**: The cache/persistence identity is per-commit (correct for freshness), but the denial path has no "any-commit" fallback read. The rebuild helper for exactly this (`getScanReportByCommit(owner, name, {})` → most recent scan, full report) already exists and powers permalinks — it just isn't reachable from the quota-denied flow.
- **Impact**: Turns a hard bounce (likely permanent for an anonymous visitor) into a served report + a soft upsell. The stale report still carries the sign-in CTA and the "limit resets…" message, so the monetization nudge survives while the user keeps getting value — strictly better retention at zero LLM/GitHub cost (one DB read).
- **Fix sketch**: (1) `src/app/api/scan/route.ts`: support `peek=1&latest=1` — when the head-pinned lookup misses, fall back to `getScanReportByCommit(parsed.owner, parsed.repo, {})` (public org only, and only when `!token`); return it with `x-ascent-stale: "true"` so the client knows it's not head-fresh. (2) `src/components/report/ReportClient.tsx`: in the 429 `weekly_quota` branch, fire that fallback peek; on a 200, render the report with the blocked message as a banner instead of the full-screen wall — add a `notice` variant to `QuotaBanner` (src/components/report/QuotaNotice.tsx) carrying "Showing the last saved scan from {report.scannedAt} — your free weekly limit is used; resets {formatResetAt}. Sign in for more." On a 204, keep today's `QuotaBlocked`. Reuses `parseScanReport` validation and the existing repo-identity check. ~80-100 LOC total; tsc + vitest (peek-handler unit on the new branch is optional) + next build verifiable.
- **Effort**: 3/10 · **Impact score**: 6/10
