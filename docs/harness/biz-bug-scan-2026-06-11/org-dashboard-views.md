## 1. Credit-skipped repos poison the Live war-room with NaN and read as clean success in OrgScanButton
- **Type**: bug
- **Severity**: High
- **Category**: silent-failure / race-window
- **File**: src/components/org/LiveWarRoom.tsx:84-99 (NaN fold), src/components/org/LiveWarRoom.tsx:134-143 and src/components/org/OrgScanButton.tsx:40-50 (dropped events); server contract: src/app/api/org/scan/route.ts:101, 122, 164
- **Scenario**: The bulk-scan stream emits THREE shapes the clients don't handle. (a) Mid-run credit exhaustion — two concurrent batches (the exact race the route's atomic `consumeScanCredit` reservation defends against, route.ts:113-127) or a per-repo scan draining the balance — makes the server send `repo` events of the shape `{ repo, skipped: "insufficient_credits" }`: no `error`, no scores. LiveWarRoom's `onRepo` only branches on `d.error`, so it falls through to `Number(d.overall)` → **NaN**, builds a `LiveRepo{overall:NaN, adoption:NaN, rigor:NaN}` and **overwrites the repo's real seeded standing** in state. `stats` filters with `r.overall != null` (NaN passes), so the headline tiles literally render "NaN" (AnimatedStat tweens `from + (NaN-from)*eased` → `String(NaN)`), the leaderboard sort comparator goes NaN-unstable, and the ticker shows a NaN delta. (b) Up-front partial coverage — balance 4, 10 watched — slices `scanList` and sends `notice {skipped: 6}` (route.ts:101); neither client handles `notice`, so the user who clicked "Scan all watched (10)" watches it finish at 4/4 with zero indication 6 paid scans never ran. (c) The final `result {skippedForCredits}` (route.ts:164) is also dropped by both. The route's own comment claims "the war-room shows scanned N · M skipped (out of credits)" — that UI does not exist.
- **Root cause**: The SSE consumers were written against the happy-path event vocabulary (`progress`/`repo+error`/`error`) before the prepaid-credit system added `skipped`, `notice`, and `result`; the producer's contract grew and neither consumer was updated. `onRepo` additionally trusts `d.overall` to be numeric with no `Number.isFinite` guard — the exact "NaN at an untrusted boundary" class the 06-08 wave closed elsewhere (vScale, parseRepositoryHistory).
- **Impact**: A paying org that hits its credit ceiling sees either a corrupted war-room (NaN org maturity on the wall it might literally project on a TV) or a silently truncated "successful" fleet scan — the two surfaces most tied to the money. Users can't tell they need a top-up, which also suppresses the upsell moment.
- **Fix sketch**: In `LiveWarRoom.onRepo`, branch on `d.skipped` (ticker-only entry, like `d.error`) before the score fold, and guard the fold with `Number.isFinite(overall)`; add `notice`/`result` handlers in both `LiveWarRoom.launch` and `OrgScanButton.run` accumulating a `skipped` count into state; render it as a warn line ("M skipped — out of scan credits") next to the existing `failed` line in OrgScanButton and in `WarRoomHeader`; give `Mover` a `skipped?: boolean` and a muted label branch in `MoversTicker` (liveWarRoomShared.ts + LiveWarRoomPanels.tsx). Extract the repo-event classification (`error | skipped | scored(finite)`) as a pure helper and unit-test it in vitest. ~80 LOC across 4-5 files; tsc + vitest + next build.
- **Effort**: 3/10 · **Impact score**: 8/10

## 2. Period-over-period deltas compare different repo cohorts — onboarding repos fabricates fleet "movement"
- **Type**: bug
- **Severity**: High
- **Category**: edge-case / wrong-numbers
- **File**: src/lib/db/org-rollup.ts:204-238; rendered at src/app/org/[slug]/page.tsx:131-153 (tile deltas) and src/components/org/PeriodSummary.tsx:28-35, 67-71 (the banner)
- **Scenario**: `getOrgRollup` computes `avgOverall` over ALL currently-scanned repos, but `baseline` over only the repos that had a scan at-or-before the window start, then `deltas = current − baseline` (org-rollup.ts:233-237). Concrete: an org has 3 mature repos averaging 80 before the quarter; during the quarter it onboards 5 new repos whose first scans average 40. Current avg = (3·80 + 5·40)/8 = 55 → the overview banner announces "Fleet maturity slipped −25 to 55 (from 80)" with a big orange ▼−25 "net maturity" badge — while the movers panel directly below it correctly shows zero regressions (no repo's own score dropped; single-scan repos are skipped by `getOrgMovers`). The dashboard contradicts itself, and the error is largest exactly when the org is doing the GOOD thing (expanding coverage). The inverse also holds: onboarding a few strong repos manufactures a fake "climbed +N".
- **Root cause**: The minuend and subtrahend are averages over different populations. Composition change (new repos entering the scanned set mid-window) is folded into what is presented as score movement; nothing reconciles the two cohorts.
- **Impact**: The headline exec read — the period-in-review banner and all three tile deltas, the first thing a fleet owner screenshots into a leadership deck — reports movement no repo experienced. Erodes trust in every other number on the page once someone cross-checks against movers.
- **Fix sketch**: In `getOrgRollup`, the baseline query already returns `repoId`s; build `baselineIds: Set<string>` and compute the current-side delta inputs over only scanned rows whose repo id ∈ baselineIds (baseline ⊆ current always, since a scan ≤ start implies a latest scan exists). Keep the fleet-wide `avgOverall/avgAdoption/avgRigor` for the tiles' main values; only `deltas` becomes cohort-matched. Extract `computeWindowDeltas(current: Snap[], baseline: Snap[])` as a pure function and unit-test the onboarding scenario in src/lib/db/org.test.ts. Optionally pass `newRepos = scannedCount − baseline.repos` to `PeriodSummary` and append "· N repos onboarded this period" so growth is reported as growth. ~40 LOC + test.
- **Effort**: 3/10 · **Impact score**: 7/10

## 3. Per-row "Rescan" on the repositories leaderboard — the backend has been waiting since ORGD-3
- **Type**: business
- **Severity**: High
- **Category**: retention / monetization
- **File**: src/app/org/[slug]/repositories/page.tsx:70-109 (leaderboard rows — Autoscan select but no rescan control); API ready at src/app/api/org/scan/route.ts:38-41
- **Scenario**: The user job: "I just merged the fix for this repo's gap — show me the score move NOW." Today the only in-dashboard option is "Scan all watched", which burns one prepaid credit per watched repo (CreditsControl economy) and minutes of wall-clock for one repo's answer; the alternative is leaving the org dashboard for /report. The scoped API shipped in wave 3 — `POST /api/org/scan` accepts `repos:[fullName]` and its comment literally anticipates "a single repo's 'Rescan' from the leaderboard" — but no UI ever calls it. This is the tracked, never-done ORGD-3 follow-up, confirmed still absent in current source. The "⚠ scan failed" chip makes it worse: the dashboard tells you a repo needs a retry and then offers no way to retry it.
- **Root cause / Opportunity**: The fix→rescan→score-moves loop is THE retention loop of a maturity product — it's the moment the tool proves causality between work and score. Making it one click, one credit, in-place (a) multiplies scan frequency per seat, (b) makes prepaid credits feel fair (pay for the repo you care about, not the fleet), and (c) gives the failed-scan chip its missing call-to-action. Zero new backend, zero new dependencies.
- **Impact**: Higher session frequency and credit consumption with higher perceived fairness — the engagement metric that converts watchers into habitual users; also closes the dead-end UX on failed scans.
- **Fix sketch**: New client component `src/components/org/RepoRescanButton.tsx` (~70 LOC), modeled on OrgScanButton: `POST /api/org/scan {org, repos:[fullName]}`, consume the stream with `readSSE`, spinner while running, surface `repo.error`/`repo.skipped` (per finding #1's vocabulary) as a title/tooltip, `router.refresh()` on completion. Mount it in the leaderboard row next to `ScheduleSelect` (repositories/page.tsx) — rendered only for `r.watched` rows since the route scopes to `listWatchedRepos` (an unwatched fullName would silently no-op). Disabled state + hint mirrors ScheduleSelect's convention. ~90 LOC total; tsc + next build, plus a vitest test if the SSE-summary fold is extracted as a pure helper.
- **Effort**: 3/10 · **Impact score**: 7/10

## 4. The overview tab — the org landing page — serializes 7 independent queries
- **Type**: bug
- **Severity**: Medium
- **Category**: performance / latency
- **File**: src/app/org/[slug]/page.tsx:69, 74, 93-97
- **Scenario**: Every render of `/org/[slug]` (force-dynamic, so every tab switch back to Overview) runs `listSegments` → `getOrgRollup` → `getOrgMovers` → `getOrgRecommendations` → `getOrgBenchmark` → `getOrgGapAnalysis` → `listGoals` strictly in sequence, each an independent await; each helper itself issues 2-3 round trips (org lookup + data). On Aurora DSQL with realistic 20-40 ms RTTs that's ~15 serialized round trips — hundreds of ms of pure waterfall added to TTFB on the single most-visited org view, stacked on top of the layout's own sequential `getOrgRollup` + `getCreditState` (layout.tsx:87, 100). The sibling tabs already do this right — practices/page.tsx:13-18, plan/page.tsx:24-30, delivery/page.tsx:31 all use `Promise.all` — the landing tab is the one that drifted.
- **Root cause**: Only `segmentId` (from `listSegments`) is an input to four of the calls; nothing else depends on a prior result, but the awaits were written sequentially as sections were added one feature at a time.
- **Impact**: The first impression of the whole org product is its slowest page for no structural reason; under DSQL token-refresh latency or cross-region RTT the gap is user-visible on every visit and every `router.refresh()` after a scan.
- **Fix sketch**: In `page.tsx`, keep `listSegments` first (it validates `?segment=`), then fetch the rest concurrently: `const [rollup, movers, orgRecs, benchmark, gaps, goals] = await Promise.all([getOrgRollup(slug, win, segmentId), getOrgMovers(slug, win, segmentId), getOrgRecommendations(slug, 5, segmentId), getOrgBenchmark(slug), getOrgGapAnalysis(slug, segmentId), listGoals(slug)])`, leaving the `!rollup` empty-state branch after the join. Optionally parallelize layout.tsx's `getOrgRollup`/`getCreditState` pair the same way. ~12 LOC, no behavior change; tsc + next build (all calls already independent and null-safe).
- **Effort**: 1/10 · **Impact score**: 5/10
