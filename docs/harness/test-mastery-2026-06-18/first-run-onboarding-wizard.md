# Test Mastery — First-Run Onboarding Wizard

Context note: the entire context has **zero tests** (no `.test.ts(x)` under `src/components/onboarding/`, none referencing onboarding). The repo has **no jsdom / React Testing Library / component-render harness** — every test is a plain `.test.ts` exercising pure functions in a Node environment (confirmed: 0 `.test.tsx` files exist). So the highest-leverage move is to test the *pure, extractable* decision logic that the components wrap — the money gate, the SSE parser, the cost-disclosure contract — not rendered DOM. Findings below are ordered by business blast radius.

## 1. Pin the mock-vs-real "money gate" so onboarding can't silently bill (or silently fabricate scores)
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/components/onboarding/OnboardingFlow.tsx:255
- **Scenario**: A refactor flips one term of `canRunReal = !!sourceInstallId && !!credit && credit.org === sourceLabel && (credit.unlimited || credit.balance > 0)`. Two silent regressions, both shippable today with green CI: (a) the `credit.org === sourceLabel` guard is dropped/inverted — a *stale* credit object from a previously-picked org now satisfies the gate, so the wizard sends `mock:false` and the import route (`/api/org/import`) draws **real prepaid credits against the wrong tenant's balance**; (b) the guard tightens incorrectly (e.g. `balance >= 1` becomes `balance > 1`, or `unlimited` handling breaks) so a paying org silently gets a **preview/mock scan presented as if it were a real scan**, and `setPreviewScan(!canRunReal)` even mislabels the disclosure banner.
- **Root cause**: The gate is an inline boolean buried in `startScan`, never extracted, never tested. It is the single decision that determines whether money is spent and whether scores are real vs synthetic — exactly the kind of expression a reviewer skims over.
- **Impact**: Cross-tenant credit spend (chargebacks / trust), or fabricated maturity scores presented as live numbers (the product's core deliverable). Both are invisible until a customer notices.
- **Fix sketch**: Extract a pure `canRunRealScan({ sourceInstallId, credit, sourceLabel }): boolean` (and have `startScan` call it). Table-test the invariants: returns **false** when `sourceInstallId` is null (public funnel is always preview); **false** when `credit.org !== sourceLabel` (stale-org credit must NOT enable billing); **false** when `balance === 0 && !unlimited`; **true** only when installId present AND credit matches the current org AND (`unlimited` OR `balance > 0`). Assert that for a false result the request carries `mock:true` and `previewScan` is set true. This is an LLM-generatable batch over a tiny truth table that locks a money/data-integrity invariant.

## 2. Test the SSE import parser for FAILURE, stall, and abort — not just the happy stream
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/components/onboarding/importScan.ts:42
- **Scenario**: `runImportScan` is the only code that turns the import route's raw byte stream into UI state, and it has no test. Regressions that ship green: the SSE frame splitter (`buffer.indexOf("\n\n")`, `event:`/`data:` prefix parsing, CRLF trimming at :95) mishandles a chunk boundary so `repo`/`result`/`error` events are dropped — the wizard hangs on "Scanning…" forever; the stall watchdog (`armStall`, `STALL_MS`) stops re-arming on progress and aborts a *healthy* long scan; or the catch-block disambiguation at :122-125 mislabels outcomes — a genuine server failure gets reported as "Scan canceled" (`aborted` vs `stalled` vs `message`), so the user never learns the scan actually failed and never retries.
- **Root cause**: The function is async and reads a `ReadableStream`, so it *looks* hard to test — but it takes an injectable `AbortController` and `fetch` is globally mockable. Nobody wrote the harness, so the entire streaming contract (the activation moment of the product) is unverified.
- **Invariant the test must assert**: a `result` event resolves `{ok:true}` and fires `onResult` exactly once; a mid-stream `error` event invokes `onError(message)`; an event split across two `read()` chunks is still parsed once; when the controller is aborted the result is `{ok:false, aborted:true}`; when no chunk arrives within `STALL_MS` the result is `{ok:false, aborted:true, stalled:true}`; a non-2xx response surfaces the route's `error` body in `message`.
- **Impact**: A broken parser strands every new user on the first scan — the highest-value activation step — with no error and no recovery path; mislabeled outcomes erode trust ("it said canceled but I didn't cancel").
- **Fix sketch**: Add `importScan.test.ts`. Mock global `fetch` to return a `Response` whose `body` is a hand-built `ReadableStream` of `Uint8Array` SSE frames (encode `event: repo\ndata: {...}\n\n`). Use `vi.useFakeTimers()` to advance past `STALL_MS` for the stall case and assert `controller.abort()` was called and `stalled:true`. Deliberately split one frame across two `enqueue`s to pin the chunk-boundary buffering.

## 3. Lock the watch-schedule cost-disclosure contract: the POST body and the disclosed copy must never drift
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/components/onboarding/importScan.ts:10,69 / src/components/onboarding/OnboardingSelectStep.tsx:40
- **Scenario**: `runImportScan` always sends `watch:true, schedule:IMPORT_WATCH_SCHEDULE` (committing every scanned repo to a recurring weekly credit-metered autoscan), and `SelectStep` discloses the cost as `selected.size * (MONTHLY_RUNS[IMPORT_WATCH_SCHEDULE] ?? 0)`. If `IMPORT_WATCH_SCHEDULE` changes to `"daily"` but `MONTHLY_RUNS` isn't updated, or someone hardcodes `"weekly"` in the disclosure while the POST sends something else, the user is told one recurring cost and committed to another. `MONTHLY_RUNS[IMPORT_WATCH_SCHEDULE]` silently returning `undefined ?? 0` makes the disclosed monthly cost render as **0 credits/month** while the backend still auto-watches — a disclosed-cost-of-zero for a real recurring charge.
- **Root cause**: The disclosure and the commitment are wired through a shared constant by *convention* (the file comment at :7-10 says "the copy and the POST body must never drift apart") but nothing enforces it. `credit-estimate.ts` is tested in isolation; the link between the schedule the wizard *sends* and the cost it *shows* is not.
- **Impact**: Users commit to recurring prepaid-credit draws they were shown as free/zero or as a different cadence — a billing-disclosure correctness bug with regulatory/trust weight.
- **Fix sketch**: A tiny `importScan.test.ts` invariant: assert `MONTHLY_RUNS[IMPORT_WATCH_SCHEDULE]` is a defined positive number (catches the `?? 0` silent-zero), and that the import request body built by `runImportScan` carries `watch:true` and `schedule === IMPORT_WATCH_SCHEDULE` (spy on `fetch`, parse `JSON.parse(init.body)`). One assertion that ties the disclosed cadence to the committed cadence.

## 4. Test the `byProminence` preselection ordering — it decides which repos auto-spend credits
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/components/onboarding/OnboardingFlow.tsx:38
- **Scenario**: `byProminence` (`b.stars - a.stars || (b.pushedAt ?? "").localeCompare(a.pushedAt ?? "")`) ranks repos; the top `MAX_SELECT` (10) are auto-checked and become the default scan set — and on the App path that set is what gets a real, credit-drawing scan + weekly watch. A regression that inverts the sign, drops the recency tie-break, or mishandles a `null`/missing `pushedAt` (the comment at :36-39 specifically relies on recency to surface the *private* repos a user actually works in) silently preselects the wrong 10 repos. Users either scan repos they don't care about (wasted credits) or miss their important private repos at the highest-value activation moment.
- **Root cause**: A pure comparator embedded in the component module, never exported, never tested — trivially testable but invisible because it's a one-liner.
- **Impact**: Wrong default scan set = wasted prepaid credits and a weaker first impression (the user's real repos buried below noise), undermining activation.
- **Fix sketch**: Export `byProminence` and table-test the ordering invariant: higher stars sort first; equal stars fall back to **more-recent `pushedAt` first**; a `null`/`undefined` `pushedAt` sorts after a present one and never throws; a list of all-zero-star repos (the private-installation case) orders strictly by recency. Then assert `[...list].sort(byProminence).slice(0,10)` is stable and deterministic for a fixed fixture. Pure, LLM-generatable, asserts a real spend-affecting invariant.

## 5. Add a changed-code coverage gate for `src/components/onboarding/**` so this context can't regress to zero
- **Severity**: Medium
- **Category**: quality-gate
- **File**: src/components/onboarding/ (whole directory — currently 0 tests)
- **Scenario**: Even after findings 1-4 land, the money gate / SSE parser / cost contract have no floor protecting them. A future PR deletes or hollows an onboarding test (or adds a new untested branch to `startScan`), CI stays green, and the regression ships. The context drifts straight back to its current state: a credit-spending, score-fabricating flow with no safety net.
- **Root cause**: No per-area coverage threshold exists; `vitest.config.js` only sets an `include` glob and a path alias — there is no `coverage` block and no gate keyed on changed files. Onboarding's pure decision logic is the kind of code a coverage floor is *for*.
- **Impact**: The two Critical findings above silently rot; "we have tests" becomes success-theater for the highest-value funnel in the product.
- **Fix sketch**: Add a calibrated, changed-code-scoped gate (not a big-bang backfill): enable `coverage` in `vitest.config.js` and require, for files under `src/components/onboarding/`, a floor on the *extracted pure logic* (`canRunRealScan`, `byProminence`, the `runImportScan` outcome branches) — e.g. via a `coverage.thresholds` per-glob entry or a CI step that fails when a changed onboarding file has no covering test. The gate's job is to assert that the money/SSE-parse invariants from findings 1-3 stay covered, not to chase lines on the JSX shells (which have no render harness anyway).
