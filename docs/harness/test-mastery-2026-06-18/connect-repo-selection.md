# Test Mastery — Connect & Repo Selection

Context 11 (`connect-repo-selection`) is the surface where users decide **which repos get watched and on what autoscan cadence** — every watched+scheduled repo draws a prepaid credit per cron run. The code is unusually self-aware about its own risk: the comments at `InstallationRepos.tsx:181-183` and `:270` explicitly name "success theater (showing a state the server never saved) on watch/schedule means scans silently never run" and "a 2xx whose every row failed is not a success". **Yet none of that logic has a single test.** The only test touching this context is `credit-estimate.test.ts`, which covers a helper imported from outside the context. There is no jsdom/testing-library in the toolchain (vitest runs in node; see `vitest.config.js`), so the optimistic-mutation logic is currently *untestable where it lives* — it must be extracted to be tested. The highest-leverage move is to pull the pure reducers/derivations out of the component and unit-test the failure branches, plus one Playwright e2e (the repo already has `e2e/org-suite/onboarding.spec.ts` as a template) for the genuinely DOM-bound rollback.

---

## 1. Test the optimistic watch/schedule rollback so a silent save-failure can't masquerade as success
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/components/connect/InstallationRepos.tsx:184-222 (`toggleWatch`, `changeSchedule`)
- **Scenario**: A refactor flips the `if (!res.ok)` guard, drops the `prevWatched`/`prevSchedule` capture, or moves the `setRowError` so the rollback `patch(...)` no longer fires. The UI keeps showing the checkbox checked / cadence set, the user walks away believing the repo is watched, but the row was never persisted — the cron rescan never scans it. This is exactly the "success theater → scans silently never run" failure the code comment warns about, and nothing would catch the regression.
- **Root cause**: The whole optimistic-then-confirm-or-rollback state machine lives inline in a client component with no test environment (no jsdom/testing-library installed). Because it can't be rendered in vitest, the failure path (non-2xx → revert to prior value + inline error) has zero coverage; only the happy path is exercised, and only by hand.
- **Impact**: A repo the customer believes is under continuous scanning is silently dropped from the watchlist/schedule — the product's core promise ("we keep scanning your fleet") fails invisibly, and on a paid plan they may have provisioned credits for runs that never happen.
- **Fix sketch**: Extract a pure reducer, e.g. `applyWatchResult(prevState, { fullName, requested, serverOk }): RepoState` (and the schedule analog), into `installationRepoTypes.ts` or a new `repoMutations.ts`. Unit-test the invariant: **on `serverOk === false` the returned state's `watched`/`scanSchedule` MUST equal the value before the optimistic patch, never the requested value**, and an error message MUST be present. Assert all four cases (watch-on-fail, watch-off-fail, schedule-fail, success-keeps-new-value). Then have `toggleWatch`/`changeSchedule` call the reducer so the component is a thin shell.

## 2. Test the bulk-watch partial-failure accounting — a 2xx that saved nothing must read as an error
- **Severity**: High
- **Category**: success-theater
- **File**: src/components/connect/InstallationRepos.tsx:245-282 (`watchAllFiltered`), :270-275 (the `ok === 0` branch)
- **Scenario**: The `/api/org/watch` bulk route (route.ts:45-60) returns `{ ok: true, count, failed }` and can legitimately return a **200 where every row is in `failed`** (each `setRepoWatch` threw). The UI math `const ok = targets.length - failed.length` then drives the message. If a refactor changes `failed` handling (e.g. treats a missing `failed` array as "all succeeded", or removes the `ok === 0` special case), the user sees "Now watching N repos" while zero were saved, and the optimistic patches are left applied for rows that actually failed (the `failed.forEach((fn) => patch(fn, { watched: false }))` rollback at :268 is the only thing keeping them honest).
- **Root cause**: This counting/rollback logic — the most subtle correctness code in the file, explicitly commented as guarding success theater — is inline and untested. There is no assertion that `failed`-listed rows get reverted, nor that an all-failed 200 produces an error-kind message rather than a "watching 0" note.
- **Impact**: Bulk "Watch all" on a 200-repo org is the primary onboarding action; a false "Now watching 200 repos" when none saved means an entire fleet is silently unwatched while the user believes setup is done — fleet dashboards stay empty and no scheduled scans run.
- **Fix sketch**: Extract `summarizeBulkWatch({ targetFullNames, failed, responseOk }): { revertFullNames: string[]; message: { kind: "note"|"error"; text } }`. Unit-test: (a) `responseOk:false` → all targets in `revertFullNames`, kind `error`; (b) every target in `failed` → kind `error` with "none were saved", and `revertFullNames === failed`; (c) partial `failed` → kind `note` with "`ok` watching · `failed.length` failed" and only failed rows reverted; (d) empty `failed` → kind `note`, no reverts. Invariant: **the count claimed in the success message equals `targets.length - failed.length` and is never > 0 when every row failed.**

## 3. Test the repo filter predicate against query/visibility/watched/language combinations
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/components/connect/InstallationRepos.tsx:231-241 (the `filtered` `useMemo`)
- **Scenario**: The filter combines a free-text query (matched against `fullName` AND `language`), a tri-state visibility (`all`/`public`/`private`), a `watchedOnly` toggle, and a language dropdown. A regression — e.g. dropping the `r.private` negation so `visibility === "public"` shows private repos, or making the query case-sensitive, or `language !== "all"` comparing against the wrong field — would either **leak private repos into a "public" filter** or hide repos the user is searching for. The `Watch all (N)` button and the credit estimate are both computed off `filtered`, so a filter bug also miscounts the billable commitment shown to the user.
- **Root cause**: A pure, deterministic predicate (no DOM, no async) sitting inline in a `useMemo`, so it never got a test despite being trivially LLM-generatable. This is the single highest value-per-line gap in the context.
- **Impact**: Visibility leakage is a trust/privacy concern (a "public" filter implying these are the safe-to-share repos); a miscounted `filtered` set drives the wrong bulk-watch target count and the wrong credit estimate at the moment of spend commitment.
- **Fix sketch**: Extract `filterRepos(repos, { query, visibility, watchedOnly, language }): AppRepo[]` into `installationRepoTypes.ts`. Table-test against a fixed fixture of ~5 repos (mixed public/private, mixed language incl. `null`, mixed `state.watched`). Assert invariants: **`visibility:"public"` never returns a repo with `private:true`** (and the symmetric private case); query matches case-insensitively on both `fullName` and `language`; `watchedOnly` excludes `state:null` and `state.watched:false`; an unknown `language` value returns `[]`; empty query returns the full set unfiltered by text.

## 4. Test the connect page's install-routing derivation (orgInSession / pendingInstall / auth-off push)
- **Severity**: Medium
- **Category**: error-branch
- **File**: src/app/connect/page.tsx:103-113 (`installs`, `orgInSession`, `pendingInstall`, the auth-off `installs.push`)
- **Scenario**: This block decides whether a `?org=`/`?installation_id=` arriving from the install redirect renders a live `<InstallationRepos>` panel or a "Finish connecting / Re-sync" prompt. The stated contract (comment at :97-102): when auth is ON and the org isn't yet in the session, **must NOT** render the panel (because `/api/app/repos` would 403 — see repos/route.ts:38); when auth is OFF, push the query-carried org directly. A regression in the `orgInSession` matching (it compares `installation_id` by id, else `login` case-insensitively) or in the `pendingInstall` guard would render a panel that hard-403s on load, or push a not-authorized org into the list under auth — surfacing a dead/erroring panel right at the install handoff.
- **Root cause**: All of this is pure derivation over `(session, org, installation_id, isAuthConfigured())`, but it's embedded in an async server component with no test, so the auth-on-vs-off branch matrix is unverified.
- **Impact**: A broken derivation breaks the most fragile moment of the funnel — the redirect back from GitHub after install — leaving new orgs staring at a 403 panel instead of the one-click re-sync, and silently churning would-be activations.
- **Fix sketch**: Extract `resolveInstallView({ session, org, installationId, authConfigured }): { installs; pendingInstall: string|null }`. Unit-test the matrix: org already in session → no `pendingInstall`, panel renders; auth-on + org not in session → `pendingInstall === org`, org NOT in `installs`; auth-off + org not in session → org pushed into `installs`, no `pendingInstall`; id-vs-login matching is case-insensitive on login and exact on id. Invariant: **when `authConfigured && !orgInSession`, the org never appears in `installs`** (so the 403-prone panel is never rendered).

## 5. Add one Playwright e2e for the watch-toggle rollback path
- **Severity**: Low
- **Category**: determinism
- **File**: src/components/connect/InstallationRepos.tsx:188-201 (the network round-trip) — new spec under e2e/
- **Scenario**: The unit-extracted reducers (findings 1-2) prove the *logic*, but nothing proves the wiring: that a failed `POST /api/org/watch` actually reverts the rendered checkbox and renders the `role="alert"` row error. A wiring regression (e.g. the reducer is correct but its result is never fed back to `setView`) would pass all unit tests yet still show success theater in the browser.
- **Root cause**: No component-render harness exists (no jsdom), so the only way to assert the DOM reflects a rollback is e2e — and there is no connect spec (`e2e/org-suite/` covers org views, not connect).
- **Impact**: Low business blast radius given findings 1-2 cover the logic; this is a cheap belt-and-suspenders against the glue breaking, on a high-traffic onboarding screen.
- **Fix sketch**: Add `e2e/connect.spec.ts` modeled on `e2e/org-suite/onboarding.spec.ts`. Use `page.route("**/api/org/watch", route => route.fulfill({ status: 500 }))` to force the failure, click a repo's "watch" checkbox, and assert: the checkbox returns to unchecked AND a `role="alert"` with "not saved" appears. Keep it deterministic by mocking the route (no live GitHub dependency), so it can't flake on rate limits.
