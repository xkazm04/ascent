# Test Mastery — CI Gate & Status Checks

The pure verdict math (`evaluateGate`, the security branch of `policyFromParams`, the fail-closed NaN floor) and the comment renderer (`gate-comment.ts` — crash-safety + markdown escaping) are genuinely well tested. The gaps are concentrated in (a) the **untrusted-policy validator** `sanitizeGatePolicy`, (b) the **HTTP gate endpoint** that turns a verdict into the 200/422 CI relies on, and (c) the **two GitHub write surfaces** (`checks.ts`, `write.ts`) — all three at **zero** coverage despite being the parts that actually block or unblock a merge.

## 1. Add tests for the `sanitizeGatePolicy` untrusted-policy validator — the gate's only input-sanitization boundary
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/lib/scoring/gate.ts:75-103
- **Scenario**: A change to `sanitizeGatePolicy` (e.g. relaxing the `clampScore` range check, dropping the `/^D[1-9]$/` key filter, or returning `{}` instead of `null` for an empty policy) ships silently. A malformed/hostile org gate policy from the settings form or DB row then flows straight into `evaluateGate` — a `minDimension` of `-5` or `999`, a `minDimensionFor` keyed on an arbitrary string, or a non-`"ungoverned"` posture in `forbidPostures` — making the gate either always-pass (every PR merges) or always-fail (every PR blocked) across the org.
- **Root cause**: This is the single function that converts untrusted persisted/form input into a trusted `GatePolicy`, and it has **no test file at all**. `policyFromParams` (the URL path) is tested; the DB/settings path through `sanitizeGatePolicy` is not. `getOrgGatePolicy` feeds its output into both the webhook PR gate (`route.ts` of the app webhook) and the fleet rollup (`governance.ts`), so a regression here has org-wide blast radius.
- **Impact**: A persisted-policy regression silently disables merge protection (security/testing floors stop enforcing) or hard-blocks all merges for an org — exactly the failure a quality gate exists to prevent, with no test to catch it.
- **Fix sketch**: Add `gate.sanitize.test.ts`. Assert the invariants: (1) out-of-range scores are rejected, not clamped wrong — `{ minOverall: 150 }` and `{ minDimension: -1 }` → those keys absent; `{ minDimension: "40" }` (string) and `{ minDimension: 39.9 }` → `40` / `39` via `Math.trunc`; (2) `minDimensionFor` keeps only `D1`..`D9` keys and drops `{ "D10": 50, "evil": 50, "D9": 60 }` to `{ D9: 60 }`; (3) `forbidPostures` keeps only `"ungoverned"` and drops `["governed","ungoverned"]` to `["ungoverned"]`; (4) `null`/`"x"`/`{}`/`{minLevel:"L9"}` all return `null` (so the caller falls back to the archetype default, never a half-built policy); (5) the returned object never contains a key the validator didn't set.

## 2. Test the gate HTTP endpoint's 200/422 contract — the literal interface CI branches on
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/app/api/gate/[owner]/[repo]/route.ts:66-84
- **Scenario**: Someone "simplifies" the response and returns `200` with `{ pass: false }` instead of `422`, or swaps the status ternary. `curl --fail https://…/api/gate/o/r` now exits **0** on a failing gate, so the GitHub Action / CI step that pipes this into a merge check goes green on a repo that should be blocked. Equally: the cache-key determinism fix (line 58, `makeCacheKey(…, !mock, sha)` reading the same mode it writes) regresses back to `cacheGet(llmKey) ?? cacheGet(mockKey)`, and the default mock gate starts returning a stochastic LLM verdict — a PR flips pass↔fail run-to-run with identical code.
- **Root cause**: No `route.test.ts` for the gate endpoint. The status-code mapping, the `?mock` parsing (`"0"`/`"false"` → real LLM), the `?ref=` ref-scoped bypass, and the "rate-limit only when `!mock`" decision are all asserted nowhere.
- **Impact**: The single contract every CI integration depends on — fail ⇒ non-zero HTTP — can silently break, letting sub-threshold code merge while the badge/dashboard still claim a gate is enforced.
- **Fix sketch**: Add a route test mocking `scanRepository`/`cacheGet`/`cacheSet`/`rateLimitRequest`. Assert: passing report → HTTP **200** and `body.pass === true`; failing report → HTTP **422** and `pass === false` with `failures.length > 0`; `?mock=0` invokes `rateLimitRequest` and returns `tooManyRequests` status when `rl.ok === false`, while default mock does **not** call it; `?ref=sha` calls `scanRepository(…, { mock, ref })` and bypasses the cache (no `cacheGet`); a thrown scan → HTTP **500** with an `error` body. The load-bearing invariant: `status === 200` iff `gate.pass === true`.

## 3. Test `upsertStickyComment`'s find-or-create + 404-fallback logic in `github/checks.ts`
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/lib/github/checks.ts:65-101
- **Scenario**: The marker-matching or pagination logic regresses — e.g. the `MAX_PAGES`/short-page break is dropped (unbounded scan of a 5000-comment thread), the `c.body.includes(marker)` match is loosened, or the `404 → fall through to POST` recovery (lines 89-92) is removed. Result: the bot **stacks a new comment on every PR sync** instead of updating its sticky one (spam on active PRs), or it **throws on a race-deleted comment** and the entire PR check write dies. `createCheckRun`'s conclusion field (`success`/`failure`/`neutral`) could also be mis-mapped and never caught.
- **Root cause**: No `checks.test.ts`. The upsert state machine (search pages → PATCH existing → POST new, with a 404-during-PATCH fallback) is pure orchestration over `githubAppFetch` and is fully mockable, yet entirely untested.
- **Impact**: A broken upsert either spams PRs with duplicate gate comments (eroding trust in the bot) or crashes the comment write so the PR shows a stale/absent verdict — directly degrading the merge-gate UX teams pay for.
- **Fix sketch**: Add `checks.test.ts` mocking `githubAppFetch`. Assert: (1) when a page contains a comment whose body includes the marker → exactly one **PATCH** to `/issues/comments/{id}`, **zero** POST, `updated === true`; (2) when no comment matches across pages → one **POST**, `updated === false`; (3) the PATCH throwing `AppApiError(404)` → falls through to a POST (recovery), but a `403`/`500` from PATCH **re-throws**; (4) the search stops at `MAX_PAGES` and on the first short page (assert call count). For `createCheckRun`, assert the POST body carries `status: "completed"`, the passed `conclusion`, and at most 3 `actions`.

## 4. Test the base-file overwrite guard in `github/write.ts` — a destructive, fleet-fanout path
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/lib/github/write.ts:81-88
- **Scenario**: The `existingFileSha(token, owner, repo, path, base)` check is moved to inspect the *branch* instead of *base*, or the `if (baseFileSha) throw 409` guard is dropped during a refactor. The "systematic apply" then opens a PR whose `PUT /contents` **overwrites a real customer file** (`SECURITY.md`, a CI workflow, `AGENTS.md`) with a TODO scaffold; merging the PR deletes the customer's content — and a 25-repo batch fans that out across the fleet from one click.
- **Root cause**: No `write.test.ts`. The guard's correctness hinges on a subtle detail (check **base**, not the generated branch — so an idempotent re-seed on our own branch is fine but a real base file is refused), and that exact distinction is asserted nowhere.
- **Impact**: Silent destruction of real repository content across an org — the highest-consequence write the app makes, gated only by this one untested branch.
- **Fix sketch**: Add `write.test.ts` mocking `githubAppFetch`/`existingFileSha`. Assert the invariant *"never PUT when the file already exists on base"*: (1) base file exists → `openDraftPr` rejects with `AppApiError` status `409` and **no** `PUT`/branch-create/PR-create call fires; (2) base file absent but the file exists only on our generated branch → it proceeds and the PUT carries the branch `sha` (idempotent update, not a clobber); (3) branch-create `422` and PR-create `422` are tolerated (reuse existing branch / return the open PR with `reused: true`).

## 5. Add `min_overall=0` / `min_dimension=0` cases to `policyFromParams` — close the always-pass asymmetry the security floor already guards
- **Severity**: Medium
- **Category**: edge-case
- **File**: src/lib/scoring/gate.ts:242-244
- **Scenario**: `?min_dimension=0` (or `?min_overall=0`) parses to a finite `0` and `params.get(...) != null`, so the policy installs a floor of **0** — an always-pass dimension/overall gate that still *looks* configured in the verdict's `policy` block. This is the exact trap the code deliberately closes for `min_security` (line 236: requires `> 0`, with two tests pinning `min_security=`/`min_security=0` → no floor), but the same defense was never applied to — or tested for — `min_dimension`/`min_overall`. A user copy-pasting a template with an empty value (`min_dimension=` → `Number("")=0`) silently disarms the gate.
- **Root cause**: The `min_security` empty/zero handling has explicit tests (CIGATE #2/#3); the parallel `min_overall`/`min_dimension` branch has no `=0`/empty-value test, so the inconsistency is invisible to the suite and would not flag if someone "fixed" one path without the other.
- **Impact**: A gate that reads as enforced but passes everything — the most dangerous kind of false confidence for a merge-blocking control, and an easy footgun for the user-authored badge/CI query string.
- **Fix sketch**: Add cases to `gate.test.ts`: `?min_dimension=0` and `?min_dimension=` and `?min_overall=0`/`min_overall=` against an `org` archetype — assert the resulting policy's `minDimension`/`minOverall` either fall back to the archetype default (preferred, matching `min_security` semantics) **or**, if `0` is intentionally honored, that a report scoring `0` on a dimension still **fails** (documenting that `0` is a real, enforceable floor, not an accidental always-pass). The invariant the test must pin: a query string can never produce a policy that passes a report which the archetype default would have failed, unless an explicit positive override says so.
