# Test Mastery — Live War Room

The only test in this context is `src/components/org/liveWarRoomShared.test.ts`, which covers `classifyRepoEvent` thoroughly and well (it is the honest model: a real trust-boundary helper, tested for FAILURE, asserting the NaN-fold invariant). Everything else in the context is untested — including the one genuinely security-critical file, `src/lib/live-share.ts`, which is the sole gate on unauthenticated access to an org's entire fleet rollup. The findings below rank by business blast radius, not line count.

## 1. Test the HMAC share-token boundary that gates unauthenticated fleet data
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/lib/live-share.ts:20 (sign), :34 (verifyLiveShareToken)
- **Scenario**: A refactor weakens `verifyLiveShareToken` — e.g. someone "simplifies" the timing-safe compare to `sig === expected` and drops the `a.length !== b.length` guard, or removes the `parsed.exp < Date.now()` expiry check, or stops verifying the signature at all and trusts the decoded `{org}` payload. Any of these ships green: there is no test on this file at all. The result is that `/live/shared/[token]` (`src/app/live/shared/[token]/page.tsx:32`) serves the full `getOrgRollup(verified.org)` — every repo, every maturity score, posture and adoption for an arbitrary org — to anyone who can craft (or guess the structure of) a token. This is the product's only capability-as-token surface and it sits OUTSIDE the `/org` session layout.
- **Root cause**: The module was added (WAR-4) with a glob-confirmed sibling `live-share.test.ts` absent. The HMAC, base64url payload, expiry, and "no secret ⇒ inert" branches were never pinned, so the security contract lives only in a comment.
- **Impact**: Silent cross-tenant data exposure — an unauthenticated attacker reading any org's full maturity posture. Highest possible blast radius for this SaaS (confidential engineering-maturity data is the product).
- **Fix sketch**: Add `src/lib/live-share.test.ts` asserting the invariants, with `process.env.LIVE_SHARE_SECRET` set per-test: (a) round-trip — `verifyLiveShareToken(signLiveShareToken("acme")!.token)` returns `{org:"acme"}`; (b) **forged signature is rejected** — flip one char of the sig segment → `null`; (c) **tampered payload is rejected** — re-base64url a payload with a different `org` but reuse the old sig → `null` (proves the sig actually covers the payload); (d) **expired token is rejected** — sign with `ttlMs = -1000` → `null`; (e) **length-mismatch sig** (`payload + ".x"`) → `null` (guards the `timingSafeEqual` length pre-check the code already has); (f) **no secret ⇒ inert** — unset both env vars → `liveShareEnabled() === false`, `signLiveShareToken(...) === null`, `verifyLiveShareToken(anyToken) === null`. The load-bearing invariant: a token only verifies when the signature was produced by the configured secret over the *exact* payload and the payload is unexpired.

## 2. Pin the org-casing contract between mint-authz and token-payload
- **Severity**: High
- **Category**: edge-case
- **File**: src/lib/live-share.ts:29 (`org.toLowerCase()`) vs src/app/api/org/live-share/route.ts:24 (`requireOrgRole(body.org, "owner")`)
- **Scenario**: `signLiveShareToken` bakes `org.toLowerCase()` into the signed payload, but the mint route authorizes the *raw-cased* `body.org` via `requireOrgRole(body.org,"owner")`, and the shared page later trusts `verified.org` directly in `getOrgRollup(verified.org)`. Today this is internally consistent only because `requireOrgRole`/`getOrgRollup` happen to lowercase (`authz.ts:130 slug = org.trim().toLowerCase()`). The day someone makes `getOrgRollup` case-sensitive, or drops the `.toLowerCase()` from `signLiveShareToken` to "match the authz arg," the two ends silently diverge: a token is authorized against `Acme` but resolves data for `acme` (or vice-versa). No test forces these to agree.
- **Root cause**: The casing normalization is duplicated across three files with no single test asserting "the org I am authorized to mint for is exactly the org the token resolves." It's an implicit invariant held together by coincidence.
- **Impact**: An authz-vs-data-resolution mismatch on a tenant boundary — best case a broken share link, worst case a token minted under one tenant's owner check resolving a different tenant's rollup. Money/security-adjacent because it's the authorization-to-data join.
- **Fix sketch**: In `live-share.test.ts` assert `verifyLiveShareToken(signLiveShareToken("Acme-Corp")!.token)?.org === "acme-corp"` (payload is canonicalized), AND add a route-level test (mirroring the existing `route.test.ts` pattern used elsewhere) that mocks `requireOrgRole` and asserts the minted `path` decodes to the same lowercased org that authz was called with. Invariant: the org slug authorized at mint == the org slug the verified token resolves data for, under any input casing.

## 3. Extract and test the war-room SSE fold (stats, leaderboard, celebration-on-crossing)
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/components/org/LiveWarRoom.tsx:136 (`onRepo`), :306 (`stats`), :324 (`leaderboard`), :175 (celebration gate)
- **Scenario**: The component holds untested business logic that drives a projected-to-leadership wall: (a) `stats` averages over only repos with `overall != null` and rounds — a regression that includes skipped/null repos in the denominator silently deflates "Org maturity"; (b) the celebration fires only when `posture === "ai-native" && prev?.posture !== "ai-native"` — a refactor that drops the `prev` guard re-fires 🎉 on every event for an already-native repo, or fires on the seeded standing; (c) `onRepo` must never let an `error`/`skipped`/`invalid` event overwrite a repo's seeded `overall` (the ticker-only path). `classifyRepoEvent` is tested, but the *fold that consumes it* is not — the actual bug the comment warns about (NaN tiles, lost seeded standing) could regress one layer above where the test stops.
- **Root cause**: This is the dominant pattern in this scan — the pure classifier was extracted and tested, but the reducer that applies its result (state transition, delta computation, celebration dedup, average) stayed trapped in a `useCallback`/`useMemo` inside a client component, and ascent's Vitest has no jsdom to render it.
- **Impact**: Wrong headline numbers and spurious/missing celebrations on the executive-facing wall — quietly erodes trust in the product's flagship demo surface; a deflated average could understate fleet maturity to leadership.
- **Fix sketch**: Extract three pure functions into `liveWarRoomShared.ts`: `foldRepoEvent(prevRepos, ev, id) -> { repos, tickerEntry, celebration? }`, `deriveStats(repos)`, `deriveLeaderboard(repos)`. Unit-test: a `skipped`/`error`/`invalid` event leaves `repos[full].overall` unchanged from its seed; `deriveStats` denominator counts only scored repos and `avgOverall` is `null` for an all-null fleet; a celebration is emitted iff posture transitions *into* `ai-native` from a non-native prev (and NOT when `prev` is already native, NOT on first-seen with no prev posture per the current code path); leaderboard sorts by `overall` desc then name asc with nulls excluded. Invariant: non-scored events are display-only and can never mutate a scored/seeded standing or the averages.

## 4. Cover the daysUntil deadline-countdown branches with an injectable clock
- **Severity**: Medium
- **Category**: error-branch
- **File**: src/components/org/LiveWarRoomHeader.tsx:25 (`daysUntil`)
- **Scenario**: `daysUntil` powers the deadline banner ("3d to deadline" / "5d past deadline"). It has three untested branches — `null` date → `null`, unparseable date (`Number.isNaN(Date.parse)`) → `null`, and the `Math.ceil((t - Date.now())/86_400_000)` arithmetic. A regression swapping `Math.ceil` for `Math.floor`, or mis-signing the past/future split, flips a "2d to deadline" into "0d"/"-1d past deadline" on a live leadership wall. It also reads `Date.now()` directly, so it can't be tested deterministically and is duplicated (a second, differently-signed `daysUntil(target, now)` exists at `src/lib/db/org-insights.ts:223`).
- **Root cause**: A pure date helper inlined in a component, coupled to the wall clock, with no exported seam — so its branches were never exercised and the sign convention is unpinned.
- **Impact**: A wrong or inverted deadline countdown visibly mis-states campaign urgency to leadership during a review.
- **Fix sketch**: Export `daysUntil` (or hoist it into `liveWarRoomShared.ts`) with an optional `now = Date.now()` param, then unit-test: `null`/`""`/`"not-a-date"` → `null`; a date exactly `now + 2*86_400_000` → `2`; same-day (sub-24h future) → `1` via `ceil`; a past date → negative. Invariant: positive = days remaining, negative = days past, with no off-by-one across the midnight boundary.

## 5. Guard the PostureMix true-distribution scaling against the regression it just fixed
- **Severity**: Low
- **Category**: coverage-gap
- **File**: src/components/org/LiveWarRoomPanels.tsx:10 (`total = Math.max(1, scored, ...)`)
- **Scenario**: The bar-width denominator was deliberately changed from "scale to the largest bucket" to "scale to the total scored fleet" because scaling-to-max made the leading posture always render as a full 100% bar — overstating the dominant posture's prevalence to leadership. There is no test, so a well-meaning revert to `Math.max(...counts)` (or dropping the `scored` term) silently reintroduces the exact overstatement the comment warns about, and the `Math.max(1, ...)` divide-by-zero guard on an empty fleet is unverified.
- **Root cause**: A known-regression fix protected only by a code comment; the width math is trapped in JSX with no extracted, asserted function.
- **Impact**: A projected war-room wall that overstates posture prevalence to leadership — a credibility/decision-quality hit on the flagship demo, low blast radius vs. the items above.
- **Fix sketch**: Extract `postureBarPct(count, scored, counts) -> number` and unit-test: with counts `{native:1, manual:1}` and `scored:2`, each bar is `50`, NOT `100`; with `scored:0` and empty counts the denominator stays `1` (no `NaN`/`Infinity`, all bars `0`); a single-posture fleet of 4 renders that bar at `100` honestly (4/4) but a 1-of-4 bucket renders `25`. Invariant: each bar's width is its fraction of the whole scored fleet, never normalized to the leading bucket.
