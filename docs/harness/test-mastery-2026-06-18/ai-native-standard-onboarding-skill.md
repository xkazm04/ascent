# Test Mastery — AI-Native Standard & Onboarding Skill

## 1. Assert the generated manifest round-trips through the generated doctor's own parsers
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/lib/standard/doctor.ts:39 (capabilities/kv/sub/flow parsers) vs src/lib/standard/manifest.ts:87 (serializeManifestYaml)
- **Scenario**: Someone tweaks `serializeManifestYaml` — e.g. switches a capability line to single quotes, indents a key differently, or makes `flowList` emit `[ a, b ]` with inner spaces, or `scalar()` wraps a value the doctor's regex no longer matches. The unit tests still pass (they only `expect(yaml).toContain("capabilities:")`), but the doctor that ships *inside the same SKILL.md* now parses `capabilities(text)` as `{}`, emits `add('fail', 'no capabilities declared')`, and `process.exit(1)`. Every adopting repo's CI conformance gate now hard-fails on a manifest Ascent itself generated.
- **Root cause**: The doctor's `capabilities()` regex (`/^\s{2}([\w-]+):\s*\{\s*command:\s*"([^"]*)"/`), `flow()` (`key: [..]`), `kv()`/`sub()` are an *executable contract* with the serializer, but the two halves are tested in isolation: `standard.test.ts` asserts the serializer's output as strings and asserts the doctor's *source text* contains markers — nothing parses one with the other. The coupling (double-quoted command, 2-space indent, flow-list shape) is implicit and unenforced.
- **Impact**: A silent serializer/parser drift turns the product's core adopt→verify→re-score loop into a CI failure for *all* downstream repos at once — the highest blast radius in this context, and exactly the kind of regression that ships green.
- **Fix sketch**: Add a test that reimplements (or string-extracts and `new Function`-evals) the doctor's pure parsers against `serializeManifestYaml(buildManifestData(report))`. Assert invariants: `capabilities(yaml)` returns `{build,test,lint,typecheck}` with the exact commands from `buildManifestData`; `flow(yaml,'prePush')` deepEquals `["lint","typecheck","scan-secrets"]`; `kv(yaml,'schema')==='ai-manifest'`; `sub(pathsBlock,'memory')==='.ai/memory/'`. Run it for TypeScript/Python/Go reports so a language with a quote-needing command (none today, but `generic` has `<...>` placeholders) is covered.

## 2. Execute the generated doctor against a fixture repo and assert the score/exit-code gate
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/lib/standard/doctor.ts:124 (`weight={pass:1,warn:0.5,fail:0}`, `exit(fails>0?1:0)`)
- **Scenario**: The conformance *score* and the *exit code* are the CI merge gate. A refactor flips a `check(ok,...,'warn')` to `'fail'` (or vice-versa), or changes the `weight` map, or the `prePush`-not-wired branch from `warn` to `fail`. A repo that should pass now blocks merges, or a repo missing `.ai/manifest.yaml` now exits 0 and the gate is toothless. No test runs the script, so the verdict logic is entirely unverified.
- **Root cause**: `standard.test.ts` only asserts `doc.body` *contains* strings like `"Conformance:"` and has no backticks. The script's actual behavior — findings collection, the `fails>0?1:0` exit, the `--json` summary shape POSTed to `/api/report/conformance` — is never run. The script is authored backtick-free precisely so it can be evaluated; that affordance is unused.
- **Impact**: The CI quality gate that the whole "shift-left" thesis sells can silently invert (false-block or false-pass) and ship green. A false-pass means adopting repos believe they're conformant when they aren't; a false-block breaks their CI.
- **Fix sketch**: In a temp dir, write the doctor body to `doctor.mjs` and `spawnSync(process.execPath, ['doctor.mjs','--json'])`. Fixture A: a valid `.ai/manifest.yaml` + `.ai/memory/` + `.ai/context-index.json` → assert exit 0 and JSON `fails===0`. Fixture B: no manifest → assert exit 1 and a `fail` finding. Fixture C: manifest present but `prePush` declared with no hook file → assert exit 1 (the "highest-severity" fail the skill promises). Assert the `--json` payload shape (`{score,fails,warns,findings}`) matches what `conformance/route.ts` ingests.

## 3. Test the SKILL.md frontmatter-injection and code-fence escaping invariants for malicious inputs
- **Severity**: High
- **Category**: success-theater
- **File**: src/lib/onboarding/skill.ts:60 (`safeDesc`) and src/lib/onboarding/skill.ts:139 (`embedFile` fence sizing)
- **Scenario (a)**: A repo `description` containing `"` then a newline then `name: evil` — `safeDesc` is meant to collapse `[\r\n]+`→space and `"`→`'` so it can't break out of the quoted YAML scalar. If that regex regresses (e.g. someone "simplifies" to only strip `\n`), the generated SKILL.md frontmatter becomes invalid/forgeable. **Scenario (b)**: A generated file body contains a 4-backtick run; `embedFile` must open a 5-backtick fence (`Math.max(4, longestRun+1)`). If the `+1` regresses to a fixed 4, the body's fence closes the outer fence early and the rest of the SKILL.md leaks as rendered markdown.
- **Root cause**: Existing tests only check happy-path *presence* (`toContain("acme/api")`, `toMatch(/^---\nname:/)`); none feed adversarial `description`/`name`/file-body and assert the structural invariant holds. These are security-shaped sanitizers tested as if they were formatters — classic false-confidence.
- **Impact**: SKILL.md is downloaded and executed by the adopting repo's Claude Code CLI. A frontmatter break corrupts the skill (won't load); a fence break leaks/garbles the doctor source the agent is told to write verbatim — both turn a "personalized onboarding" into broken or misleading agent instructions.
- **Fix sketch**: (a) `buildOnboardingSkill(reportWith({description:'x"\nname: pwned\n---\nmalicious'}))` → assert the frontmatter block (text between the first two `---`) is exactly two keys (`name`, `description`), the description line has no raw newline, and a YAML parse of the block succeeds with `name==='ascent-onboard'`. (b) Construct a `GeneratedFile` whose `body` contains ```` ```` ```` (4 backticks) via a stubbed `buildFoundation`, or assert directly on `embedFile`: the chosen fence length `>` the longest backtick run in the body, and `body.split(fence).length===2` boundaries are the wrapper only.

## 4. Cover the /api/report/skill route: auth gate, 404/503, and Content-Disposition sanitization
- **Severity**: Medium
- **Category**: error-branch
- **File**: src/app/api/report/skill/route.ts:26
- **Scenario**: The route is the auth boundary for a *private* report's skill ("as sensitive as the report"). A regression that drops the `requireOrgRead(orgSlug)` call, or passes `orgSlug` incorrectly to `getScanReportByCommit`, would leak a private repo's baked-in scan facts to an unauthenticated caller. Equally untested: the `@sha` parsing edge (`parseRepo` returns null on `owner/` or `/name`), the 404 on no saved scan, the 503 on no DB, and the `safe()` sanitizer that strips a caller-supplied `sha` out of the `Content-Disposition` filename (header-injection guard).
- **Root cause**: There is no `route.test.ts` anywhere under `src/app/api/report/`. The sibling PDF route is cited as the mirror but its gating isn't reused/asserted here. All four branches and the filename sanitizer are unverified.
- **Impact**: A private report's onboarding skill (and thus its dimension scores, gaps, headline) leaking cross-tenant is a data-confidentiality incident; a header-injection via `sha` is a response-splitting risk. These are the money/security branches of an otherwise pure context.
- **Fix sketch**: Route test with mocked `readableOrgForOwner`/`requireOrgRead`/`getScanReportByCommit`/`isDbConfigured`. Assert: `requireOrgRead` returning a denial response short-circuits (no report read); `isDbConfigured()===false`→503; missing/invalid `?repo`→400; no report→404; success sets `content-type: text/markdown` and a `content-disposition` whose filename, when `sha='../../etc"\r\nX:'`, contains no `/`, `"`, `\r`, or `\n` (only `[A-Za-z0-9._-]`). Add one assertion that a failing `recordSkillGeneration` (rejected promise) does not reject the response — the fire-and-forget `.catch(()=>{})` contract.

## 5. Add a focused property test for maintain.mjs memory-entry numbering/slug invariants
- **Severity**: Low
- **Category**: edge-case
- **File**: src/lib/standard/maintain.ts:56 (`note` id derivation + slug)
- **Scenario**: The `note` subcommand parses existing `NNNN-` filenames, takes `max(...)+1`, zero-pads to 4, and slugifies the text to `[a-z0-9-]`, ≤40 chars. A regression in the `parseInt`/`Math.max` (e.g. off-by-one, or non-`NNNN` files like the `README.md` slipping into the id set) would produce a duplicate or `NaN` id, silently overwriting a prior memory entry — the append-only ledger the whole standard sells. The slug edge (all-punctuation text → empty → fallback `'note'`) is also unguarded.
- **Root cause**: maintain.mjs is emitted as a string and only checked for substring presence; its id/slug logic is pure and trivially testable but never exercised. "Append-only memory" is an asserted product property with no test behind it.
- **Impact**: Low blast radius (it's the adopting repo's local upkeep script, not Ascent's data), but a duplicate-id overwrite quietly destroys recorded decisions/failed-approaches — the exact knowledge the memory store exists to preserve.
- **Fix sketch**: Extract the pure helpers (or eval the script's functions) and assert: given files `['0001-a.md','0007-b.md','README.md']`, next id is `'0008'` (README ignored, max not count); empty dir → `'0001'`; text `'!!! @@@'` → slug falls back to `'note'`; a 60-char text → slug ≤40 chars and has no leading/trailing `-`. These are LLM-generatable table cases asserting a real monotonic-id invariant.
