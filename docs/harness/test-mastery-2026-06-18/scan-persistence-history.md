# Test Mastery — Scan Persistence & History

The only test in this context, `src/lib/db/scans.test.ts`, covers the three concurrency *primitives* (`isUniqueConstraintError`, `upsertRacing`, `withRepoLock`) with genuinely good failure-path assertions. But it explicitly defers the actual business logic — "the persist path itself is exercised by the e2e suite" (`scans.test.ts:6`) — and that claim is hollow: a grep across `e2e/` finds **zero** references to dedup, carry-forward, or `persistScanReport`, and the only persistence-touching e2e test (`e2e/scan.spec.ts`) scans a live repo once and asserts UI rendering, never re-scanning to assert any persistence invariant. The entire dedup / carry-forward / head-recency / reconstruction surface ships untested. Findings are ranked by blast radius: silent double-billing and lost recommendation-tracking state first.

---

## 1. Test the commit-SHA dedup so an unchanged re-scan can never create a second metered Scan row
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/lib/db/scans-persist.ts:144-159, 327-334
- **Scenario**: A refactor to the dedup block — e.g. the `findScanByCommit(repo.id, headSha)` lookup starts returning the wrong repo's row, the `if (headSha)` guard is inverted, or the P2002 cross-instance backstop (line 327) stops re-reading the winner — makes `deduped` come back `false` for a commit that was already scored. Every cron rescan and every double-click then inserts a fresh Scan row and (per the `PersistResult` doc, line 22) triggers "redundant LLM persistence and a second usage-based charge."
- **Root cause**: `persistScanReport` has no unit test at all. The dedup decision (`existing → {deduped:true}` vs insert) is the single most expensive branch in the product to get wrong — it gates billing — yet nothing asserts it. The e2e suite scans each repo exactly once, so the dedup path is never exercised even there.
- **Impact**: Silent over-billing of customers on every repeat scan, plus duplicated LLM cost on Ascent's side. A regression here is invisible until an invoice complaint.
- **Fix sketch**: Add `scans-persist.test.ts` with a Prisma mock (mock `findScanByCommit`, `prisma.scan.create`, `repository.upsert`). Assert: (a) when `findScanByCommit` returns an existing row, the result is `{deduped:true, scanId:<existing>}` AND `prisma.scan.create` is **never called** (the load-bearing invariant — zero new metered rows); (b) when it returns null, exactly one `scan.create` runs and `deduped:false`; (c) when `scan.create` throws a P2002 and `findScanByCommit` then returns a winner, the result is `{deduped:true}` reusing the winner's id and the error is swallowed; (d) when P2002 fires but no winner is found, the error re-throws (line 329).

## 2. Test carry-forward so a re-scan can never silently reset tracked recommendation status/assignee/due-date
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/lib/db/scans-persist.ts:161-181, 231-246
- **Scenario**: The `matchRecommendations` integration drifts — the `prevRecs`/`report.roadmap` argument order flips, `carryMatch[i]` indexing desyncs from the `report.roadmap.map((r,i)=>…)` insert order, or the `previous` query's tiebreaker (`orderBy: [scannedAt desc, createdAt desc, id desc]`, line 174) regresses to a bare `scannedAt desc` and picks a non-canonical predecessor. Every re-scan then writes `status:"open", assigneeLogin:null, targetDate:null` for items a team had marked in-progress/done and assigned.
- **Root cause**: `matchRecommendations` itself is well-tested in `compare.test.ts` (9 refs), but its **wiring into the persist transaction** — that the matched `prev` index actually maps to the recommendation being created at the same roadmap index, and that the carried `status/assigneeLogin/targetDate` land on the new row — is untested. A correct matcher wired up wrong loses exactly the same data.
- **Impact**: Teams lose their entire maturity-roadmap planning state (progress, ownership, due dates) silently on every rescan — corrupting the backlog/goals dashboards that read these. Erodes trust in the core tracking value prop.
- **Fix sketch**: In `scans-persist.test.ts`, mock `previous` to return prior recs with non-default `status:"in_progress"`, an `assigneeLogin`, and a `targetDate`, and a `report.roadmap` whose titles are *rephrased* (forces tier-2/3 matching). Capture the `recommendations.create` payload and assert the carried fields land on the index `matchRecommendations` paired — and that a roadmap item with **no** prior match defaults to `status:"open", assigneeLogin:null, targetDate:null`. Also assert the tiebreaker: given two prior scans sharing a `scannedAt`, the carried state comes from the one with the later `createdAt`/`id`.

## 3. Replace the hollow "exercised by the e2e suite" claim with a real head-pointer recency-guard test
- **Severity**: High
- **Category**: success-theater
- **File**: src/lib/db/scans-persist.ts:121-135 (claim at scans.test.ts:6)
- **Scenario**: The recency guard `where: { id: repo.id, OR: [{lastScanAt: null}, {lastScanAt: {lt: scannedAtDate}}] }` (line 131) regresses to an unconditional `update` (the exact bug the comment at lines 81-84 says it fixed). A delayed/replayed scan of an *older* commit then rolls `headSha`/`headEtag`/`lastScanAt` backwards, so the next conditional re-scan sends `If-None-Match` for the wrong commit and `lastScanAt` moves back in time.
- **Root cause**: The comment block documents this as a previously-shipped data-corruption bug, but no test pins the fix. The `scans.test.ts:6` comment asserts the persist path is covered by e2e; grepping `e2e/` for dedup/persist/carry-forward yields nothing, and `scan.spec.ts` scans each repo once — so the recency guard is verified nowhere. This is false confidence by reference.
- **Impact**: Silent regression to head-pointer rollback breaks conditional re-scan freshness (wasted scans / stale "up to date" UI) and re-introduces headSha↔headEtag tearing. Class of bug the team already paid to fix once.
- **Fix sketch**: Unit-test the head-advance call: with a mocked `prisma.repository.updateMany`, persist a report whose `scannedAt` is *older* than the stored `lastScanAt` and assert the `where` clause carries the `lastScanAt < scannedAtDate` predicate (i.e. the update is a no-op for an older scan); persist a *newer* one and assert `headSha`+`headEtag` are both in `data` together. Also delete or rewrite the misleading `scans.test.ts:6` comment so it stops claiming coverage that doesn't exist.

## 4. Add an LLM-generatable batch for the persisted-JSON parse helpers in scans-read (corruption defenses)
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/lib/db/scans-read.ts:555-609 (parseStringArray / parseJsonObject / parseNumberArray / parseDiscrepancies)
- **Scenario**: `getScanReportByCommit` rebuilds the public report (the `/report/{owner}/{repo}@{sha}` permalink) directly from stored JSON columns. These four pure guards are the only thing standing between a malformed/legacy/hand-edited row and a client crash: `parseNumberArray` is documented (lines 588-590) to stop a NaN-positioned SVG point in the commit-activity sparkline; `parseJsonObject` stops a non-object `prStats`/`governance` reaching charts as something that NaNs or throws. If any guard regresses (e.g. `parseNumberArray` stops filtering non-finite entries, or `parseJsonObject` stops rejecting arrays), a single bad row renders the report page broken.
- **Root cause**: These are pure, total functions — the cheapest possible thing to test — yet none are covered. They're exported only indirectly through `getScanReportByCommit`, which itself has no test, so the corruption defenses are unverified.
- **Impact**: A corrupted or legacy scan row silently crashes / NaN-renders a shareable public report, with no test to catch a weakening of the guard.
- **Fix sketch**: These are pure functions — extract or test them via a small batch asserting real invariants: `parseNumberArray('[1,"x",NaN,2]')` → `[1,2]`; `parseNumberArray('{"a":1}')` → `null`; `parseJsonObject('[1,2]')` → `null` (array rejected) and `parseJsonObject('{"x":1}')` → the object; `parseDiscrepancies` drops entries missing `dimension`/`claim` and keeps well-formed ones; every helper returns `[]`/`null` (never throws) on invalid JSON, `null`, and `undefined`. Pure, deterministic, zero-mock — ideal LLM-generatable batch.

## 5. Test the sha-less `findScanByScannedAt` dedup fallback and the negative/NaN `limit` clamp in getRepositoryHistory
- **Severity**: Medium
- **Category**: edge-case
- **File**: src/lib/db/scans-read.ts:51-65 (findScanByScannedAt); src/lib/db/scans-read.ts:138 (limit clamp)
- **Scenario**: (a) For a report with no resolvable commit SHA, dedup falls back to matching on `scannedAt` (persist line 155). If the equality/ordering regresses, coalesced followers / double-submits / retried lanes each insert a duplicate sha-less Scan row — re-introducing the duplication the fallback exists to prevent (and the code itself flags this as "inherently fragile"). (b) `getRepositoryHistory`'s `limit` clamp `Math.max(1, Math.min(200, Math.trunc(opts.limit ?? 30) || 30))` (line 138) is documented to stop a negative `take` returning the OLDEST scans (Prisma's other-end behavior) and to coerce NaN→30 and cap unbounded heavy queries; a regression silently serves wrong/oldest history to a trend chart or lets a probing `?limit=` run an expensive query.
- **Root cause**: The sha-less fallback and the clamp are both pure-decision boundaries with documented prior bugs, neither tested. `findScanByScannedAt` is mockable; the clamp is pure arithmetic that can be tested by extracting/asserting the expression.
- **Impact**: Duplicate sha-less rows inflate history and (per the dedup contract) risk a second metered charge; a mis-clamped limit shows the wrong scans on the trend view or opens a cheap heavy-query vector.
- **Fix sketch**: Assert the clamp directly: `limit(-5)→1`, `limit(NaN)→30`, `limit(undefined)→30`, `limit(5000)→200`, `limit(50)→50`. For the fallback, mock `findScanByScannedAt` and assert persist reuses the existing id with `{deduped:true, headSha:null}` when a same-`scannedAt` row exists, and inserts when it doesn't — and that the read query carries the `[createdAt desc, id desc]` tiebreaker so it's deterministic on a timestamp tie.
