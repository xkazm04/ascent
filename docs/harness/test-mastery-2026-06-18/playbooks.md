# Test Mastery — Playbooks

The Playbooks context lets an org author a best-practice standard once and roll it out across a fleet of repos — including a route (`/api/org/playbooks/[id]/apply`) that writes a real draft PR into a customer repository using the org's GitHub App installation token. The entire context ships with exactly **one** test file (`playbook-brief.test.ts`), and it covers only the pure `playbookMarkdown` helper on its happy path. Every money/security/data-integrity path — the PR-writing apply route, its cross-tenant tenancy guard, the adoption/lift analytics, and the steps-bounding serializer — is completely untested. Findings are ranked by business blast radius.

## 1. Test the playbook apply route for tenancy + the 409 "won't overwrite" branch it silently mishandles
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/app/api/org/playbooks/[id]/apply/route.ts:25 (whole route); error mapping at :96-103
- **Scenario**: A signed-in user from org A POSTs `{ repo: "victimOrg/secret" }` to a playbook id belonging to org B (or to victimOrg's), and the route mints an installation token and writes a draft PR into a repo they don't own — a cross-tenant write IDOR. Equally: a repo that already has `docs/playbooks/<slug>.md` on its base branch triggers `openDraftPr`'s 409 guard (write.ts:82), but this route maps any non-403/404 `AppApiError` to a **502 "GitHub rejected the write. Check the repo and base branch."** (line 97/101) — the user is told the write *failed* when it was *correctly refused to protect their file*, and a regression that removed the owner-match guard at line 49 would ship green.
- **Root cause**: The route has **no test file at all**. The two safety gates it relies on — `requireOrgAccess(org)` (line 43) and the `parsed.owner === org` match (line 49) — and the AppApiError→HTTP mapping are exercised only in production. The sibling `practices/apply` route handles 409 explicitly (apply/route.ts:85-90) with the right message and status; the playbook clone **diverged** and drops the 409 case — a divergence no test catches.
- **Impact**: A cross-tenant PR-write into an arbitrary customer repo (supply-chain/trust blast radius), plus a confusing/wrong error that erodes trust in the "won't clobber your files" promise — both on a path that touches every customer's source.
- **Fix sketch**: Add `route.test.ts` mirroring the repo harness (`vi.mock("next/server")`, `@/lib/db`, `@/lib/github/app`, `@/lib/github/write`, `@/lib/authz`). Assert: (a) when `getPlaybookOrgSlug` returns org B but `requireOrgAccess` returns a 403 Response, the route returns that 403 and **never calls `getInstallationToken`/`openDraftPr`** — invariant: *no token is minted and no write is attempted when the tenancy gate denies*; (b) `parsed.owner !== org` returns 400 and no write; (c) when `openDraftPr` throws `new AppApiError(409, …)`, the response status is **409** (not 502) with a "won't overwrite" message — invariant: *the 409 collision surfaces as a refusal, identical to practices/apply*; (d) on success `applyPlaybook` + `recordAudit("playbook.pr_opened", …)` are both invoked once.

## 2. Test getPlaybookAdoption's lift math for the "honest only-after-a-later-scan" invariant
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/lib/db/playbooks.ts:183-249 (lift computation at :233-239)
- **Scenario**: A refactor of the baseline/current selection (the `[...series].reverse().find(p => p.at <= a.appliedAt)` baseline and the `current.at > baseline.at` guard) starts counting the *apply-time* scan as both baseline and "after", or includes applications with **no post-apply scan**, inflating `lift` and `measured`. The PlaybookCard then renders a fabricated "▲ +N avg D5 since" improvement (PlaybookCard.tsx:161-165) that no scan supports.
- **Root cause**: `getPlaybookAdoption` — the function that produces the headline adoption metric and the `measured` honesty counter — has **zero tests**. The module comment explicitly promises "Honest — only counts an application toward `lift` when there's a scan after the apply date," but nothing pins that promise. It's pure-enough to test against an in-memory Prisma mock (it only reads).
- **Impact**: A maturity-SaaS selling "your standards are working" on fabricated lift numbers is a credibility/contract risk — leadership makes investment decisions off this figure on the Practices and Executive surfaces.
- **Fix sketch**: Mock `getPrisma()` to return canned `playbook`, `playbookApplication`, `repository`, `scan` rows. Assert invariants: (a) an application with **only a scan at-or-before `appliedAt`** yields `lift: null` and `measured: 0` (the strict `current.at > baseline.at` guard); (b) an application whose dimension score went 40→70 with a later scan yields `lift: 30`, `measured: 1`; (c) `repos` counts **distinct** `repoFullName` (dedupes double-applies); (d) a playbook with no `dimId` match contributes `lift: null`. Each test asserts the *number*, not just that a key exists.

## 3. Test cleanSteps/parseSteps round-trip and bounds — the only thing protecting stored playbook content
- **Severity**: High
- **Category**: edge-case
- **File**: src/lib/db/playbooks.ts:30-46 (`parseSteps` :30, `cleanSteps` :40)
- **Scenario**: A regression raises or removes the `.slice(0, 20)` step cap or the `.slice(0, 300)` per-step cap, or `parseSteps` stops filtering non-strings — letting a caller persist a 10k-element array or a giant blob (storage bomb), or letting non-string junk into `steps[]` that later renders unescaped. Or `parseSteps` throws on malformed stored JSON instead of returning `[]`, 500-ing the list route for the whole org.
- **Root cause**: These two functions are the **single place** `steps` is (de)serialized and bounded (per the module header), yet neither has a test. The cap is a defense-in-depth control with no regression net.
- **Impact**: Unbounded free-text storage from authenticated members (DSQL row-size / cost blast) and a brittle list endpoint that one bad row can break for an entire org.
- **Fix sketch**: Pure unit tests (no DB): `cleanSteps(Array(50).fill("x"))` → parsed length is **20**; `cleanSteps(["  a  ", "", 5, null])` → `["a"]` (trim + drop empties/non-strings); a 400-char step is truncated to **300**; `parseSteps("not json")` and `parseSteps('{"a":1}')` both return `[]`; `parseSteps(cleanSteps(["a","b"]))` round-trips to `["a","b"]`. Invariant: *stored steps are always a bounded `string[]`, regardless of input or corrupted storage*.

## 4. Test playbookStarterFile — the exact artifact committed into customer repos — and a "success-theater" gap in the brief test
- **Severity**: High
- **Category**: success-theater
- **File**: src/lib/org/playbook-brief.ts:35-53 (`playbookStarterFile`); existing test src/lib/org/playbook-brief.test.ts:1-24
- **Scenario**: The starter file's empty-steps fallback (line 41) or its checklist rendering (`- [ ] ${s}`, line 40) regresses, and the route commits a malformed/empty `docs/playbooks/<slug>.md` into a customer's PR — but tests stay green because `playbookStarterFile` is **never tested**. The existing suite tests only `playbookMarkdown` and asserts only *presence* of strings (`toContain`), giving false confidence that "the brief module is covered."
- **Root cause**: `playbookStarterFile` is single-sourced between the UI "Preview starter" (PlaybookCard.tsx:153) and the committed PR body (apply/route.ts:67) — the comment says they must be "identical" — but no test pins its output. The brief test's coverage of the *other* function masks the gap.
- **Fix sketch**: Add cases to `playbook-brief.test.ts`: with steps, output contains `- [ ] lint` checklist lines and the `> Org standard for **D5 · CI/CD**` header; with `steps: []`, output contains the `<!-- TODO: break this playbook into concrete steps -->` fallback line and **no bare `- [ ]` with empty text**; with empty `summary`, the summary paragraph is absent. Invariant: *the committed artifact is well-formed Markdown for both populated and empty playbooks* — the same single source the preview shows.

## 5. Test the [id] PATCH/DELETE per-row gate + version-bump branch
- **Severity**: Medium
- **Category**: error-branch
- **File**: src/app/api/org/playbooks/[id]/route.ts:17-70 (gate :17, PATCH :24, DELETE :59); version logic in db at playbooks.ts:104-106
- **Scenario**: A regression flips DELETE from admin-gated (`gate(id, "admin")`, line 61) to member-gated, letting any member destroy the org's standards; or the `P2025`→404 mapping (lines 54/67) breaks and a missing-playbook edit 500s; or the content-vs-archive version-bump rule (`updatePlaybook`, playbooks.ts:105) stops bumping `version` on a content edit, silently breaking the adoption "v2 / repo on older version" signal (`appliedVersion`, playbooks.ts:157).
- **Root cause**: The per-row authorization (`getPlaybookOrgSlug` → `requireOrgAccess`/`requireOrgRole`) and the version-bump invariant are untested. DELETE's admin requirement is a privilege boundary asserted only by code comment.
- **Fix sketch**: Route test: PATCH/DELETE with `getPlaybookOrgSlug` → null returns **404** and never calls `updatePlaybook`/`deletePlaybook`; DELETE calls `requireOrgRole(org, "admin")` (assert the *admin* arg, not "member"); a thrown `{ code: "P2025" }` maps to 404, any other error to 500. DB unit test: `updatePlaybook(id, { title })` issues `version: { increment: 1 }`; `updatePlaybook(id, { archived: true })` does **not** bump version. Invariant: *destructive delete requires admin, and only content edits advance the change-history version*.
