# Test Mastery — Repositories & Segments

The context's *pure* helpers are well-covered (`normalizeSegmentName`, `normalizeColor`, `buildSegmentComparison` all have honest, real-invariant tests in `segments.test.ts`). But every byte of the DB-write and route layer above them is untested: zero tests reference `setRepoSegment`, `setRepoSegmentsBulk`, `getRepoSegmentMap`, `compareSegments`, `getSegmentOrgSlug`, `createSegment`, `deleteSegment`, `updateSegment`, or any of the four `/api/org/segments` routes. The risk lives exactly one layer above where the tests stop — in the tenant-isolation gates and the idempotent membership writes. The route-test pattern already in the repo (mock `@/lib/db` + `@/lib/authz`, assert orchestration — see `src/app/api/org/import/route.test.ts`) and the db-helper pattern (mock `getPrisma`, assert the `where` fragment — see `members.test.ts`) make all of the below cheap to add without a real database.

## 1. Pin the cross-tenant org-scoping of repo tagging (single + bulk)
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/lib/db/segments.ts:109 (`setRepoSegment`), :145 (`setRepoSegmentsBulk`)
- **Scenario**: A member of org A passes `{ org: "A", fullName: "victim/repo", member: true }` to `POST /api/org/segments/<B's-segment-id>/repos`. `requireOrgAccess("A")` passes (caller is a member of A). The *only* thing stopping A from writing a `RepoSegment` row into org B's segment is the internal `prisma.segment.findFirst({ where: { id: segmentId, orgId } })` guard returning a segment. If a refactor drops the `orgId` from that `where`, or someone "optimizes" `setRepoSegment` to look the segment up by id alone, the cross-tenant write opens silently and no test fails.
- **Root cause**: The route gates on the *client-supplied* `body.org` (not on the segment's true owner, unlike the PATCH/DELETE route which derives the org via `getSegmentOrgSlug`). So the entire tenant boundary rests on the `{ id, orgId }` compound filter inside the db function — and that filter is never asserted anywhere.
- **Impact**: Cross-tenant data corruption: one customer tags/untags another customer's repos into their segments, polluting fleet rollups, comparisons, and the security/audit views that segment-scope. A silent IDOR in a multi-tenant SaaS.
- **Fix sketch**: In a `segments.test.ts` block, mock `getPrisma()` to return a stub whose `segment.findFirst` returns `null` when the supplied `orgId` doesn't match. Assert: (a) `setRepoSegment("A", segBelongingToB, "r", true)` resolves `false` and never calls `repoSegment.upsert`; (b) on the happy path the `segment.findFirst` and `repository.findUnique` calls both carry the resolved `orgId` (capture the args and assert `where.orgId` is present). Mirror for `setRepoSegmentsBulk` → returns `-1` (so the route 404s) and never calls `createMany`/`deleteMany`. Invariant: *no membership write occurs unless both the segment AND the repo resolve under the caller's orgId.*

## 2. Test the segment-scoped rollup actually filters to the segment's repos
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/lib/db/segments.ts:255 (`summarizeSegment`) → src/lib/db/org-rollup.ts:147 (`getOrgRollup`) via src/lib/db/org-shared.ts:14 (`segmentScope`)
- **Scenario**: `summarizeSegment` / `compareSegments` reduce a segment to its headline maturity by calling `getOrgRollup(orgSlug, undefined, seg.id)`, which threads `seg.id` into `segmentScope()` → `{ segments: { some: { segmentId } } }`. If `segmentScope` ever returns `{}` for a non-null id (e.g. a bad ternary edit), or `getOrgRollup` stops forwarding the `segmentId` arg, *every segment silently reports the whole-fleet average* — and the segment-vs-segment comparison shows two identical columns with a zero delta. That looks like "platform and legacy are equally mature," which is precisely the business insight the feature exists to surface.
- **Root cause**: `segmentScope` is a one-liner with no test, and the segment→rollup wiring (the single source of truth for every per-segment number on the page) has no test asserting the filter is applied. `buildSegmentComparison` is tested in isolation but is fed hand-built summaries, so it can't catch an unscoped upstream.
- **Impact**: Comparison/rollup theater: leadership reads "no difference between segments" and mis-allocates investment. A whole analytics surface becomes confidently wrong with no failing test.
- **Fix sketch**: Unit-test `segmentScope` directly: `segmentScope(null)` / `segmentScope(undefined)` → `{}`; `segmentScope("s1")` → `{ segments: { some: { segmentId: "s1" } } }` (this is LLM-generatable and pins the exact Prisma fragment). Then a focused `getOrgRollup` test (mock `getPrisma`) capturing the `repository.findMany` args and asserting `where.segments.some.segmentId === "s1"` when a segmentId is passed, and that the key is *absent* when it isn't. Invariant: *a non-null segmentId always narrows the repo query; a null one never does.*

## 3. Assert the membership-write idempotency and count contract
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/lib/db/segments.ts:145 (`setRepoSegmentsBulk` return value), :126 (`setRepoSegment` upsert)
- **Scenario**: The bulk route returns `changed` to the client, and the optimistic UI in `RepoSegmentsPanel`/`RepoLeaderboard` bumps `repoCount` by its own count. The db function's contract is precise and load-bearing: dedup `fullNames` (`new Set`), drop non-strings, return `0` (not `-1`) when the list is empty or no repos match, return `-1` *only* when the segment isn't the org's, and use `skipDuplicates` so re-tagging an already-member repo is a no-op. A refactor that returns `res.count` without dedup, or conflates the empty-list `0` with the not-found `-1`, would either inflate counts or make the route 404 on a legitimately empty selection — with no test to catch it.
- **Root cause**: The `0` vs `-1` vs `res.count` tri-state is the route's branching contract (`changed < 0 ⇒ 404`) and it is asserted nowhere; `skipDuplicates` idempotency is asserted nowhere.
- **Impact**: Wrong repo counts in the UI (already drifted by the optimistic update), spurious 404s on valid bulk operations, or duplicate-key errors if `skipDuplicates` is dropped — eroding trust in the tagging feature.
- **Fix sketch**: With `getPrisma` mocked: assert `setRepoSegmentsBulk` returns `0` for `[]`, for all-non-string input, and for "no repos matched"; returns `-1` when `segment.findFirst` → null; returns `res.count` from `createMany` (and that `createMany` is called with `skipDuplicates: true`) on add, and from `deleteMany` on remove. Add a single dedup assertion: `["a","a","b"]` collapses to two repo lookups. Invariant: *return is `-1` iff the segment isn't the org's, `≥0` is the exact rows changed, and duplicate tags are skipped not errored.*

## 4. Gate-test the four segment routes (auth + per-row tenant resolution)
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/app/api/org/segments/[id]/route.ts:14 (`gate` → DELETE requires admin), src/app/api/org/segments/[id]/repos/route.ts:19, .../repos/bulk/route.ts:22
- **Scenario**: DELETE deliberately requires the **admin** role (`gate(id, "admin")`) while PATCH is a member-level write, and the gate derives the org from the segment id via `getSegmentOrgSlug` (404 on unknown id). If someone "simplifies" DELETE to `gate(id)` (member), any member can destroy segments and their membership rows. Likewise, if the 503/400 ordering in the repos routes is reshuffled, an unconfigured DB or a missing `fullName` could fall through to a write. None of this is tested.
- **Root cause**: No route tests exist for this context. The differentiated authz (member PATCH vs admin DELETE), the `getSegmentOrgSlug` → 404 path, and the `requireOrgAccess` denial short-circuit are pure orchestration that's easy to regress and currently unguarded.
- **Impact**: A privilege downgrade on the destructive DELETE path (any member nukes another team's segments) or an auth bypass on the write routes — both ship green.
- **Fix sketch**: Following `import/route.test.ts`: mock `@/lib/authz` and `@/lib/db`. Assert (a) DELETE calls `requireOrgRole(org, "admin")` (or that a non-admin denial Response is returned and `deleteSegment` is never called); PATCH uses the member gate. (b) Both repos routes return the `requireOrgAccess` denial unchanged and never call the db write when denied. (c) `gate` returns 404 when `getSegmentOrgSlug` → null. (d) P2002/P2025 from the db map to 409/404 in POST/PATCH. Invariant: *destructive delete is admin-gated; every write is preceded by a passing org gate; unknown segment ⇒ 404, never a write.*

## 5. Cover `getRepoSegmentMap` inversion + sorted, deduped output
- **Severity**: Medium
- **Category**: edge-case
- **File**: src/lib/db/segments.ts:176 (`getRepoSegmentMap`)
- **Scenario**: `getRepoSegmentMap` builds `fullName → [{id,name,color}]` and is consumed two ways that both depend on its exact shape: the repositories page inverts it into `membership` for the tagging chips, and the segments page *re-inverts* it into `reposBySegment` to drive per-segment scan/schedule actions. The function sorts each repo's segments by `name.localeCompare`. A repo tagged into zero segments must be absent (not an empty array surprising the consumers), and a repo in N segments must list all N, deduped. None of this row→map reduction is tested; a regression (e.g. `||=` vs overwrite, or dropping the sort) would scramble the chip order and the inverted action targets.
- **Root cause**: Pure-ish reduction logic over Prisma rows, untested. The two consumers silently trust the contract.
- **Impact**: Wrong segment chips per repo, or segment "scan this slice" actions firing against the wrong repo set — a confusing but not catastrophic data-shape bug.
- **Fix sketch**: Mock `getPrisma().repoSegment.findMany` to return flat rows (`{ repo:{fullName}, segment:{id,name,color} }`) covering: a repo in 2 segments (assert both present, sorted by name), two repos sharing a segment (assert the segment appears under each fullName), and assert the query carries `where.segment.orgId`. Invariant: *each fullName maps to its complete, name-sorted segment list, and the query is org-scoped.*
