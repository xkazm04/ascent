# Test Mastery — Roadmap & Recommendation Tracking

This context (the prioritized recommendation backlog: list, status/assignee/due-date edits, the per-item activity timeline, and the in-transaction audit row) has **zero tests on every file it owns** — `src/lib/db/scans-recommendations.ts`, all three `/api/recommendations` routes, `toPersistedRec` in `scans-shared.ts`, and the `RecommendationTracker` UI. (The pure scoring math the `RoadmapSandbox` consumes — `projectSandbox` / `cheapestPathToNextLevel` / `projectedGain` — *is* well covered in `engine.test.ts`, which belongs to context 3, so it is correctly excluded below.) The risk lives in the mutation/audit/tenant-gate layer that has no coverage at all.

## 1. Pin the recommendation mutation transaction: atomicity, audit tenant-scope, and change-detection
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/lib/db/scans-recommendations.ts:43 (`updateRecommendation`), esp. lines 78-124
- **Scenario**: A refactor changes the order or contents of the `$transaction` block (lines 102-121) — e.g. someone re-introduces the old best-effort post-transaction `recordAudit`, or computes `orgId` lazily and forgets to pass it. The row update + events still commit, so the backlog UI looks fine, but the audit row is written with `orgId: null` (or not at all). `getAuditLog` filters `where: { orgId }`, so the compliance trail for every backlog mutation silently disappears — exactly the gap the in-transaction audit was added to close. Nothing fails; the regression ships green.
- **Root cause**: The function bundles four invariants that are each easy to break and have no test: (a) the row update, the `recommendationEvent.createMany`, and the `auditLog.create` all commit in **one** transaction; (b) the audit row carries the recommendation's resolved `orgId` (Recommendation→Scan→Repo→org) so it's readable; (c) an event is appended **only** for a field that actually changed; (d) a fully no-op patch writes nothing and returns the current row. None is asserted anywhere.
- **Impact**: Backlog status/assignee/due-date changes are part of the audited "who did what" product. A silent audit-write regression is a compliance/data-integrity failure on a money-adjacent enterprise feature; a broken change-detector spams the timeline (or drops real history), corrupting the activity trail teams plan against.
- **Fix sketch**: Add `src/lib/db/scans-recommendations.test.ts` using the repo's established `vi.hoisted` + `vi.mock("@/lib/db/client")` fake-prisma pattern (see `credits.test.ts`). With a fake `$transaction` that records the calls made against `tx`, assert: (1) on a status change, exactly one `recommendationEvent` row AND one `auditLog.create` are written **inside** the same `tx`, and the audit row's `data.orgId` equals the org resolved from the recommendation chain (not null); (2) `auditLog.meta` parses to `{ id, actor, changes: [{ kind:"status", from, to }] }`; (3) a patch where `status === current.status` writes **no** event and **no** audit row and returns `toPersistedRec(current)` (assert `tx.recommendation.update` never called); (4) a multi-field patch where only `targetDate` differs by calendar day produces exactly one `target_date` event (invariant: `events.length` === count of fields whose value-key actually changed).

## 2. Lock the per-row tenant gate: cross-tenant IDOR + public-funnel poisoning must stay closed
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/app/api/recommendations/[id]/route.ts:34-51 (PATCH gate); src/app/api/recommendations/[id]/events/route.ts:22-28 (GET gate)
- **Scenario**: A maintainer "simplifies" the PATCH handler — drops the `getRecommendationOrgSlug(id)` lookup and authorizes on "is signed in", or removes the `org === PUBLIC_ORG → 403` block. Now any signed-in user can mutate **another tenant's** backlog (status/assignee/due-date) and write to its audit log by guessing/lifting a recommendation id, and anyone can poison every public scan's shared recommendation state. The comment block at lines 35-48 documents these exact attacks, but no test enforces the gate, so the regression is invisible.
- **Root cause**: The authorization decision is order-dependent and subtle (resolve owning org from the **row**, 404 on unknown, 403 on `PUBLIC_ORG`, then `requireOrgAccess(org)`), and it is duplicated across PATCH and the events GET (which must use `requireOrgRead`). None of these branches is tested. There is also no test that an unknown id returns 404 (not 500/200).
- **Impact**: Cross-tenant write IDOR on the backlog plus audit-log poisoning is a direct security/data-integrity breach across org boundaries — the highest-severity class for a multi-tenant SaaS.
- **Fix sketch**: Add `src/app/api/recommendations/[id]/route.test.ts` (mirror `scan/route.test.ts`: stub `next/server.NextResponse`, mock `@/lib/db`, `@/lib/auth`, `@/lib/authz`). Assert, for PATCH: (a) unknown id (`getRecommendationOrgSlug` → null) ⇒ 404 and `updateRecommendation` never called; (b) org resolves to `"public"` ⇒ 403 and no mutation; (c) `requireOrgAccess` returns a denial Response ⇒ that exact Response is returned and `updateRecommendation` never called; (d) authorized ⇒ `updateRecommendation` is called with the parsed patch. Add a sibling `events/route.test.ts` asserting the GET path uses `requireOrgRead` (not `requireOrgAccess`) and 404s an unknown id. Invariant: **no mutating/reading code path executes before the owning-org gate passes.**

## 3. Test the PATCH body validators for FAILURE — they are the only thing keeping junk out of the row
- **Severity**: High
- **Category**: error-branch
- **File**: src/app/api/recommendations/[id]/route.ts:59-104
- **Scenario**: The `targetDate` regex/`Date.parse` guard (lines 86-93) is loosened back to "Date.parse-able" during a tidy-up, so `"June 9 2026"`, a full ISO datetime, or `"2026/06/09"` are accepted and stored verbatim — violating the documented `YYYY-MM-DD` contract and silently corrupting due-date history and the `target_date` event `to` values. Likewise the `assigneeLogin` shape check (`/^[A-Za-z0-9-]{1,39}$/`, line 75) could be dropped, turning the field into free-text storage; or the "at least one field" guard (line 97) could regress so an empty patch writes a no-op. None is tested.
- **Root cause**: These are pure request-validation branches with rich, easy-to-LLM-generate boundary cases, but the route has no test file, so every rejection branch (invalid status, invalid login, `"2026-13-45"` calendar-invalid date, empty patch) is unverified. The status check additionally guards against writing an out-of-enum `status` straight into the DB.
- **Impact**: Bad data reaching the `Recommendation` row degrades the backlog UI (an unparseable due date), the diff/compare views, and the audit `meta` — and an over-permissive `assigneeLogin` is a stored-text injection surface.
- **Fix sketch**: In the route test from finding #2, add a table-driven block of pure validation cases asserting 400 + `updateRecommendation` NOT called for: `status:"foo"`; `assigneeLogin:"has space"`, `assigneeLogin:"a".repeat(40)`; `targetDate:"2026/06/09"`, `"June 9 2026"`, `"2026-13-45"`, `"2026-02-30"`; and an empty `{}` body. Plus accept-cases: valid `status`, `assigneeLogin:null` (clears), `targetDate:null` (clears), `targetDate:"2026-06-09"`. Invariant: **the stored patch is exactly the validated subset and the `YYYY-MM-DD`-or-null / GitHub-login-or-null contracts hold at the boundary.**

## 4. Cover `toPersistedRec` — the corrupt-data firewall on every recommendation read
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/lib/db/scans-shared.ts:181-214 (`toPersistedRec`)
- **Scenario**: The `explore` JSON parsing (lines 194-200) is "simplified" to `JSON.parse(r.explore)` without the try/catch + array-guard + `typeof === "string"` filter. A row whose `explore` column is malformed, `null`, a JSON object, or an array containing a number/`null` (hand-edited, written by an older path, or from an LLM that emitted mixed types) now throws or returns non-string entries — crashing **both** the API list route and `updateRecommendation`'s return mapping, or shipping `[null, 42]` into the report UI. The `targetDate` → `YYYY-MM-DD` slice (line 212) is similarly load-bearing and untested.
- **Root cause**: `toPersistedRec` is a pure mapper shared by the read path (`getLatestRecommendations`) and the write path (`updateRecommendation`), making it the single normalization choke point — and it has no test. Its whole reason to exist (tolerate corrupt persisted JSON without crashing) is exactly the behavior nobody verifies.
- **Impact**: One bad `explore` row blanks the recommendation list and breaks every edit on that scan — a data-integrity failure that scales with corpus age and any future LLM/import path that writes the column.
- **Fix sketch**: Add `src/lib/db/scans-shared.test.ts` (pure, no mocks — LLM-generatable). Assert: `explore` of `"[]"`, valid `["a","b"]`, malformed `"{not json"`, `'{"a":1}'` (object), and `'["ok", 1, null]'` each yield a **string-only** array (the last → `["ok"]`), never a throw; `explore: undefined` → `[]`; a `targetDate` `Date` maps to a `YYYY-MM-DD` string and `null` → `null`; `levelUnlock: null` → `undefined`. Invariant: **a corrupt persisted row degrades to a safe shape, never a throw or a non-string `explore` entry.**

## 5. Verify the timeline read order and the tracker's per-row optimistic rollback
- **Severity**: Medium
- **Category**: edge-case
- **File**: src/lib/db/scans-recommendations.ts:153-168 (`getRecommendationEvents`); src/components/report/RecommendationTracker.tsx:75-120 (`setStatus` rollback)
- **Scenario (a)**: The `orderBy: [{ createdAt: "desc" }, { id: "desc" }]` tiebreak (line 157) is dropped during a refactor; two events created in the same millisecond (a multi-field patch writes several rows in one `createMany`) return in arbitrary order, so the activity timeline shows due-date-then-status when it was status-then-due-date — a quietly wrong audit narrative. **Scenario (b)**: The tracker's targeted rollback (lines 81-89) is reverted to a whole-list `setItems(prev)` snapshot; when two rows are saved concurrently and one PATCH fails, the failure clobbers the **other** row's already-confirmed optimistic change — silent UI data loss.
- **Root cause**: The newest-first + stable-tiebreak ordering and the "roll back only this row" behavior are deliberate (comments call them out) but untested. The component logic is pure enough to extract; per the harness note, ascent's Vitest has no jsdom, so DOM-bound assertions belong in Playwright while the rollback reducer should be unit-tested as a pure function.
- **Impact**: A misordered timeline misleads anyone auditing who-changed-what; a rollback regression loses a user's confirmed status edit on an unrelated row — eroding trust in the tracking surface.
- **Fix sketch**: (a) In `scans-recommendations.test.ts`, mock `recommendationEvent.findMany` and assert it is called with `orderBy: [{ createdAt:"desc" }, { id:"desc" }]` and that the mapped result preserves that order and ISO-formats `at`. (b) Extract the row-merge/rollback into a pure helper (`applyOptimistic(items, id, status)` / `rollbackRow(items, id, priorStatus)`) and unit-test that rolling back row A leaves a concurrently-changed row B untouched; cover the DOM interaction (failed PATCH → row reverts, Retry re-fires) with a Playwright e2e rather than a React render test. Invariant: **a failed save reverts only its own row, and the timeline is strictly newest-first with a deterministic same-instant tiebreak.**
