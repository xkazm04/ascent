# Test Mastery — LLM Provider Abstraction

The LLM-provider primitives (`validateAssessment`, `isAssessmentUsable`, `providerByName`, `providerAvailable`, `priceForModel`, `parseJsonLoose`) are well unit-tested in isolation. The honesty contract those primitives exist to enforce — "either genuinely score, or VISIBLY degrade; never serve mock scores under a real provider's name" — lives in the **orchestration** in `src/lib/scan.ts` (lines 197–296), and that wiring is the gap. Every unit test asserts a function's local behavior; nothing asserts that the scan actually calls the gate, attributes tokens only on success, or surfaces the `llmFailed` caveat. Three of the six providers (`gemini`, `openai`, `claude-cli`) and `mock` have no test at all, and the model-output-injection hardening guards are untested.

---

## 1. Pin the assessment-usability gate + degradation honesty in the scan orchestration
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/lib/scan.ts:197-296 (consumes src/lib/llm/provider.ts:205 `isAssessmentUsable`)
- **Scenario**: A refactor of the `attemptAssess` / failover loop drops or inverts the `if (p.name !== "mock" && !isAssessmentUsable(a, signals.length))` gate at scan.ts:212 — or stops setting `llmFailed`/emitting the `fallback:true` event at lines 278-291. A model that returns `{}` or all-unknown dimension ids now coerces (via `validateAssessment`, which never throws) to a near-empty assessment, passes straight through, and the report renders the **deterministic signal floor under "gemini"/"bedrock" with no "AI was unavailable" warning**. `provider.test.ts` still passes (it tests `isAssessmentUsable` in isolation); the regression ships green.
- **Root cause**: `scan.test.ts` only covers SHA-drift threading and runs exclusively with `mock: true`, so the primary→retry→`LLM_FALLBACK_PROVIDER`→mock chain, the usability gate, and the `llmFailed` warning/`fallback` SSE event are never exercised. The unit tests prove the bricks; nothing proves the wall is built from them. This is the exact success-theater class the whole module was hardened against (comments at provider.ts:194-204 and scan.ts:197-201 describe the bug — but only in prose).
- **Impact**: Customers (incl. the enterprise Bedrock path) silently shown deterministic scores branded as AI analysis, and the gate that prevents it has no integration test guarding it. Directly undermines the product's core claim and the calibration credibility of every gated CI verdict downstream.
- **Fix sketch**: Add `scan.test.ts` cases with a stub `LLMProvider` injected as primary: (a) primary returns a 0-dimension/empty-coverage assessment → assert the returned report's `engine`/provider is **mock**, `report.warnings` contains the "AI analysis was unavailable" string, and a `fallback:true` progress event was emitted; (b) primary throws once then a fallback provider returns a full assessment → assert the report's engine is the **fallback** provider, no `llmFailed` warning; (c) `intendedProvider === 'mock'` (keyless) → assert **no** spurious "AI unavailable" warning. Invariant: a report can only carry a real provider's name if that provider returned a `>= MIN_ASSESSMENT_COVERAGE` assessment.

## 2. Cover the usage/token-attribution path — bill only the provider that actually scored
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/lib/scan.ts:204-218, 312 (`attemptUsage` → `capturedUsage` → `report.usage`)
- **Scenario**: A change to `attemptAssess` commits `capturedUsage = attemptUsage` **before** the usability check (or moves the `onUsage` capture), so a failed primary attempt (malformed JSON, sub-threshold coverage) that already fired `onUsage` leaves its input/output tokens on `report.usage` even though the scan degraded to mock. The user is **billed for an attempt that contributed nothing**, or a `bedrock→gemini` failover double-counts both attempts' tokens. The comment at scan.ts:206-209 documents this exact hazard; no test guards it.
- **Root cause**: Token attribution is money-path logic with zero coverage. `priceForModel` (config.test.ts) prices a model, but nothing tests that the *right* token counts reach `report.usage` — i.e. only the winning attempt's, latency always stamped, mock/keyless = no tokens.
- **Impact**: Over-billing on every degraded or failed-over scan; the credit-estimate and usage panels inherit the inflated numbers. Silent revenue/trust bug.
- **Fix sketch**: Stub a primary provider whose `assess` calls `onUsage({inputTokens:1000,outputTokens:500})` then returns an unusable assessment, and a fallback that calls `onUsage({inputTokens:10,outputTokens:5})` and returns a good one. Assert `report.usage.inputTokens === 10` (only the winner's), `outputTokens === 5`, and `report.usage.latencyMs` is present. Add a degrade-to-mock case asserting `report.usage` carries **no** input/output tokens. Invariant: persisted usage equals exactly the tokens of the provider whose assessment was used.

## 3. Test the model-output hardening guards in validateAssessment (field caps, dup/oversize arrays, level-unlock sanity)
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/lib/llm/provider.ts:59-88 (`cap`, `asStringArray`, `validLevelUnlock`), 101-184
- **Scenario**: A model emits a hallucinated/prompt-injected reply: a multi-megabyte `summary`/`headline`, a million-element `strengths` array, a hostile array of thousands of valid-id **duplicate** dimensions, or a `levelUnlock` of `"L5->L7"` (out of range) / `"L3->L2"` (a downgrade). Today `cap` (2000-char bound), the `slice(0, max*4)` pre-slice, the `DIMENSIONS.length*4` dimension de-dupe, and `validLevelUnlock`'s advance-only normalization all defend against these — but `provider.test.ts` only asserts the score-coercion branch. A refactor that drops `cap`, removes the input pre-slice (reintroducing O(n) allocation on a hostile array), or loosens the level-unlock regex bloats the DB row / SSE payload / UI and lets a downgrade reach the user-facing roadmap, all with the suite green.
- **Root cause**: The defensive coercion that exists *specifically* to survive adversarial/verbose model output is largely unverified; only the numeric-score branch has tests. These are pure functions over arbitrary JSON — the cheapest possible LLM-generatable test batch, and the one closing the largest untested surface.
- **Impact**: DB/SSE/UI bloat (DoS-ish persistence cost) and confidently-wrong roadmap content (a "fix" that *lowers* maturity) shipped to users.
- **Fix sketch**: Add cases asserting: `validateAssessment({headline:'x'.repeat(5000), dimensions:[{id:'D1',score:50,summary:'y'.repeat(5000)}]})` → headline and summary length `<= 2000`; a `strengths` array of 10_000 entries → result length `<= 6`; `dimensions` with 1000 `{id:'D1',score:50}` dups → result has exactly one `D1` (first wins); `validLevelUnlock('L5->L7')` and `('L3->L2')` and `('L3→L2')` → `undefined`, `('L2→L4')` → `'L2->L4'`. Invariants: every string field `<= MAX_FIELD_LEN`, every array `<= max`, dimensions unique, levelUnlock only ever a strict in-range advance.

## 4. Cover priceForModel's bare CLI aliases and the longest-prefix tie-break that prevents mispricing
- **Severity**: Medium
- **Category**: edge-case
- **File**: src/lib/llm/config.ts:48-75 (`MODEL_PRICES` `sonnet`/`haiku`/`opus` aliases + `priceForModel`)
- **Scenario**: A claude-cli scan persists `engineModel: "sonnet"` (or `"haiku"`/`"opus"`). `config.test.ts` only asserts the three providers' *default* model constants price, the Bedrock geo-strip, and the `gpt-4o-mini` vs `gpt-4o` tie-break — it never tests the bare CLI aliases. A future reorder/edit of `MODEL_PRICES` (e.g. adding `anthropic.claude-sonnet-4` length precedence interacting with the bare `sonnet` prefix, or dropping an alias) silently makes a claude-cli scan return `null` ("no estimate") or, worse, match the wrong tier — a confidently-wrong bill.
- **Root cause**: The CLI-alias pricing rows have no direct assertion, and the longest-prefix rule is only spot-checked on one OpenAI pair, not on the case where a short bare alias and a long `anthropic.` prefix could both be candidates.
- **Impact**: claude-cli (a first-class local/eval provider) scans show no cost estimate or a mistier'd one in the usage panel — a quiet metering gap on a real provider.
- **Fix sketch**: Assert `priceForModel('sonnet')`, `('haiku')`, `('opus')` each non-null with the documented in/out rates, and `priceForModel(DEFAULT_CLAUDE_MODEL)` is non-null (derive from `claude-cli.ts` like config.test.ts already does for the other defaults). Add a tie-break assertion that `priceForModel('anthropic.claude-sonnet-4-6')` matches the `anthropic.claude-sonnet-4` row, not a shorter accidental match. Invariant: every shipped default/alias prices, and the longest matching prefix always wins.

## 5. Assert parseJsonLoose's recovery-cost guards (oversize reply + structural-start cap) stay enforced
- **Severity**: Low
- **Category**: determinism
- **File**: src/lib/llm/json.ts:32-33, 88-90, 116-132 (`MAX_RECOVERY_BYTES`, `MAX_START_ATTEMPTS`, `balancedParse`)
- **Scenario**: A truncated/adversarial reply full of unclosed `{` (no valid JSON) reaches `parseJsonLoose`. The balanced recovery is O(starts × N) and synchronous (uninterruptible by the per-request AbortSignal), so the guards exist to bound it: throw above 256KB before recovery, and cap structural-start attempts at 512. `json.test.ts` covers fences/prose/arrays/escapes but never the size cap or the attempt cap, so a refactor that removes `MAX_RECOVERY_BYTES`/`MAX_START_ATTEMPTS` reintroduces an event-loop stall on a hostile reply with the suite green.
- **Root cause**: The DoS-bounding behavior is untested — only the happy-path extraction is.
- **Impact**: A single malformed model reply could pin the single-threaded server during a scan; low likelihood/blast radius but a determinism/availability regression that would slip silently.
- **Fix sketch**: Assert `parseJsonLoose('{'.repeat(300_000))` throws `ProviderParseError` whose message names the byte size (proves the >256KB short-circuit, not a slow scan). Assert a reply of many thousands of bare `{` under the size cap still throws (proves the attempt cap returns rather than scanning all starts). Optionally bound the test with `vi.useFakeTimers`/a wall-clock assertion. Invariant: an unparseable reply fails fast and bounded, never O(n²) over the whole input.
