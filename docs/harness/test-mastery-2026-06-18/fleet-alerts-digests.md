# Test Mastery — Fleet Alerts & Digests

The pure layer of this context (`detectRegression`, `buildRegressionMessage`, `buildFleetDigestMessage`, `buildLowCreditsMessage`, `creditsAlertThreshold`, `isLowCreditsCrossing`, `resolveAlertWebhook`, `validateAlertWebhookUrl`) is genuinely well-tested in `src/lib/alerts.test.ts` — failure cases, blank-vs-zero env rules, and SSRF private-range rejection are all asserted. The risk lives exactly one layer above where the tests stop: **the orchestration that decides whether to alert in production, the cron digest that pushes tenant data to external sinks, the admin config route, and the `dispatchAlert` side-effect itself are all completely untested.** Every finding below is in that untested band.

## 1. Cover the regression-alert orchestrator (`checkAndAlertRegression`), the only code that decides whether to interrupt a human
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/lib/scan-alerts.ts:53 (no `src/lib/scan-alerts.test.ts` exists)
- **Scenario**: `checkAndAlertRegression` is wired into 5 production scan paths (sync `/api/scan`, GitHub-App webhook re-scan, cron rescan, org import, org scan). It is the function that, in production, turns a diff into an audit row + a dispatched alert. It has no test. A regression that breaks the "missing `prev` → no-op", the per-org threshold override merge (lines 63-67), or the `isAlertConfigured(webhookUrl)` gate (line 85) would ship silently — e.g. an inverted gate that POSTs even when no sink resolves, or an override merge that uses `??` wrong and reverts every org to defaults so a tenant's tightened `overallDrop` never fires.
- **Root cause**: The pure detector below it is exhaustively tested, which creates false confidence that "alerts are tested." The glue that consumes it — the actual decision point — is not. `dispatchAlert` and `getOrgAlertThresholds` are real I/O, so this layer was skipped.
- **Impact**: A silent break here means real fleet regressions stop alerting (the product's core "live intelligence" promise) OR one tenant's regression alert is POSTed to another tenant's channel — a cross-tenant data leak with no test guarding it.
- **Fix sketch**: Add `scan-alerts.test.ts` mocking `@/lib/db` (`getOrgAlertThresholds`, `getOrgAlertWebhook`, `recordAudit`) and `@/lib/alerts` `dispatchAlert`. Assert: (a) `prev=null` ⇒ `{regressed:false, dispatched:false}` and `dispatchAlert` NOT called; (b) a non-regressing diff records no audit and dispatches nothing; (c) a regressing diff with an org webhook resolved calls `dispatchAlert` with exactly that `webhookUrl` and records a `scan.regression` audit; (d) an org threshold of `{overallDrop:2}` flips a −3 overall diff from non-regression to regression (invariant: per-org override beats `DEFAULT_THRESHOLDS` per-field).

## 2. Assert the regression alert never throws into the scan path — including when `recordAudit` fails
- **Severity**: Critical
- **Category**: error-branch
- **File**: src/lib/scan-alerts.ts:71 (the `recordAudit(...)` call is NOT `.catch()`-wrapped, unlike `orgWebhook`/`getOrgAlertThresholds`)
- **Scenario**: The module's contract (file header + every docstring) is "Never throws — alerting must not fail the scan." But inside the `try`, the `recordAudit` call on line 71 is `await`ed without a `.catch`. It happens to be caught by the outer `try/catch`, so a `recordAudit` throw is swallowed — BUT swallowing it means the **alert is never dispatched** (the `return` after the catch is `{dispatched:false}`), and there is no test pinning either property. If a future refactor moves dispatch before audit, or removes the outer catch, a flaky audit write would fail the whole scan. Today, a flaky audit write silently suppresses the alert. Neither behavior is asserted.
- **Root cause**: No test exercises the failure branches. The "never throws" guarantee and the audit-vs-dispatch ordering are load-bearing but unverified, so the difference between "audit failed, alert still sent" and "audit failed, alert dropped" is invisible.
- **Impact**: The scan pipeline's resilience guarantee is undefended. Worst case a webhook/audit hiccup fails customer scans; best case a real regression silently never alerts because an unrelated audit-table write blipped.
- **Fix sketch**: In `scan-alerts.test.ts`: (a) make `recordAudit` reject and assert `checkAndAlertRegression` resolves (does not throw) — invariant: scan path is never failed by alerting; (b) make `dispatchAlert` reject/throw and assert the same; (c) decide & pin the intended ordering — assert that a regression still dispatches even when `recordAudit` rejects (move `recordAudit` to `.catch(()=>{})` to make audit and dispatch independent, matching the `org/alerts` route convention).

## 3. Cover the cron digest route end-to-end — auth fail-closed, per-tenant routing, and partial-failure isolation
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/app/api/cron/digest/route.ts:32 (no `route.test.ts` in that dir)
- **Scenario**: This route fans tenant fleet data out to external Slack webhooks on a schedule and has zero tests. Untested invariants that each carry real blast radius: (a) **fail-closed auth** — line 33-39 returns 503 when `CRON_SECRET` is unset; a refactor reintroducing the old `if (secret)` opt-in would silently reopen an endpoint that pushes fleet data externally; (b) **per-tenant routing** — each org must receive ONLY its own digest at its OWN resolved webhook (line 59-95); an off-by-one or shared-variable bug would cross-post; (c) **partial-failure isolation** — line 96-98 catches per-org so one org's failure doesn't abort the loop; (d) `skippedNoSink` accounting vs orgs with no sink.
- **Root cause**: It's an integration route with many DB dependencies (`listOrgsWithWatchedRepos`, `getOrgRollup`, `getOrgMovers`, etc.), so it was left for "later." But the security-critical auth gate and the cross-tenant routing are exactly what a test must lock down.
- **Impact**: A reopened auth gate exposes a fleet-data-exfil endpoint to the internet. A routing bug leaks org A's maturity/regression summary into org B's channel. A loop-abort bug means one bad org silently starves every later org of its digest.
- **Fix sketch**: Add `route.test.ts` mocking `@/lib/db` and `@/lib/alerts`. Assert: unset `CRON_SECRET` ⇒ 503 and `dispatchAlert` never called; wrong bearer/key ⇒ 401; with 2 orgs each having distinct webhooks, `dispatchAlert` is called once per org with that org's `webhookUrl` (invariant: no cross-tenant POST); an org whose `getOrgRollup` throws is counted in `errors` and the OTHER org still gets `sent`; an org with no resolvable sink increments `skippedNoSink` and does NO rollup work (line 60 short-circuit before `getOrgRollup`).

## 4. Cover the `dispatchAlert` side-effect: 2xx vs non-2xx vs network throw, and the no-sink short-circuit
- **Severity**: High
- **Category**: success-theater
- **File**: src/lib/alerts.ts:317 (`dispatchAlert`; `alerts.test.ts` imports it nowhere — verified: no `fetch`/`dispatchAlert` reference in the test file)
- **Scenario**: `dispatchAlert` is the single real side-effect of the whole feature and the function whose boolean return every caller (`checkAndAlertRegression`, `maybeAlertLowCredits`, digest route, `org/alerts` test-send) trusts to mean "delivered." It is never tested. The contract it must honor: return `false` (not throw) on a non-2xx, return `false` on a network rejection, return `false` when `resolveAlertWebhook` yields null (and NOT call `fetch`), return `true` only on a 2xx, and POST a body of exactly `{text, blocks}` with `content-type: application/json`. A regression that throws on a 500 webhook, or that treats a 4xx as success, or that fires `fetch` against a null URL, would pass the entire existing suite green.
- **Root cause**: The suite tests everything *around* the I/O (URL resolution, message shape) but stubs out the one function that actually talks to the network, leaving the success/failure mapping — the thing callers branch on — unasserted. That's success-theater: a green suite that never proves an alert is actually deliverable or that a flaky webhook can't crash a scan.
- **Impact**: If `dispatchAlert` ever throws instead of returning `false`, the "never fails the scan" guarantee of every caller breaks. If a 4xx is read as delivered, admins see a green "test alert delivered" while nothing reached their channel.
- **Fix sketch**: In `alerts.test.ts`, mock global `fetch`. Assert: 200 ⇒ `true` and `fetch` called with the resolved URL, method POST, JSON body `{text, blocks}`; 500 ⇒ `false` (no throw); `fetch` rejecting ⇒ `false` (no throw); no sink (`webhookUrl` null + no env) ⇒ `false` and `fetch` NOT called (invariant: never POST without a resolved sink); `opts.signal` is threaded to `fetch`.

## 5. Pin the low-credits "fires exactly once, never skipped" crossing at the orchestrator level (multi-unit debit)
- **Severity**: Medium
- **Category**: edge-case
- **File**: src/lib/scan-alerts.ts:105 (`maybeAlertLowCredits`); helper `isLowCreditsCrossing` at src/lib/alerts.ts:237
- **Scenario**: `isLowCreditsCrossing(balanceAfter, threshold)` returns true only when `balanceAfter === 0 || balanceAfter === threshold` — and its docstring stakes correctness on the assumption "debits are unit-sized, so the balance lands EXACTLY on the threshold once." The pure helper is tested for exact-landing, but the *call sites* (org import line 201, org scan line 131, cron rescan line 113, sync scan line 155) debit per-repo in batches and could land a balance that **jumps over** the threshold (e.g. threshold 5, balance 7 → 4 in one multi-credit settle) — and then the low-credits alert silently never fires. No test asserts what happens when a debit skips the line, nor that `maybeAlertLowCredits` no-ops cleanly when no sink is configured.
- **Root cause**: The unit-sized-debit invariant is documented but unenforced; the only tests are on the pure predicate in isolation, so the gap between "predicate is correct given exact landing" and "production debits always land exactly" is untested.
- **Impact**: A prepaid org can deplete to zero without ever getting the low-water warning — the exact silent-churn moment the feature exists to prevent — if any path debits more than one unit at a time. Revenue/retention path with no guard.
- **Fix sketch**: In `scan-alerts.test.ts` mock `getOrgAlertWebhook`/`dispatchAlert`: assert `maybeAlertLowCredits(org, threshold)` dispatches once and `maybeAlertLowCredits(org, 0)` dispatches once; assert no-sink ⇒ returns `false`, no dispatch; and add a `isLowCreditsCrossing` assertion documenting the skip case (`isLowCreditsCrossing(4,5)===false`) so the unit-debit invariant is explicit — then either enforce unit debits or change the predicate to `balanceAfter <= threshold` crossing detection with caller-side dedupe. Invariant to assert: a balance transit through the low-water mark fires exactly one alert and is never silently skipped.
