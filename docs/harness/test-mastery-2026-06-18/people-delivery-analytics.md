# Test Mastery — People & Delivery Analytics

This context is three thin RSC view pages (`contributors`, `teams`, `delivery`) over a fleet-rollup data layer. The pages contain almost no logic; the business-critical behavior lives in the aggregators they call: `getContributorInsights` (`org-contributors.ts`), `getOrgPrSignals` / `getOrgGovernance` / `getOrgActivity` (`org-signals.ts`), `getOrgTeamRollup` (`org-teams.ts`), and the CSV export route at `/api/org/export`. Of these, only the team path has a real pure-function test (`teamRollup.test.ts` → `rollupTeams`). **Every contributor and delivery aggregator, plus the people-data export route, ships with zero meaningful test coverage** — verified by grepping the whole `src` tree: no `.test.ts` references `getContributorInsights`, `getOrgActivity`, `getOrgPrSignals`, `getOrgGovernance`, or `/api/org/export`. `org-insights.test.ts` covers only `percentileOf`; `org.test.ts` covers only `dueBucketFor`/`computeWindowDeltas` — neither is on this context's data path. Findings are ranked by blast radius: a tenant-isolation gap on a PII export first, then silent-corruption math in the rollups the dashboard reads.

## 1. Test the people-data CSV export's tenant gate against a non-member caller
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/app/api/org/export/route.ts:41-46
- **Scenario**: A regression that reorders, weakens, or removes `requireOrgRead(org)` (e.g. someone "simplifies" the handler, or `canReadOrg` gains an early-true branch) lets any signed-in viewer — or, with auth misconfigured, an anonymous caller — `GET /api/org/export?org=<victim>&kind=contributors&format=csv` and download every contributor's `login`, real `name`, commit counts, AI-commit counts, and `lastActiveAt` for an org they don't belong to. This is the single highest-PII surface in the context (the pages render the same data, but this route serializes the full unbounded `insights.contributors` list, not the page's top-50 slice). Nothing in the suite would fail.
- **Root cause**: The route has no test at all. `authz.test.ts` exercises `requireOrgRead`/`canReadOrg` in isolation with `@/lib/auth` mocked, but never asserts that *this handler actually calls the gate before reading data*, nor that the gate's Response (401/403) short-circuits the body. The data-fetch (`getContributorInsights`) sits two lines below an `await`ed gate whose ordering is load-bearing and untested.
- **Impact**: Cross-tenant exfiltration of employee/contributor PII — a privacy/compliance incident, not just a bug. Blast radius is the whole fleet's people data for every org in the DB.
- **Fix sketch**: Add `src/app/api/org/export/route.test.ts`. Mock `@/lib/authz.requireOrgRead` and `@/lib/db` (`getContributorInsights`, `getOrgGovernance`, `listSegments`, `isDbConfigured`). Assert invariants: (a) when `requireOrgRead` returns a 403 `NextResponse`, the handler returns it **and** `getContributorInsights` is **never called** (`expect(getContributorInsights).not.toHaveBeenCalled()`) — proves the gate precedes the read; (b) `kind` other than `contributors|delivery` → 400 before any gate/data; (c) `format=csv` sets `content-disposition: attachment` with a `safeFilenameSlug`-sanitized filename (feed `org="a/../b,x"` and assert no `,`/`/`/`"` in the header); (d) a contributor `name` containing a comma/quote/newline is RFC-4180 quoted in the CSV body (lock `csvField`). The non-negotiable assertion is (a): gate-before-read.

## 2. Extract and unit-test `getContributorInsights`' bus-factor / concentration / orgAiShare math
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/lib/db/org-contributors.ts:142-185
- **Scenario**: The concentration block computes `busFactor` (contributors needed to cover >50% of commits), `topShare`, `soloMaintainer` (1 author **or** top ≥80%), and the org tiles `aiActiveShare` / `orgAiShare`. A refactor that flips the bus-factor loop's `>` to `>=`, or rounds `topShare` differently, or changes the `soloMaintainer` threshold, silently mislabels which repos carry "key-person risk" — the orange `key-person`/`bus factor` warnings the Contributors page renders to drive staffing decisions. No test pins these numbers. The team path got a `rollupTeams` pure function + `teamRollup.test.ts`; the contributor path did not, despite identical risk.
- **Root cause**: All this logic is inlined inside an `async` DB function (`prisma.repoContributor.findMany` at the top), so it can't be unit-tested without a DB. There's no exported pure aggregator (the `rollupTeams` pattern) to test, and none exists.
- **Impact**: Wrong bus-factor/solo-maintainer flags misdirect "who's a single point of failure" calls; wrong `orgAiShare`/`aiActiveShare` headline tiles misstate fleet AI adoption to leadership. Both are decision inputs surfaced prominently with risk styling.
- **Fix sketch**: Extract the post-query body into `export function rollupContributors(rows: ContributorRow[]): ContributorInsights` (mirroring `rollupTeams`), with `getContributorInsights` reduced to query + delegate. Add `org-contributors.test.ts` asserting concrete invariants on a hand-built fleet: a repo with commits `[6,3,2,1]` → `busFactor === 1` (6 > 12/2? no → 6+3=9 > 6 → busFactor 2; pick numbers that pin the boundary) and assert the exact boundary case where `acc` first **exceeds** half; `soloMaintainer === true` for a single-author repo and for an 80/20 split, `false` at 79%; `aiActiveShare = round(aiActive/total*100)`; `orgAiShare` is commit-**weighted** not contributor-averaged (give one heavy AI committer + many light non-AI committers and assert the weighted result, catching a mean-of-shares regression). Exclude `[bot]`/`unknown` and assert they don't appear.

## 3. Test `getOrgActivity`'s unequal-length weekly-series alignment
- **Severity**: High
- **Category**: edge-case
- **File**: src/lib/db/org-signals.ts:174-182
- **Scenario**: The fleet commit chart sums each repo's weekly series aligned by **most-recent week** (`offset = maxLen - s.length`, write at `series[offset + i]`). If a future change aligns by index-0 instead (the obvious "bug fix" a maintainer might make), a repo with 8 weeks of history and one with 52 weeks get added oldest-to-oldest, smearing recent activity into the wrong buckets — the Delivery page's "this week" bar would silently show stale/ghost numbers. There is no test, so the alignment convention isn't pinned.
- **Root cause**: Pure, deterministic, eminently testable array math buried in an `async`/Prisma function with no extracted helper and no test. The element-wise offset sum is exactly the kind of off-by-one that survives code review and ships.
- **Impact**: Corrupted delivery throughput visualization — the headline "is the fleet shipping?" signal — with no error, just wrong bars. Misleads delivery/flow decisions.
- **Fix sketch**: Extract `export function alignActivitySeries(seriesList: number[][]): { weeks; series; total; repos }` and test: two series `[1,2,3]` and `[10,20]` align to `[1, 12, 23]` (recent weeks `2+20`, `3` alone in oldest slot per the offset) — assert the **last** element sums both repos' latest weeks and the total equals the sum of all inputs (`total === 36`); a single series passes through unchanged; empty/malformed entries are dropped (so `repos` counts only contributing series). The invariant: the most-recent week (last index) is always the sum of every repo's most-recent week.

## 4. Test the null-vs-zero "no sample" semantics in `getOrgPrSignals` / `getOrgGovernance`
- **Severity**: Medium
- **Category**: error-branch
- **File**: src/lib/db/org-signals.ts:43-63
- **Scenario**: `avgReviewedRate` and `avgAiGovernedRate` are deliberately `null` when **no repo has a measurable sample** (the page renders `—` and "no human-merged PRs"), but a real `0` when every PR was unreviewed. A regression that collapses `null` to `0` (e.g. dropping the `reviewed.length ? … : null` guard, or `mean([])` returning 0) turns "we have no data" into "0% reviewed" — a false governance-failure alarm on the Delivery tab. Also untested: the `JSON.parse(raw)` in a `try/catch` that silently `continue`s on malformed `prStats`/`governance`, and the `p.analyzed > 0` / `g.readable` filters. A malformed row that should be skipped but instead inflates the denominator goes unnoticed.
- **Root cause**: Same shape as #3 — aggregation logic with meaningful branch behavior (null-vs-zero, parse-failure swallow, readable/analyzed filters) inlined in `async` DB functions, no extracted helper, no test.
- **Impact**: False "0% review coverage / 0% AI-governed" alarms (or the inverse — silently dropping bad rows that should count), distorting the governance posture leaders read on the Delivery tab.
- **Fix sketch**: Extract `export function rollupPrSignals(statsList: PrStats[])` and `rollupGovernance(rows)` and test: all-`null` `reviewedRate` inputs → `avgReviewedRate === null` (not 0); a mix of `null` and numeric → mean over **only** the numeric samples; a `PrStats` with `analyzed === 0` is excluded from `totalPrs`/`repos`; a governance row with `readable === false` is dropped from `perRepo` and from every `*Rate` denominator; `perRepo` is sorted unprotected-first. Assert the parse-failure path by feeding the function pre-parsed objects (so the test targets the math) and separately unit-test that a malformed string is skipped, not throwing.

## 5. Pin the "small-population success-theater" guards for champions and AI-active share
- **Severity**: Medium
- **Category**: success-theater
- **File**: src/app/org/[slug]/contributors/page.tsx:75 ; src/lib/db/org-contributors.ts:170-179
- **Scenario**: The Contributors page deliberately hides the "AI champions" leaderboard until `insights.totalContributors >= 3` (comment: a single Copilot user would otherwise become a celebrated "#1 ★ champion" and tiles read "100% AI-active" for a team of one). The Teams page has the analogous `totalContributors >= 3` champion floor in `org-contributors`'s `champions` filter (`commits >= 3 && aiCommits > 0`). These overstatement guards are **load-bearing product decisions with no test** — both the page-level `>= 3` gate and the data-level `commits >= 3` champion floor. A refactor that drops either re-introduces the "barely-adopted fleet looks fully AI-native" theater the comments explicitly warn against.
- **Root cause**: The population gate lives in JSX (`page.tsx:75`), untestable by the Vitest setup (no jsdom/RTL), and the champion floor lives in an untested DB function. Neither invariant is captured anywhere, so the guard rails are one careless edit from vanishing.
- **Impact**: A 1–2 person repo with one Copilot user gets presented as a "100% AI-active" fleet with a star "champion" — false confidence that overstates adoption to leadership and undermines the "inputs not rankings" framing the product leans on.
- **Fix sketch**: In the extracted `rollupContributors` test (from #2), assert: a fleet of 1 AI committer yields `champions: []` is NOT enough — assert `aiActiveShare` and the champion list behave at the boundary (2 contributors → still no champions surfaced by the `>= 3` rule the page enforces; encode the floor as a tested constant shared between data + page rather than a magic `3` in JSX). For the DOM-bound gate, add a Playwright e2e (the project's prescribed path for view behavior): seed a 2-contributor org, load `/org/<slug>/contributors`, assert the "AI champions" section is absent and the involvement table still renders. The invariant: the champions leaderboard never renders below the documented population floor.
