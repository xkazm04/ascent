# Test Mastery — Repo Report Shell & Tabs

This context is the single-repo report shell: it hydrates an untrusted scan payload (SSE or peeked cache), validates it at a trust boundary, guards against cross-repo data leakage, and reconciles the live report against persisted history. The entire `src/components/report/` directory and `src/lib/report/validate.ts` have **zero unit tests** (verified: no `*.test.ts(x)` under `src/components/report/` or `src/lib/report/`, and `parseScanReport`/`parseRepositoryHistory`/`parseSSE`/`repoKey` are referenced nowhere outside their definition + call sites). The only coverage is `e2e/scan.spec.ts`, which exercises the happy path and one invalid-repo path against a **live** repo — it cannot deterministically reach the failure, salvage, identity-guard, or malformed-payload branches. Every comment in these files documents a real regression that was fixed but has no regression test pinning it. The validators and SSE/identity helpers are pure functions — exactly the LLM-generatable, jsdom-free batch the project's Vitest setup (alias-only, no DOM) is built for.

## 1. Pin the cross-repo identity guard that prevents one repo's report rendering under another's URL
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/components/report/ReportClient.tsx:30 (`repoKey`) and :94-98 / :142-148 (the two `gotKey === reqKey` gates)
- **Scenario**: A refactor normalizes `repoKey` slightly differently (e.g. stops stripping `.git`, or the peek/salvage payload's `repo.owner/name` casing drifts), so `gotKey === reqKey` passes for a *colliding cache entry* belonging to a different repo. The user opens `/report?repo=acme/private-app` and is shown `acme/other-repo`'s maturity scores — a cross-tenant data leak — with no test failing.
- **Root cause**: `repoKey` (URL-stripping + lowercasing + slash-trim) and both identity comparisons are pure string logic with no test. The comment at :91-93 ("a stale/colliding cache entry … could otherwise show another repo's report") names the exact hazard, but nothing pins the invariant.
- **Impact**: Wrong-repo disclosure on a shared/public surface. For a private-repo customer this is a confidentiality breach; for any user it silently shows false data attributed to their repo.
- **Fix sketch**: Extract `repoKey` (export it) and add a `repoKey.test.ts` asserting canonicalization invariants: `repoKey("https://github.com/Acme/App.git/") === repoKey("acme/app") === "acme/app"`, trailing-slash/leading-slash idempotence, and that two *different* repos never collide. Then a small `reportClient.identity.test.ts` over an extracted `matchesRequestedRepo(reqUrl, report)` helper asserting it returns `false` when `report.repo.owner/name` differs from the requested key (so the peek/salvage paths fall through to a fresh scan) and `true` only on an exact canonical match.

## 2. Test the scan-payload trust boundary (`parseScanReport`) for FAILURE, field by field
- **Severity**: Critical
- **Category**: coverage-gap
- **File**: src/lib/report/validate.ts:23-104 (`parseScanReport`)
- **Scenario**: The scoring engine or LLM-output schema drifts — `overallScore` arrives as a string, `dimensions` arrives empty, `repo.stars` is `null`, or the result channel carries `{ error }`. Because nothing tests the guard, a relaxed check (or a removed branch) lets a malformed payload through `parseScanReport` and `ReportView` throws mid-render on `repo.stars.toLocaleString()` / `dimensions.map()` — white-screening the report instead of showing the friendly error. Conversely, an over-strict edit could reject every *valid* report and no test would catch the false-negative.
- **Root cause**: This function is explicitly "the trust boundary between the /api/scan(/stream) payload and ReportView" (file header) yet has no test. Every `fail(...)` branch and the success path are unverified.
- **Impact**: A single schema drift turns a successful scan into a blank/crashed page for every user — the product's core output silently breaks, with no CI signal.
- **Fix sketch**: Add `validate.test.ts` with a `validReport()` fixture builder (full ScanReport) and one `expect(parseScanReport(valid).ok).toBe(true)` baseline, then a table-driven batch that mutates exactly one field per case and asserts `{ ok: false, error: <the specific message> }`: missing/non-object `repo`, non-string `repo.url`, non-number `repo.stars`, malformed `level`/`posture`/`engine`/`aiUsage`, non-number `overallScore`, empty `dimensions`, a dimension missing `evidence`, and a top-level `{ error: "boom" }` → `ok:false, error:"boom"`. Invariant: **valid stays valid; any one corrupted field fails with a human message and never throws.**

## 3. Test `parseRepositoryHistory` so junk history can never crash or fabricate trend points
- **Severity**: High
- **Category**: coverage-gap
- **File**: src/lib/report/validate.ts:114-146 (`parseRepositoryHistory`)
- **Scenario**: `/api/history` returns a 200 with a drifted body (`scans` not an array, a point with a non-numeric `overallScore`, an unparseable `scannedAt`, a `dimensions[]` entry missing `score`). The coercion is supposed to *drop* the bad point and never throw; a refactor that loosens the `Number.isNaN(Date.parse(...))` guard would feed an unplottable point into `TrendChart`/`forecastTrajectory` (date math → NaN axis) or, worse, a `flatMap` change throws and the whole trend section white-screens.
- **Root cause**: The contract ("NEVER throw — always return a well-formed object, dropping any point that can't be coerced", :110-113) is asserted only in a comment. No test exercises the drop-bad-point or empty-on-junk behavior.
- **Impact**: A transient backend shape change blanks the trend panel for repos that have real history — the feature that demonstrates progress over time silently dies, undermining the product's core value narrative.
- **Fix sketch**: Add cases to `validate.test.ts`: `parseRepositoryHistory(null)` / `(42)` / `({scans:"x"})` → `{ repo:{owner:"",name:"",fullName:""}, scans:[] }`; a mixed array where one point has `overallScore:"50"` and another has `scannedAt:"not-a-date"` → both dropped, valid points retained in order; a point whose `dimensions` contains one good `{dimId,score}` and one missing `score` → only the good dim survives. Invariant: **output is always a well-formed `RepositoryHistory`, every retained point has a finite score and a `Date.parse`-able `scannedAt`, and no input ever throws.**

## 4. Test the SSE frame parser (`parseSSE`) and the stream drain for multi-line / CRLF / no-trailing-blank-line frames
- **Severity**: High
- **Category**: error-branch
- **File**: src/components/report/ReportClientStatus.tsx:21-40 (`parseSSE`); src/components/report/ReportClient.tsx:208-237 (frame splitter + final drain)
- **Scenario**: The server sends the terminal `result` event right before close **without** a trailing blank line, or splits a pretty-printed JSON payload across multiple `data:` lines, or a proxy rewrites `\n\n` to `\r\n\r\n`. The comments at ReportClient:225-231 and ReportClientStatus:29-31 say each of these *was* a bug that dropped a perfectly good scan to "ended unexpectedly" / corrupted multi-line JSON. A future tweak to the regex (`/\r?\n\r?\n/`) or the `data:` join silently reintroduces "scan ended unexpectedly" on successful scans — invisible to the live-repo e2e, which only ever sees the current server's framing.
- **Root cause**: `parseSSE` is exported and pure but untested; the drain loop's "trailing frame with no blank line" handling is the riskiest line in the file and has no regression test.
- **Impact**: Users see a spurious "The scan ended unexpectedly" error on a scan that actually succeeded — the most damaging kind of false failure, eroding trust in every result.
- **Fix sketch**: `parseSSE.test.ts`: assert `parseSSE("event: result\ndata: {\"a\":1}")` → `{event:"result", data:{a:1}}`; a payload split across two `data:` lines joins with `\n` and parses; CRLF line endings parse identically; a malformed `data:` yields `data:null` without throwing; a block with no `event:` yields `event:null`. Then extract the buffer-drain into a pure `drainSSE(chunks: string[]): {event,data}[]` and assert: two `\n\n`-separated frames both emit; a final frame with **no** trailing blank line still emits (the regression guard); CRLF frame boundaries split correctly. Invariant: **every complete frame the server writes is dispatched exactly once, including the unterminated terminal `result`.**

## 5. Cover the history HTTP-status branch and the abort/timeout taxonomy that decide which message the user sees
- **Severity**: Medium
- **Category**: error-branch
- **File**: src/components/report/ReportView.tsx:42-52 (503/401 vs other status); src/components/report/ReportClient.tsx:238-250 (timeout vs interrupt vs network abort)
- **Scenario (a)**: ReportView treats history `503`/`401` as "legitimate no-trends" (quiet baseline) but any other non-OK (e.g. `500` from a transient DSQL token expiry) as a real failure → `histError`. A refactor that lumps all non-OK together would, per the :43-46 comment, render "Baseline established" over months of real history — silently hiding regressions. **Scenario (b)**: ReportClient maps `AbortError` to "timed out" only when `timedOut` is true, else "interrupted"; a network error → "Network error". Reordering these branches mislabels a 3-minute timeout as a generic network error, hiding the "try a smaller repository" guidance.
- **Root cause**: Both are conditional branches with documented past bugs, gated on `h.status` / `timedOut` / `e.name` — pure decision logic embedded in effects, untested because they live inside React effects rather than extracted helpers.
- **Impact**: (a) misclassifying server failure as "no history" hides real maturity regressions from leadership; (b) wrong error copy sends users down dead-end retries. Both degrade trust in the report's honesty.
- **Fix sketch**: Extract two pure helpers and test them: `classifyHistoryResponse(status: number, ok: boolean): "ok" | "no-trends" | "error"` — assert `200→ok`, `503→no-trends`, `401→no-trends`, `500→error`, `404→error`; and `classifyScanAbort({name, timedOut, cancelled}): "timeout"|"interrupted"|"network"|"none"` — assert `AbortError`+`timedOut→timeout`, `AbortError`+`!timedOut→interrupted`, other error→`network`, `cancelled→none` (no state change). Invariant: **only `503`/`401` suppress the trend panel; a genuine timeout always yields the smaller-repo guidance, never "Network error."**
