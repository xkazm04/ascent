## 1. `ReportTabBar` component is fully dead — only its `ReportTab` type is still used
- **Severity**: High
- **Category**: dead-code
- **File**: src/components/report/ReportTabBar.tsx:13-74
- **Scenario**: The entire `ReportTabBar` function component — the segmented `role="tablist"` widget, its roving-tabindex keyboard handler `onKeyDown` (Arrow/Home/End), the `btnRefs` ref array, the per-tab `aria-controls`/`aria-selected` wiring, and the `report-tab-btn-${id}` test ids — is never rendered. A repo-wide search for `<ReportTabBar` / `ReportTabBar(` returns zero hits. `ReportView.tsx` switched the report's tab switching to `SideNav` (ReportView.tsx:18, 161-163, 189), and `ScoringTab.tsx:39-42` documents that migration ("ReportView migrated the tab switcher to SideNav … the old tabpanel ARIA here was orphaned"). The only live consumer of this file is `ReportView.tsx:12`, which imports **only** `import { type ReportTab }` — the component itself has no callers.
- **Root cause**: A prior refactor replaced the tablist UI with the left-rail `SideNav` (aria-current nav buttons) but left the original `ReportTabBar` widget file in place. The shared `ReportTab` union type still lived inside it, so the import line kept the file "referenced" and it survived dead-code sweeps — the type kept the lights on for a component nobody renders.
- **Impact**: ~75 lines of unrendered, accessibility-critical UI (a hand-rolled WAI-ARIA tabs implementation) sit in the report tree masquerading as live code. A maintainer touching report tab behavior will reasonably edit `ReportTabBar` and see no effect, or try to keep its ARIA in sync with the real `SideNav` path — pure wasted effort and a misleading source of truth. It also carries `"use client"` weight and ships in the report chunk for no reason.
- **Fix sketch**: Move the `export type ReportTab = "scoring" | "roadmap" | "sandbox" | "contributors";` declaration to a neutral home — either `reportTaxonomy.ts` (already the report's pure-logic module) or inline at the top of `ReportView.tsx` where `tab`/`setTab`/`tabs` live. Update the single importer `ReportView.tsx:12` to pull `ReportTab` from the new location. Then delete `ReportTabBar.tsx` entirely. Behavior-preserving: the component renders nowhere, and no test references `report-tab-btn-*` or `report-panel-*` ids (confirmed by grep — only the surviving section `data-testid`s like `report-tab-scoring`/`report-tab-roadmap` are used).

## 2. `parseRepoParam` duplicated verbatim across the two permalink-route files
- **Severity**: Medium
- **Category**: duplication
- **File**: src/app/report/[owner]/[repo]/page.tsx:22-26 ; src/app/report/[owner]/[repo]/opengraph-image.tsx:19-23
- **Scenario**: The helper that splits a `repo` path segment carrying an optional pinned commit (`name` or `name@sha`) is defined twice, byte-for-byte identical (same JSDoc, same `indexOf("@")` logic, same `slice(at + 1) || undefined`). Both files live in the same route folder and both parse the same `{ owner, repo }` route param for the same purpose (resolving `getScanReportByCommit(owner, name, { headSha: sha, … })`).
- **Root cause**: `opengraph-image.tsx` was added as a co-located social-card generator after `page.tsx` already had the parser; the one-liner was copy-pasted rather than shared, since Next.js segment files don't naturally suggest a shared sibling.
- **Impact**: Two copies of route-param parsing that must stay in lockstep — if the permalink grammar ever grows (e.g. a branch ref, or URL-decoding the `@`), one copy will be updated and the OG card will silently diverge from the page (wrong/blank social unfurl vs. correct page, or vice-versa). It's the classic "drift waiting to happen" duplication on a user-visible path.
- **Fix sketch**: Extract the function once into a small route-local module, e.g. `src/app/report/[owner]/[repo]/repoParam.ts` (or fold it next to `repoKey.ts` in `src/components/report/`), exporting `parseRepoParam(repoParam: string): { name: string; sha?: string }`. Import it in both `page.tsx` and `opengraph-image.tsx` and delete the two inline copies. Behavior-preserving: identical implementation, three existing call sites (`page.tsx:34`, `page.tsx:60`, `opengraph-image.tsx:63`) keep the same signature.

## 3. Shared "pill" button styling duplicated as inline class strings (FreshnessControl ↔ ReportHeader) plus a duplicated Suspense-fallback wrapper
- **Severity**: Medium
- **Category**: duplication
- **File**: src/components/report/FreshnessControl.tsx:23-24 ; src/components/report/ReportHeader.tsx:60 ; (related) src/app/report/page.tsx:19 and src/app/report/[owner]/[repo]/page.tsx:86
- **Scenario**: The rounded outline-pill class string `inline-flex items-center gap-1 rounded-full border border-slate-700 px-2.5 py-1 … font-medium text-slate-300 transition hover:border-accent hover:text-white` is hand-written in `FreshnessControl` (the Re-test button/link, stored in a `retestClass` const) and again on the "Export PDF" anchor in `ReportHeader` — these two pills sit side-by-side in the same header row and are meant to look identical. Separately, the Suspense fallback wrapper `<div className="mx-auto w-full max-w-md py-12"><ReportSkeleton /></div>` is duplicated verbatim between `report/page.tsx:19` and `report/[owner]/[repo]/page.tsx:86`.
- **Root cause**: Tailwind utility strings are easy to copy when two adjacent elements should match; `FreshnessControl` was extracted from the header but the shared pill look was re-typed rather than centralized. The Suspense wrapper was copied when the permalink route was added alongside the search route.
- **Impact**: Visual drift risk on a tight, side-by-side button group — a hover/border/spacing tweak applied to one pill won't reach the other, so the Re-test and Export buttons can silently fall out of alignment. Low bug-risk but real maintenance friction across the header chrome. (The same pill string also recurs in several `src/components/org/*` files; those are out of scope here but reinforce that this is a genuine shared-token candidate.)
- **Fix sketch**: Define one constant for the report-header pill class (e.g. `REPORT_PILL` in a shared `src/components/report/` styles module, or a tiny `<Pill>`/`pillClass()` helper next to the existing `Surface`/`Kicker` UI primitives) and use it in both `FreshnessControl` (`retestClass`) and `ReportHeader`'s Export-PDF/skill anchors; keep the per-element accent variants (the skill link's `bg-accent/10`) as additive classes. Separately, hoist the `<div className="mx-auto w-full max-w-md py-12"><ReportSkeleton /></div>` fallback into a tiny `ReportSkeletonFallback` (or export it from `ReportSkeleton.tsx`) and reference it from both pages. Both changes are purely class/markup consolidation — no behavior change.

## 4. `isMock` recomputed in `ReportView` and again in its child `ReportHeader`
- **Severity**: Low
- **Category**: duplication
- **File**: src/components/report/ReportView.tsx:24 ; src/components/report/ReportHeader.tsx:12
- **Scenario**: `const isMock = report.engine.provider === "mock";` is computed in `ReportView` (which passes it down to `ScoringTab` via the `isMock` prop) and computed a **second** time, independently, inside `ReportHeader` — even though `ReportView` renders `<ReportHeader report={report} … />` directly (ReportView.tsx:168) and already holds the value. Both files carry the same explanatory comment ("Drive every engine-related treatment off this single flag so the demo signal stays consistent everywhere"), so the intent is explicitly a single source of truth — yet there are two.
- **Root cause**: `ScoringTab` and `ContributorsPanel` were extracted from `ReportView` and `isMock` was correctly threaded to `ScoringTab` as a prop; `ReportHeader` predates or sidesteps that threading and kept its own local derivation.
- **Impact**: Minor — the derivation is trivial and can't currently disagree (same input). But it quietly contradicts the stated "single flag" intent: if the "mock/demo" definition ever broadens (e.g. include another keyless provider or a `report.engine.deterministic` field), a maintainer must remember to edit two places, and the header could show "engine: …" while the body shows the demo notice.
- **Fix sketch**: Pass the already-computed `isMock` into `ReportHeader` as a prop (`<ReportHeader report={report} isMock={isMock} onRetest={onRetest} />`) and drop the local `const isMock` in `ReportHeader.tsx:12`, mirroring how `ScoringTab` already receives it. This makes `ReportView` the one place the "is this a mock/demo engine" decision lives. Behavior-preserving (identical value today). Lowest priority — fold in opportunistically when next touching the header.
