# Code Refactor — People & Delivery Analytics
> Context group: Org Dashboard & Analytics
> Total: 4 findings (Critical: 0, High: 1, Medium: 2, Low: 1)

These three pages (`contributors`, `teams`, `delivery`) are individually clean — no dead code, no stray `console.log`, no commented-out blocks, no stale TODOs, and every import is referenced. The findings are all about *duplication* of small presentational blocks that have already been copy-pasted across the in-scope files (and visibly drifted between them), plus one localized structure note. None of the imports or local components are unused, so there is nothing to delete outright; the value here is consolidation.

## 1. The segment + tech-stack selector pair is hand-rolled in all three pages (and 7 more siblings) with drifting wrappers
- **Severity**: High
- **Category**: duplication
- **File**: src/app/org/[slug]/contributors/page.tsx:33-38; src/app/org/[slug]/teams/page.tsx:134-139; src/app/org/[slug]/delivery/page.tsx:52-63
- **Scenario**: Each page resolves scope via `resolveOrgScope(...)` then builds the *same* selector pair inline — `{segments.length > 0 && <SegmentSelector segments={segments} active={segmentId} />}` followed by `<TechStackSelector groups={techGroups} active={activeStack?.key ?? null} />` — inside a flex wrapper. The three in-scope copies have already drifted: contributors gates the *whole* bar on `(segments.length > 0 || techGroups.length > 0)` and renders it twice (empty-state line 44 + header line 57); teams uses `mb-4 flex flex-wrap justify-end gap-2` and does NOT gate the bar (so it renders an empty wrapper when both lists are empty); delivery uses `flex flex-wrap items-center justify-end gap-2` and appends an export anchor. The identical inner pair is repeated in passports, repositories, security, executive, live, practices, and the org root — 10 files total (confirmed via grep on `<TechStackSelector groups={techGroups}`).
- **Root cause**: `resolveOrgScope` (src/lib/org/scope.ts) was extracted to dedupe the *data* side of the scope filters, but the matching *render* side was never extracted, so every new scoped tab re-types the selector JSX. The wrapper class and empty-bar guard were copied imperfectly, so each page guards differently.
- **Impact**: A change to the selector pair (e.g. selector order, a new "clear filters" affordance, or fixing the empty-wrapper render) must be made in ~10 places and is easy to miss; the existing drift means the bar already behaves inconsistently across tabs (gated vs. always-rendered). Pure maintenance + UX-consistency cost.
- **Fix sketch**: Add a server-safe `ScopeFilterBar` component next to the selectors (e.g. `src/components/org/ScopeFilterBar.tsx`) taking `{ segments, segmentId, techGroups, activeStack, className?, children? }`, internally gating `segments.length > 0` for `SegmentSelector`, always rendering `TechStackSelector`, and only rendering the wrapper when at least one selector would show. Accept a `children` slot for the trailing export anchor (delivery). Replace the three in-scope inline blocks with `<ScopeFilterBar {...scope} />` (delivery passes the export link as a child). Behavior-preserving if the wrapper classes are preserved per call site via a `className`/`align` prop.

## 2. The "Export CSV" anchor is copy-pasted between contributors and delivery (and again in repositories/security)
- **Severity**: Medium
- **Category**: duplication
- **File**: src/app/org/[slug]/contributors/page.tsx:115-120; src/app/org/[slug]/delivery/page.tsx:56-61
- **Scenario**: Both files render the same anchor — identical `className="focus-ring … rounded-md border border-slate-700 px-3 py-1.5 font-mono text-sm text-slate-300 transition hover:border-accent hover:text-white"`, the same `href={`/api/org/export?org=${encodeURIComponent(slug)}&kind=<X>&format=csv${segmentId ? `&segment=${segmentId}` : ""}`}` URL shape (only `kind=contributors` vs `kind=delivery` differs), and the same `Export CSV` label. (contributors adds a `shrink-0` class.) repositories and security carry a near-identical anchor against a different endpoint, so this className+anchor is effectively a shared widget that was never named.
- **Root cause**: The export affordance was added per-tab as each export `kind` shipped, with no shared link component, so the URL-building + styling was re-typed each time.
- **Impact**: The export URL contract (`org`, `kind`, `format`, optional `segment`) lives in string templates in multiple files; if a new scope param is added (e.g. the `stack`/`techGroupId` filter, which is currently *not* threaded into the export URL on either page despite scoping the rest of the view) every copy must be edited, and the omission is easy to overlook. Styling drift between copies is already visible (`shrink-0`).
- **Fix sketch**: Extract `ExportCsvLink({ org, kind, segmentId, className? })` (e.g. in `src/components/org/ui.tsx` or a small `ExportCsvLink.tsx`) that owns the URL-building and the brand button class. Replace both in-scope anchors with `<ExportCsvLink org={slug} kind="contributors" segmentId={segmentId} className="shrink-0" />` / `kind="delivery"`. Optionally extend it to the repositories/security endpoints via a `href`/`kind` discriminator. Behavior-preserving.

## 3. Three near-twin "labelled Meter row" micro-components exist across the people/adoption pages
- **Severity**: Medium
- **Category**: duplication
- **File**: src/app/org/[slug]/contributors/page.tsx:11-18 (`AiBar`); src/app/org/[slug]/teams/page.tsx:12-22 (`MetricBar`); sibling src/app/org/[slug]/adoption/page.tsx:117-125 (`DeliveryRow`)
- **Scenario**: Each page declares its own tiny "a `Meter` plus a numeric/percent readout" component. `AiBar` = `Meter w-24` + right-aligned `{pct}%`; `MetricBar` = label-above + `Meter` with `threshold={50}` colored by `scoreHex`; `DeliveryRow` = left label + flex `Meter` + right value. They are structurally close but each lays out the label/value slightly differently, so they aren't literally identical — but they are the same conceptual primitive re-invented three times.
- **Root cause**: `Meter` is shared (src/components/org/ui.tsx), but the common "meter + label/value" composition was never lifted, so each tab wrote its own thin wrapper for its specific label placement.
- **Impact**: Low bug-risk (these are simple), but it's three places to touch for any change to how a metered value is presented (e.g. tabular-nums, color rules, accessibility labelling on the bar). Mild confusion about which wrapper to reuse when adding a new metered row.
- **Fix sketch**: Add one configurable `MeterRow` to `src/components/org/ui.tsx` with a `layout: "inline" | "stacked" | "labelled"` (or `label?`, `value`, `valueSide?`, `threshold?`, `color?`) prop covering the three placements, then replace `AiBar`/`MetricBar`/`DeliveryRow` with it. Keep the call sites' exact classes via props so output is pixel-identical. Lower priority than #1/#2 because the layouts genuinely differ; only consolidate if a unified `MeterRow` stays readable. (Note: `DeliveryRow` is outside the 3-file scope — listed only as the third drift point; the in-scope action is consolidating `AiBar` + `MetricBar`.)

## 4. `ContributorInsightsPage` inlines two full tables + a champions grid in one 200-line default export
- **Severity**: Low
- **Category**: structure
- **File**: src/app/org/[slug]/contributors/page.tsx:50-225
- **Scenario**: The default export renders the summary tiles, the champions grid (lines 71-95), a `<details>`-wrapped individual-involvement table with its own CSV link and `.slice(0, 50)` truncation (lines 101-165), and the concentration/bus-factor table (lines 168-212) all inline. It's coherent and well-commented, but it's the largest of the three pages and mixes four distinct presentational concerns in one function.
- **Root cause**: The page grew section-by-section; each new table was appended inline rather than extracted, which is normal for a server page that renders once.
- **Impact**: Minor — readability and diff-noise only. No correctness or bundle cost (it's a server component). Extracting would also make the champions-grid and the two tables individually testable/reusable (e.g. the champions grid block is conceptually shared with teams' champion chips and adoption's champions list).
- **Fix sketch**: Optionally extract `<ChampionsGrid champions={…} />`, `<IndividualInvolvement insights={…} slug={…} segmentId={…} />`, and `<ConcentrationTable rows={insights.concentration} />` into co-located components (same file or a `contributors/` folder). Behavior-preserving pure-move refactor; only worth doing if the page is touched again or if the champions block is unified with the other tabs (ties into the broader champions de-dup theme noted across `CHAMPION_MIN_POP` call sites). Listed Low because the current file, while long, is clear and self-documenting.
