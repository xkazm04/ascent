# Code Refactor — GitHub Repo Data Access
> Context group: Identity & GitHub Connectivity
> Total: 4 findings (Critical: 0, High: 1, Medium: 3, Low: 0)

The eight in-scope files are well-documented and the pure transforms (`parseCodeowners`, `rankDiscoveredOrgs`, selection helpers) are clean and well-tested. The cruft is concentrated in the I/O layer: the same GitHub-request scaffolding (auth headers + abortable timeout fetch) is hand-rolled four/three times across `source.ts`, `governance.ts`, `graphql.ts`, and `discover.ts`, and the org/user repo listing+normalization is duplicated between `list.ts` and `discover.ts`. One small genuinely-dead export rounds it out. No stray `console.log`, no commented-out code, no unused imports were found.

## 1. GitHub request `headers()` rebuilt in four I/O modules
- **Severity**: High
- **Category**: duplication
- **File**: src/lib/github/source.ts:111-119, src/lib/github/governance.ts:14-21, src/lib/github/graphql.ts:54-60, src/lib/github/discover.ts:67-73
- **Scenario**: Every GitHub I/O file builds the same request-header object by hand — `Accept: application/vnd.github+json`, `Authorization: Bearer <token>`, `User-Agent`, `X-GitHub-Api-Version: 2022-11-28`. `source.ts` and `governance.ts` expose it as a private `headers()` helper; `graphql.ts` and `discover.ts` inline the object literal. The same shape repeats again in the obvious siblings `app.ts:101-106` and `auth.ts:510`, and `list.ts:81-85` is a lowercased variant of the same set.
- **Root cause**: Each module was written self-contained against the REST/GraphQL API; `host.ts` was extracted later to centralize the *base URL* but the *headers* (the other half of "consistent auth" the context purpose calls out) were never pulled with it. There is no `src/lib/github/index.ts` barrel and no shared request module, so copy-on-write was the path of least resistance.
- **Impact**: The pinned API version (`2022-11-28`), the User-Agent convention, and the optional-`token` → `Authorization` rule are five+ copies that must be edited in lockstep. They already drift in casing (`source`/`governance`/`graphql`/`discover` use TitleCase header names; `list.ts` uses lowercase) and in token-optionality (`source`/`list` treat the token as optional, the rest require it). A future API-version bump or auth tweak silently misses whichever copy the author forgets.
- **Fix sketch**: Add a `githubHeaders(opts?: { token?: string; accept?: string })` (and, if useful, a `rawHeaders()` for the User-Agent-only raw fetch) to `host.ts`, returning the canonical header set with `Authorization` added only when a token is present. Replace the private `headers()` in `source.ts`/`governance.ts` and the inline literals in `graphql.ts`/`discover.ts`/`list.ts` with calls to it (lowercasing in `list.ts` is irrelevant — HTTP header names are case-insensitive). Behavior-preserving: same header values emitted; only the construction site moves. (Siblings `app.ts`/`auth.ts` are out of scope but are the same target.)

## 2. Abortable-timeout fetch wrapper hand-rolled three times
- **Severity**: Medium
- **Category**: duplication
- **File**: src/lib/github/source.ts:138-152 (`fetchWithTimeout`), src/lib/github/governance.ts:23-40 (`getJson`), src/lib/github/graphql.ts:47-62 (inline in `githubGraphql`)
- **Scenario**: Three modules independently implement the identical "fetch with a per-call timeout that also honors a caller AbortSignal" pattern: `new AbortController()` → `setTimeout(() => controller.abort(), ms)` → `const combined = signal ? AbortSignal.any([controller.signal, signal]) : controller.signal` → fetch with `signal: combined` → `clearTimeout` in `finally`. The merge line is byte-for-byte identical in all three (confirmed by grep: only these three files contain `AbortSignal.any([controller.signal, signal])`).
- **Root cause**: Same as #1 — each I/O module grew its own request primitive before any shared fetch helper existed. The signal-merge was added later (client-disconnect support) and was copied verbatim into each existing wrapper rather than extracted.
- **Impact**: The timeout/abort contract — the thing that guarantees no upstream call can hang a serverless function — lives in three places. A fix to the abort handling (e.g. swapping `AbortSignal.any` for a polyfill, or adding jitter/`AbortSignal.timeout`) must be applied three times, and the wrappers can quietly diverge (they already differ in what they return: a raw `Response`, a `{status, body}`, or parsed JSON).
- **Fix sketch**: Add a single `fetchWithTimeout(url, init, ms, signal?)` returning a `Response` to `host.ts` (or a small `request.ts`), identical to `source.ts`'s existing implementation. Have `governance.ts`'s `getJson` and `graphql.ts`'s `githubGraphql` call it for the controller/timeout/merge, keeping their own response-shaping (the `204`/`json().catch` handling in `getJson`, the GraphQL partial-data handling) on top. Behavior-preserving: the timeout values (`TIMEOUT_API_MS`/`TIMEOUT_MS`) stay per-module; only the controller plumbing is shared.

## 3. Org/user repo listing + `GhRepo` normalization duplicated between `list.ts` and `discover.ts`
- **Severity**: Medium
- **Category**: duplication
- **File**: src/lib/github/discover.ts:55-64 + 96-106, src/lib/github/list.ts:18-29 + 87-96 + 126
- **Scenario**: Both modules define a near-identical local `GhRepo` shape for a `/…/repos` response and both normalize it the same way: drop forks and archived repos (`if (r.fork || r.archived) continue` / `.filter((r) => !r.fork && !r.archived)`), then map to a typed item (`map(r)` in `list.ts`, the `.map` in `fetchUserRepos`). `discover.ts:88` even documents the coupling: "Forks/archived repos are dropped … mirroring the public listing in github/list.ts." The `GhRepo` interfaces differ only in fields each module happens to read (`discover` adds `owner.type`/`pushed_at` nullable; `list` adds `stargazers_count`/`description`).
- **Root cause**: `discover.ts` was added for post-OAuth auto-discovery after `list.ts` already existed; rather than reuse `list.ts`'s normalizer (which lists by `org`/`user` path, not the `/user/repos` viewer path discovery needs), the author re-derived the fork/archived filter and a parallel `GhRepo` type and left a comment acknowledging the mirror.
- **Impact**: The "what counts as a listable repo" rule (exclude forks + archived) is the product-level definition of a scannable repo and now lives in two places that must stay in sync — if one later also excludes, say, disabled or template repos, the two listing surfaces silently diverge. Two `GhRepo` interfaces describing the same upstream payload also invite field drift.
- **Fix sketch**: Extract the shared upstream row type (the union of fields both read) and a single `isListableRepo(r) => !r.fork && !r.archived` predicate into a small shared spot (e.g. `host.ts` or a `repoList.ts`), and have both `list.ts`'s loop and `discover.ts`'s `fetchUserRepos` filter through it. Keep the two public mappers (`OrgRepoListItem` vs `UserRepo`) since their output shapes legitimately differ. Behavior-preserving: identical filtering; only the predicate/type are centralized.

## 4. Dead export `teamOrgName`
- **Severity**: Medium
- **Category**: dead-code
- **File**: src/lib/github/codeowners.ts:83-85
- **Scenario**: `teamOrgName(slug)` is exported alongside `teamDisplayName(slug)` as a "display helper for a `@org/team` slug," but it has no production caller. A repo-wide grep finds it only at its own definition and in `codeowners.test.ts:97-100`; its sibling `teamDisplayName` IS used (imported by `src/lib/db/org-teams.ts:12` and called at line 229). There is no `index.ts` barrel in `src/lib/github/` that could re-export it, and no dynamic/string-keyed access.
- **Root cause**: The pair was added speculatively when CODEOWNERS team attribution was built — the rollup display only ever needed the team name (`teamDisplayName`), so the org-name half was never wired up. Its test exists only because it was authored next to the used helper.
- **Impact**: Low but real: a public API-surface item that maintainers must read and account for, plus a test that gives false "covered" credit to code nothing depends on. Misleads anyone reasoning about the module's actual contract.
- **Fix sketch**: Delete `teamOrgName` (codeowners.ts:83-85) and its assertion in `codeowners.test.ts` (the `teamDisplayName / teamOrgName` block at lines 97-101 — narrow it to `teamDisplayName` only). No other call sites to update. If the org-name split is wanted for an imminent feature, leave it but mark it intentionally — current state reads as forgotten cruft. Behavior-preserving (the symbol is unreferenced in product code).
