# Code Refactor — Data Retention & Purge
> Context group: Data & Persistence
> Total: 3 findings (Critical: 0, High: 1, Medium: 1, Low: 1)

This context is mostly clean and disciplined: every public export (`purgeExpiredData`, `envRetentionDefaults`, `resolveRetention`, `clampBatchSize`, `PURGE_ACTION`, `RETENTION_DEFAULT_BATCH_SIZE`, `RetentionPolicy`, `OrgPurgeResult`, `PurgeSummary`) is re-exported through the `@/lib/db` barrel (`src/lib/db/index.ts:47-57`) — the established public-API convention for this layer — and is exercised by the colocated test. None are dead (the seemingly "test-only" exports are part of the deliberately published surface, so they are intentionally NOT flagged). The route handler's `console.warn`/`console.error` are legitimate operational logging, not debug cruft, and there is no commented-out code or stale TODO in any of the three files. The findings below are duplication of DB-resilience plumbing that has already drifted away from the shared, better-tested implementation the rest of the `db/` layer uses.

## 1. retention.ts re-implements `withRetry` + serialization-conflict classification that already exists (and is better) in `db/client.ts` — and its copy has drifted to miss the DSQL conflict signal
- **Severity**: High
- **Category**: duplication
- **File**: src/lib/db/retention.ts:81-109 (`isSerializationConflict` + `withRetry`); collides with src/lib/db/client.ts:155-172 (`isSerializationConflictError`) and 256-280 (`withRetry`)
- **Scenario**: `retention.ts` defines a private `isSerializationConflict()` (lines 81-92) and a private `withRetry()` (lines 96-109) to retry batch deletes on DSQL optimistic-concurrency conflicts. The exact same two concerns are already implemented and EXPORTED from `db/client.ts` and re-exported via `@/lib/db` (`index.ts:5` `withRetry`, `index.ts:9` `isSerializationConflictError`). Every other DB module that needs OCC retries imports the shared one — `credits.ts:10`, `public-scan-quota.ts:23` (the latter is even called by this same purge job). `retention.ts` is the lone module that re-rolls its own.
- **Root cause**: The retention module was written self-contained (its header advertises "DSQL-safe by design") and inlined a minimal retry rather than reaching for the shared client helper — most likely authored before, or in parallel with, the consolidation that produced `client.ts`'s exported `withRetry`. The two copies then diverged.
- **Impact**: This is the dangerous kind of duplication — the two copies have already drifted, and the local copy is the inferior one on the exact production target (Aurora DSQL):
  - `client.ts`'s `isSerializationConflictError` recognizes DSQL's native `OC###` conflict codes (e.g. `OC000`/`OC001`), the Postgres `40P01` deadlock SQLSTATE, and several more conflict messages. `retention.ts`'s `isSerializationConflict` recognizes only `P2034` + the substrings `40001`/`serializ`/`write conflict`/`deadlock`. A DSQL conflict surfaced purely as an `OC000` *code* (no matching message) is therefore NOT retried by the purge job, so a destructive batch delete can abort on a routine OCC conflict that the rest of the codebase shrugs off — directly undermining the "DSQL-safe" promise in this file's own header (lines 16-20).
  - `client.ts`'s `withRetry` uses AWS-recommended exponential backoff with FULL JITTER (so a herd of conflicting batch deletes spreads out); `retention.ts`'s uses fixed linear backoff (`50 * (i + 1)`) with no jitter, re-colliding in lockstep.
  - Two implementations of the same safety-critical retry path mean every future DSQL-conflict refinement has to be made (and tested) twice, and the better-tested one (`client.test.ts:402-525` covers `OC###`, jitter, the auth-vs-OCC boundary) doesn't protect this caller.
- **Fix sketch**: Delete `retention.ts`'s private `isSerializationConflict` (81-92) and `withRetry` (96-109). Import the shared pair instead — extend line 23 to `import { getPrisma, isDbConfigured, withRetry } from "@/lib/db/client";` and drop the bespoke classifier entirely (the shared `withRetry` already defaults `isRetryable` to `isSerializationConflictError`). The two call sites (lines 145 and 181) call `withRetry(() => …)` with no second arg, which is signature-compatible with the shared `withRetry(fn, opts = {})`; optionally pass `{ label: "retention.prune" }` for clearer retry logs. After removal, `Prisma` is used only as a *type* (`Prisma.AuditLogWhereInput`, line 172) — change line 22 to `import type { Prisma } from "@prisma/client";`. Behavior-preserving for the local-Postgres test path (the shared classifier is a strict superset on the patterns retention already matched) and strictly safer on DSQL.

## 2. Two near-identical "page IDs → delete-by-id batch → stop on short page" loops in the same file
- **Severity**: Medium
- **Category**: duplication
- **File**: src/lib/db/retention.ts:131-165 (`pruneRepoScans` loop) and 176-184 (`pruneAudit` loop)
- **Scenario**: Both helpers implement the same DSQL-friendly batched-delete skeleton: `findMany({ orderBy, take: batchSize, select: { id: true } })` → map to ids → if empty `break` → `withRetry(() => deleteMany by id)` → accumulate count → `break` when the page was short. `pruneAudit` (176-184) is essentially the generic form; `pruneRepoScans`'s loop (131-165) is the same shape with a `skip: max` window and a multi-table sub-graph delete in place of the single `deleteMany`.
- **Root cause**: The audit sweep and the scan-graph prune were written separately (different tables, different child handling), so the shared paging/termination control flow was copied rather than factored out.
- **Impact**: Moderate. The termination guard is subtly different between the two (`pruneRepoScans` also breaks on `counts.sc === 0` to avoid an infinite loop when a delete removes nothing — a safety check `pruneAudit` doesn't need because it has no `skip`), so a future maintainer fixing or tuning the batch loop has to remember there are two of them with different stop conditions. Low bug-risk today, but it's latent drift surface in the most destructive code in the module.
- **Fix sketch**: Extract a small higher-order paging helper, e.g. `async function deleteInPages(selectIds: () => Promise<string[]>, deleteByIds: (ids: string[]) => Promise<number>, batchSize: number): Promise<number>` that owns the loop + the short-page/empty-page termination, and have both callers supply only their `selectIds`/`deleteByIds` closures (the scan path's closure wraps the `$transaction` sub-graph delete; the audit path's wraps the single `deleteMany`). Keep `pruneRepoScans`'s `counts.sc === 0` zero-progress guard inside the helper (harmless for the audit path, which always makes progress when it deletes a non-empty page). Behavior-preserving; consolidate only after finding #1 so the helper uses the single shared `withRetry`.

## 3. Local `sleep` helper duplicates `client.ts`'s `defaultSleep` and exists only to feed the duplicated `withRetry`
- **Severity**: Low
- **Category**: dead-code
- **File**: src/lib/db/retention.ts:94 (`const sleep = …`)
- **Scenario**: `const sleep = (ms) => new Promise((r) => setTimeout(r, ms));` is defined solely so the private `withRetry` (finding #1) can back off. `client.ts:244-245` already has the identical `defaultSleep`.
- **Root cause**: Came along for the ride with the inlined `withRetry`.
- **Impact**: Cosmetic — one trivial duplicated helper. It has exactly one caller (the private `withRetry` on line 105); once that is removed, `sleep` becomes unreferenced dead code.
- **Fix sketch**: No standalone action needed — applying finding #1 (deleting the private `withRetry`) removes the only caller of `sleep`, so delete line 94 in the same change. Listed separately only to make the dead line explicit so it isn't left orphaned.
