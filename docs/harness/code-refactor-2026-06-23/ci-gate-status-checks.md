## 1. `evaluateGate` and `evaluateGateLite` duplicate the entire failure-building rule set
- **Severity**: High
- **Category**: duplication
- **File**: src/lib/scoring/gate.ts:118-246
- **Scenario**: `evaluateGate(report, policy)` (lines 119-178) and `evaluateGateLite(snap, policy)` (lines 199-246) implement the SAME five gate rules — `minLevel`, `minOverall`, `minDimension` floor sweep, `minDimensionFor` per-dimension floor sweep (including the "skip dims already failed by the global min" guard and the fail-closed `Number.isFinite` unscored-vs-scored message split), `forbidPostures`, and the readable-gated `requireProtectedBranch` check. The two copies differ only in (a) field shape (`report.dimensions`/`x.score`/`d.id`/`d.name`/`report.posture.id` vs `snap.dims`/`x.score`/`d.dimId`/`dimName(d.dimId)`/`snap.posture`) and (b) where the dimension display name comes from (the report carries `d.name`; the lite path resolves it via `DIMENSION_BY_ID[...]?.name ?? id`). The `minDimensionFor` block alone is ~16 lines copied twice (lines 146-162 vs 220-236), down to the identical `alreadyFailed` comment.
- **Root cause**: The lite variant was added later to gate a cheap org rollup snapshot without re-scanning (header comment lines 195-198), and the author copy-edited `evaluateGate` rather than factoring the rule core out of the two shapes. The module's own comment at line 196 asserts the lite path runs "the SAME rules as evaluateGate(), so the dashboard's fleet status and the CI gate agree" — which is currently true only by manual synchronization.
- **Impact**: This is a live bug-source: any change to a gate rule (a new failure code, a tweak to the unscored fail-closed wording, a change to the `alreadyFailed` de-dup, a future posture besides "ungoverned") must be made in two places or the fleet dashboard and the merge-blocking CI gate silently disagree — exactly the divergence the comment promises cannot happen. It is also ~50 lines of near-identical maintenance surface.
- **Fix sketch**: Behavior-preserving extraction. Define one internal evaluator over a normalized shape, e.g. `evaluateNormalized({ level: number, overall: number, posture: string, dims: { id: string; name: string; score: number }[], protected?: boolean; govReadable?: boolean }, policy)` that returns `GateFailure[]`. Have `evaluateGate` map a `ScanReport` into it (`levelNum(report.level.id)`, `report.dimensions.map(d => ({ id: d.id, name: d.name, score: d.score }))`, `report.posture.id`, `report.governance`) and `evaluateGateLite` map a `GateSnapshot` into it (parsing `snap.level` via the existing `Number(snap.level.replace(/^L/i,""))||0`, filling `name` from `DIMENSION_BY_ID[id]?.name ?? id`). Both wrappers keep their current signatures and return `{ pass, policy, failures }`, so all callers (`route.ts`, `governance.ts`, both test files) are untouched. The two governance message strings differ slightly today (`evaluateGate` names the branch; lite says "Default branch") — preserve each by letting the wrapper pass its own message, or accept the unified text (verify against `governance.test.ts:187`).

## 2. The "effective floor = max(global min, per-dim floor)" helper is re-implemented in three places
- **Severity**: Medium
- **Category**: duplication
- **File**: src/lib/scoring/gate-comment.ts:88-89
- **Scenario**: `gate-comment.ts` defines `floorFor(dimId) = Math.max(gate.policy.minDimension ?? 0, gate.policy.minDimensionFor?.[dimId] ?? 0)` to decide which dimensions to show in the "Where the score falls short" table. The identical idea is re-implemented in `org/governance.ts:123` (`Math.max(policy.minDimension ?? 0, policy.minDimensionFor?.[dimId] ?? 0)`) for the green-path gap math, and the same precedence is implicit a third time inside `evaluateGate`/`evaluateGateLite` (the `minDimension` sweep plus the `minDimensionFor` sweep with the `alreadyFailed` guard). The copies have already drifted in a subtle way: `gate-comment.ts` includes unscored dims via `!Number.isFinite(d.score) || d.score < floorFor(d.id)` (line 93), whereas `governance.ts:152` uses a plain `d.score < floorFor(d.dimId)` and would silently drop an unscored dimension — the exact fail-closed case finding #1 / gate.ts works to preserve.
- **Root cause**: The per-dimension effective-floor concept is a genuine cross-cutting policy primitive, but it was never named as a shared function; each consumer (PR comment renderer, fleet green-path, the evaluator itself) inlined its own copy as it was written.
- **Impact**: Three subtly different notions of "is this dimension below its effective floor" mean the PR comment table, the dashboard green-path, and the gate verdict can disagree about which dimensions are failing — e.g. on unscored dims they already do. Future floor-precedence changes must be hunted down in three spots.
- **Fix sketch**: Export a single helper from `gate.ts` next to `belowFloor`, e.g. `export function effectiveFloor(policy: GatePolicy, dimId: string): number { return Math.max(policy.minDimension ?? 0, policy.minDimensionFor?.[dimId as DimensionId] ?? 0); }`, and consider pairing it with the existing `belowFloor` as `failsFloor(policy, dimId, score)`. Replace the inline `floorFor` in `gate-comment.ts:88-89` (in scope) with it; in a follow-up, point `governance.ts:123` at the same export (out of scope for this context but the consolidation target). This is behavior-preserving for `gate-comment.ts`; aligning `governance.ts`'s finite-check is a deliberate bug-fix to do under that context, not silently here.

## 3. Local `signed` helper duplicates an existing shared formatter
- **Severity**: Low
- **Category**: duplication
- **File**: src/lib/scoring/gate-comment.ts:25
- **Scenario**: `const signed = (n: number) => (n > 0 ? `+${n}` : `${n}`);` is declared locally and used only in `deltaPhrase` (line 39). The exact same one-liner already exists as `signed` in `src/lib/report/compare.ts:173` and `src/components/org/plan/Simulator.tsx:44`, and an equivalent `signedDelta` is exported from `src/components/ui/format.ts:14` (`(d) => `${d > 0 ? "+" : ""}${d}``).
- **Root cause**: A trivial formatter copied inline at each call site rather than imported, because it is shorter to retype than to locate the shared one.
- **Impact**: Cosmetic — four copies of a signed-number formatter. No correctness risk today; minor churn/inconsistency surface (the `ui/format.ts` version is the de-facto shared one but isn't used here).
- **Fix sketch**: Import `signedDelta` from `@/lib/...` (note: `format.ts` currently sits under `components/ui`, a client path — confirm it is import-safe from this server module before reusing it; if not, the lighter move is to leave gate-comment's copy and instead centralize all four into one util in a follow-up). Lowest-risk in-scope action: replace `gate-comment.ts:25`'s local `signed` with the shared formatter only if a server-safe shared util exists; otherwise leave as-is and note it. Behavior-preserving either way (both render `+8` / `-3` identically for integer deltas).
