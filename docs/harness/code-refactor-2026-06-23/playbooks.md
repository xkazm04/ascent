# Code Refactor — Playbooks
> Context group: Org Planning & Execution
> Total: 3 findings (Critical: 0, High: 1, Medium: 2, Low: 0)

The Playbooks context is in good shape. The pure brief/template helpers are single-sourced and well tested (`playbook-brief.ts` + its byte-exact test, `playbook-templates.ts` derived from `PRACTICES`), every export is referenced (`getPlaybook`, `getPlaybookAdoption`, `PlaybookAdoption`, `getPlaybookOrgSlug` all consumed by routes/pages/tests), and there is no dead code, commented-out code, stale TODO, or stray debug logging in scope (the one `console.error` in `apply/route.ts:105` is a deliberate 500-path log). The findings below are all duplication / redundant-work cleanups in the route layer.

## 1. `isDimId` dimension-id validator copy-pasted across org route handlers
- **Severity**: High
- **Category**: duplication
- **File**: src/app/api/org/playbooks/route.ts:13; src/app/api/org/playbooks/[id]/route.ts:15 (siblings: src/app/api/org/initiatives/route.ts:13, src/app/api/org/simulate/route.ts:17)
- **Scenario**: The exact validator `const isDimId = (v: string) => /^D[1-9]$/.test(v);` is declared independently in both in-scope route files, and two obvious siblings declare the same regex (the sibling variants additionally narrow to `v is DimensionId`). There is no shared dim-id validator in `@/lib` — the rest of the codebase validates dimension ids ad hoc via `VALID_DIM_IDS` (`lib/llm/provider.ts`) or `DIMENSION_BY_ID` (`lib/maturity/model.ts`), so this `/^D[1-9]$/` literal is a fourth, parallel definition of "what is a valid dimension id".
- **Root cause**: Each org route was authored independently and needed the same cheap guard; with no canonical `isDimensionId` helper next to `DimensionId` in `@/lib/types` / `@/lib/maturity`, copy-paste was the path of least resistance.
- **Impact**: Four sources of truth for the dimension-id range. `DimensionId` is currently `D1..D9`; if the rubric ever adds `D10` (or drops a dimension) the regex `[1-9]` silently rejects valid ids in some handlers and not others, and the bug surfaces inconsistently per endpoint. It also obscures that the type-narrowing form (`v is DimensionId`) is the more correct one — the playbooks copies return plain `boolean` and then pass the unnarrowed string on to the DB.
- **Fix sketch**: Add one canonical guard beside the type, e.g. `export const isDimensionId = (v: string): v is DimensionId => /^D[1-9]$/.test(v);` in `src/lib/maturity/model.ts` (or `src/lib/types.ts`). Replace the local `isDimId` in both playbooks routes (and, opportunistically, the `initiatives`/`simulate` siblings) with an import of it; delete the four local declarations. Behavior-preserving — same regex, and callers already treat a true result as "this is a `DimensionId`".

## 2. Resolve-org-from-playbook-then-authorize block triplicated across the per-row routes
- **Severity**: Medium
- **Category**: duplication
- **File**: src/app/api/org/playbooks/[id]/route.ts:17-22; src/app/api/org/playbooks/[id]/repos/route.ts:14-22; src/app/api/org/playbooks/[id]/apply/route.ts:39-44
- **Scenario**: Three sibling per-row routes each reimplement the same gate: `getPlaybookOrgSlug(id)` → return 404 `"Playbook not found."` when null → `requireOrgAccess(org)` and return on denial. `[id]/route.ts` wraps it in a local `gate()`, `repos/route.ts` wraps it in a local `resolveAndGate()` (which also returns the resolved org), and `apply/route.ts` inlines the identical steps. All three share the same not-found string and the same member-level default.
- **Root cause**: The per-row authorization pattern ("the org is resolved from the playbook, then authorized") was established once and then re-typed into each new route file as the feature grew (mark-applied, then PR-apply), rather than extracted to a shared helper. The `apply` route additionally needs an `installId`/`getPlaybook` afterward, so it inlined instead of reusing `repos`' helper.
- **Impact**: The 404 message, the role default, and the resolve-then-gate ordering must be kept in sync by hand across three files; the `repos` route already drifted to return the org (because it needs it later) while `[id]` discards it. A future change to the not-found contract or to how a playbook's org is resolved has to be applied in three places, and a missed one is a silent authz/consistency gap on a tenant boundary.
- **Fix sketch**: Extract one helper, e.g. `async function resolvePlaybookOrg(id, min: OrgRole = "member"): Promise<{ org: string } | NextResponse>` in a small shared module (e.g. `src/app/api/org/playbooks/_gate.ts` or `src/lib/org/playbook-gate.ts`), encapsulating the db-configured check + `getPlaybookOrgSlug` + 404 + `requireOrgAccess`/`requireOrgRole`. Have all three routes call it: `[id]/route.ts`'s `gate()` becomes a thin wrapper (or is replaced), `repos`' `resolveAndGate()` is deleted in favor of it, and `apply` replaces its inline block (lines 39-44) with a call. Behavior-preserving as long as the helper returns the resolved org so `apply`/`repos` keep their downstream use.

## 3. PATCH re-fetches the playbook's org via a second `getPlaybookOrgSlug` round-trip
- **Severity**: Medium
- **Category**: cleanup
- **File**: src/app/api/org/playbooks/[id]/route.ts:17-22,47-49
- **Scenario**: In `PATCH`, `gate(id)` (line 26) already calls `getPlaybookOrgSlug(id)` at line 19 to authorize, but throws the resolved slug away (it returns only `NextResponse | null`). Then for the audit record at line 47 the handler calls `getPlaybookOrgSlug(id)` a second time to recover the same org, and even re-handles the null case (`org ? … : undefined`) that the gate has already proven cannot happen on this path.
- **Root cause**: `gate()` was designed to return a yes/no (`NextResponse | null`) for terse early-return ergonomics, so the org it resolves is not surfaced to the caller; the later audit code (PLAY-6) was added afterward and needed the org again, so it simply re-queried.
- **Impact**: An extra DB round-trip on every successful playbook edit, plus dead defensive branching (`org ?`) for a state the gate has excluded — minor latency, and a small correctness smell (a TOCTOU window where the second lookup could return null/404-worthy after the gate passed, currently swallowed to `undefined`). It also muddies the read of the handler.
- **Fix sketch**: Have the gate return the resolved org (this folds naturally into the finding #2 helper: `{ org } | NextResponse`), then reuse that `org` for the `getOrgId(org.toLowerCase())` audit lookup at lines 48-49 and drop the second `getPlaybookOrgSlug(id)` call and its `org ?` guard. Behavior-preserving: same audit payload, one fewer query, no functional null branch lost (the gate already guaranteed a non-null org).
