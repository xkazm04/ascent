# Code Refactor — Live War Room
> Context group: Org Planning & Execution
> Total: 3 findings (Critical: 0, High: 2, Medium: 1, Low: 0)

This context is in good shape: the pure fold/classify logic is cleanly extracted into `liveWarRoomFold.ts` / `liveWarRoomShared.ts` with thorough tests, no stray `console.log`/commented-out code, and no unused imports. The findings below are two genuine duplications (both drift-risks for a projected wall) and one dead data field threaded through the celebration path. A `daysUntil` helper exists in both `LiveWarRoomHeader.tsx` and `lib/db/org-insights.ts`, but they have different signatures (`string|null` clock-reading + `Math.ceil` vs `(Date, Date)` UTC-date-only + `Math.round`) and each has its own test — deliberately distinct, so it is **not** flagged.

## 1. `POSTURE_ORDER` is duplicated across `liveWarRoomShared.ts` and `org/ui.tsx`
- **Severity**: High
- **Category**: duplication
- **File**: src/components/org/liveWarRoomShared.ts:57 (duplicate of src/components/org/ui.tsx:19)
- **Scenario**: `liveWarRoomShared.ts:57` declares `export const POSTURE_ORDER = ["ai-native", "ungoverned", "manual", "early"]`, a byte-for-byte copy of the canonical `POSTURE_ORDER` already exported from `src/components/org/ui.tsx:19`. The war-room consumer (`LiveWarRoomPanels.tsx:1,3`) already imports `POSTURE_LABEL` from `ui.tsx` while importing `POSTURE_ORDER` (and `postureBarPct`, which itself depends on `POSTURE_ORDER`) from `liveWarRoomShared.ts` — so the same component pulls the posture taxonomy from two files.
- **Root cause**: The war-room wall was built as a self-contained slice and re-declared the posture quadrant order locally rather than reaching for the existing org-ui constant, even though it was already importing the sibling `POSTURE_LABEL` from that same module.
- **Impact**: Two sources of truth for the posture quadrant set. If a posture is added/renamed/reordered (e.g. a 5th quadrant, or `ungoverned` re-spelled), one copy will be updated and the other silently left behind — `postureBarPct`'s denominator and the dashboard's bar order would diverge, and the projected war-room wall would mis-bucket or omit a posture in front of an audience. This is exactly the "two copies that can drift" class.
- **Fix sketch**: Delete `POSTURE_ORDER` from `liveWarRoomShared.ts:57` and re-export or import the canonical one from `@/components/org/ui`. Either (a) in `liveWarRoomShared.ts` add `export { POSTURE_ORDER } from "@/components/org/ui";` so existing importers (`LiveWarRoomPanels.tsx`, and `postureBarPct`'s internal use at line 67) keep resolving unchanged; or (b) point `LiveWarRoomPanels.tsx`'s `POSTURE_ORDER` import at `@/components/org/ui` and have `postureBarPct` import it from `ui`. Option (a) is the smaller, behavior-preserving change. `POSTURE_HEX` (war-room-only color map) is genuinely local and should stay in `liveWarRoomShared.ts`.

## 2. Identical `LiveRepoSeed` rollup→seed mapping copied in both entry pages
- **Severity**: High
- **Category**: duplication
- **File**: src/app/org/[slug]/live/page.tsx:41-49 (duplicate of src/app/live/shared/[token]/page.tsx:36-44)
- **Scenario**: Both the authenticated wall (`org/[slug]/live/page.tsx`) and the shared/kiosk wall (`live/shared/[token]/page.tsx`) build the `LiveRepoSeed[]` from an org rollup with the same six-field projection: `{ fullName: r.fullName, name: r.name, overall: r.latest?.overall ?? null, adoption: r.latest?.adoption ?? null, rigor: r.latest?.rigor ?? null, level: r.latest?.level ?? null, posture: r.latest?.posture ?? null }`. A repo-wide grep confirms this exact `r.latest?.*`→seed block exists only in these two files, with no shared helper.
- **Root cause**: The shared-link page (WAR-4) was added later and copy-pasted the seed-derivation from the original live page rather than extracting a helper, since `LiveRepoSeed` itself was the only shared piece factored out.
- **Impact**: The seed shape is the contract between the server rollup and the client wall. If a field is added to `LiveRepoSeed` (e.g. a new axis to display) or the `?? null` fallback rule changes, one page can be updated and the other missed — leaving the shared/TV wall seeded differently from the authenticated wall (e.g. a new tile blank only on the kiosk view). Low bytes, but a real correctness-divergence surface for the exact "show it on a TV" use case.
- **Fix sketch**: Add one pure helper next to the shared types — e.g. in `liveWarRoomShared.ts`, `export function toLiveSeed(repos: { fullName: string; name: string; latest?: { overall?: number|null; adoption?: number|null; rigor?: number|null; level?: string|null; posture?: string|null } | null }[]): LiveRepoSeed[]` returning the mapped array — and replace both `rollup.repos.map((r) => ({ … }))` blocks with `toLiveSeed(rollup.repos)`. Both callers already import from `@/components/org/LiveWarRoom` / `liveWarRoomShared`, so no new import path is introduced. Behavior-preserving (same projection, same `?? null` fallbacks).

## 3. Dead `Celebration.level` — set and tested, never rendered
- **Severity**: Medium
- **Category**: dead-code
- **File**: src/components/org/liveWarRoomShared.ts:31-36, src/components/org/liveWarRoomFold.ts:89
- **Scenario**: The `Celebration` interface (`liveWarRoomShared.ts:33`) carries a `level: string | null` field, and `foldRepoEvent` populates it (`liveWarRoomFold.ts:89`: `{ id, name, level, overall }`). But the sole consumer, `LiveWarRoomCelebrations.tsx`, renders only `c.name` and `c.overall` — a grep for `.level` in both `LiveWarRoomCelebrations.tsx` and `LiveWarRoom.tsx` returns no reads. The field is write-only: computed and stored on every AI-Native crossing but never displayed.
- **Root cause**: The celebration toast originally planned to show the achieved maturity level (it sits right alongside `overall`), but the final card design dropped the level chip while the data plumbing — interface field, fold assignment, and a test assertion — stayed.
- **Impact**: Misleading surface area: a maintainer reading the `Celebration` type reasonably assumes the level is shown, and may waste effort "fixing" a level that no UI consumes. Minor type/payload bloat on the celebration path. Not a bug, but dead threaded state.
- **Fix sketch**: Drop `level` from the `Celebration` interface (`liveWarRoomShared.ts:33`) and from the object built in `foldRepoEvent` (`liveWarRoomFold.ts:89` → `{ id, name, overall }`). Update the two test assertions that pin it (`liveWarRoomFold.test.ts:69` `toEqual({ id: 9, name: "api", level: "L5", overall: 90 })` → drop `level`). No runtime consumer reads it, so this is behavior-preserving for the rendered wall. If the level chip is instead *wanted* on the toast, the cheaper resolution is to render `c.level` in `LiveWarRoomCelebrations.tsx` — but as the code stands today it is dead.
