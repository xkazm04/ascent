# Code Refactor — Org Import, Scan & Watchlist
> Context group: Org Scanning & Fleet Rollups
> Total: 4 findings (Critical: 0, High: 2, Medium: 2, Low: 0)

The fleet-scan engine is well-factored at the data layer (`org.ts` is a clean barrel; `pool.ts` is a single shared primitive already adopted by all three scan paths). The cruft is concentrated where the three scan entry points (`/api/org/import`, `/api/org/scan`, `/api/cron/rescan`) each re-implement the same SSE plumbing and the same money-handling loop inline, and in a schedule-vocabulary literal copy-pasted across the whole feature. One small dead export remains from the cron-claim refactor.

## 1. The per-repo credit reserve → scan → refund loop is copy-pasted across all three scan entry points
- **Severity**: High
- **Category**: duplication
- **File**: src/app/api/org/import/route.ts:196-258 ; src/app/api/org/scan/route.ts:114-173 ; src/app/api/cron/rescan/route.ts:79-159
- **Scenario**: The credit-handling core of every fleet scan is reproduced three times, near-identically: (a) the optimistic up-front balance slice + `insufficient_credits` notice; (b) the per-repo `consumeScanCredit(...).catch(() => null)` reservation with the `!res || (!res.unlimited && !res.ok)` skip; (c) `reserved = res.charged` + `if (reserved) await maybeAlertLowCredits(org, res.balance)`; (d) the `refundCredit` closure (`grantCredits(org, 1, { reason: "refund", actor: "system" })`); (e) the `if (report.engine.provider === "mock" || persisted?.deduped) await refundCredit()` policy; and (f) the identical `console.warn("[<route>] persisted with partial write failures", { repo, scanId, failures })` block. The only real differences are the per-repo `scanRepository` options and the SSE-vs-counter way each surfaces progress.
- **Root cause**: The refund/dedup policy was added once, then the pattern was hand-propagated to the second and third caller as each scan path was built (import funnel, then dashboard bulk scan, then cron). Each route was edited in place rather than extracting the shared money loop, so the invariant now lives in triplicate.
- **Impact**: This is the highest-risk duplication in the context because it is the billing-correctness path. The three copies must stay byte-for-byte in lockstep — the comments even say "Mirrors the cron rescan's refund policy" — yet a future change to the refund rule (e.g. also refunding a partial-write failure, or changing the alert threshold) has to be applied in three places or the copies silently drift and an org gets mis-billed. The comment cross-references are themselves evidence the author knows these are meant to be one rule.
- **Fix sketch**: Extract a shared helper, e.g. `src/lib/scan-credit.ts` exporting `reserveScanCredit(org, repoFullName, { metered, unlimited })` → `{ reserved, skip }` (wrapping the catch + skip predicate + `maybeAlertLowCredits`) and `refundIfUnbillable(reserved, report, persisted, org)` (the mock/deduped + `grantCredits` policy), plus a `logPartialWrites(tag, repo, persisted)` for the `console.warn` block. Replace the three inline copies with calls. Behavior-preserving: the extracted functions are exactly the current bodies; callers keep their own per-route progress/SSE/counter emission (which legitimately differs). Update the three route files only.

## 2. Server-side SSE stream plumbing (`send` closure + identical response headers) is duplicated per route
- **Severity**: High
- **Category**: duplication
- **File**: src/app/api/org/import/route.ts:143-152,269-275 ; src/app/api/org/scan/route.ts:74-83,183-189 (sibling: src/app/api/scan/stream/route.ts:100-,246-248)
- **Scenario**: Both in-scope SSE routes hand-roll the exact same machinery: `new ReadableStream<Uint8Array>({ async start(controller) { const enc = new TextEncoder(); const send = (event, data) => { try { controller.enqueue(enc.encode(\`event: ${event}\ndata: ${JSON.stringify(data)}\n\n\`)); } catch {} }; … } })` and the identical response-header block `{ "content-type": "text/event-stream; charset=utf-8", "cache-control": "no-cache, no-transform", "x-accel-buffering": "no" }`. The `send` closure is character-for-character identical in `import`, `scan`, and the out-of-scope `scan/stream` route (confirmed via grep: three identical `const send = (event: string, data: unknown) =>` and three identical `x-accel-buffering` header blocks).
- **Root cause**: There is already a *client*-side SSE module (`src/lib/sse.ts`: `parseSSE`/`readSSE`) but no *server*-side counterpart, so each route author re-created the encode/enqueue/headers boilerplate by copy.
- **Impact**: Maintenance and consistency cost: the SSE wire-format (frame encoding, the swallowed-on-closed-controller `catch`, the proxy-buffering headers) is defined in three places. A fix to any of it — e.g. adding SSE keepalive comments, or correcting the `cache-control` for a CDN — must be repeated and can drift between the funnel and the dashboard stream. It also obscures each route's actual logic under ~10 lines of identical scaffolding.
- **Fix sketch**: Add a server SSE helper alongside the client one, e.g. `src/lib/sse-server.ts` exporting `sseStream(producer: (send: (event: string, data: unknown) => void) => Promise<void>): Response` that builds the `ReadableStream`, the `send` closure, the try/finally `controller.close()`, and attaches the standard headers. Each route becomes `return sseStream(async (send) => { … existing body … })`. Behavior-preserving (same frames, same headers, same close semantics). Update the two in-scope routes (and optionally the `scan/stream` sibling) to call it.

## 3. The schedule vocabulary `["off","daily","weekly","monthly"]` is redefined in four in-scope spots (and a shared export already exists)
- **Severity**: Medium
- **Category**: duplication
- **File**: src/app/api/org/import/route.ts:44 ; src/app/api/org/schedule/route.ts:13 ; src/components/org/ScheduleSelect.tsx:12 ; src/lib/db/org-watch.ts:6
- **Scenario**: The set of valid autoscan cadences is hard-coded four separate times within this context: `const SCHEDULES = new Set([...])` (import), `const VALID = new Set([...])` (schedule), `const OPTIONS = [...] as const` (ScheduleSelect, which also derives its `Schedule` type and `normalize()` from it), and the keys of `const SCHEDULE_DAYS: Record<string, number> = { off: 0, daily: 1, weekly: 7, monthly: 30 }` (org-watch). Grep shows the same literal also lives in three sibling files outside this scope (`installationRepoTypes.ts`, `InstallationRepos.tsx`, `SegmentActions.tsx`) — and crucially `installationRepoTypes.ts:22` already exports `export const SCHEDULES = ["off","daily","weekly","monthly"]`, so a canonical source partly exists and isn't being used here.
- **Root cause**: No single source of truth for the cadence enum; each layer (route validation, UI options, DB day-mapping) declared its own copy as it was written.
- **Impact**: Adding or renaming a cadence (e.g. a "biweekly" tier) requires editing 4 in-scope spots plus the sibling files, and the validators can silently disagree with the UI or the DB day-map — a value the `<select>` offers could be rejected by the route, or accepted by the route but unmapped in `SCHEDULE_DAYS` (falling through to `?? 0` → treated as "off"). It is latent drift, not a live bug today.
- **Fix sketch**: Promote one canonical definition — e.g. in `src/lib/schedule.ts` (or reuse/move the existing `installationRepoTypes.ts` `SCHEDULES`) export `export const SCAN_SCHEDULES = ["off","daily","weekly","monthly"] as const; export type ScanSchedule = (typeof SCAN_SCHEDULES)[number]; export const SCHEDULE_DAYS: Record<ScanSchedule, number> = {...}`. Then: import route's `SCHEDULES` and schedule route's `VALID` become `new Set(SCAN_SCHEDULES)`; `ScheduleSelect`'s `OPTIONS`/`Schedule`/`normalize` derive from the export; `org-watch`'s `SCHEDULE_DAYS` is the exported map. Behavior-preserving (identical values, same validation semantics).

## 4. Dead export: `advanceSchedule` is unused since the cron switched to claim-based scheduling
- **Severity**: Medium
- **Category**: dead-code
- **File**: src/lib/db/org-watch.ts:177-181 (re-exports: src/lib/db/org.ts:13 ; src/lib/db/index.ts:127)
- **Scenario**: `advanceSchedule(repoId, schedule)` — "After a SUCCESSFUL autoscan, advance the repo's next due time by its full cadence" — has no functional caller anywhere in `src/`. Grep across the whole tree returns only the definition and the two barrel re-exports (`org.ts`, `index.ts`); no route, lib, test, or script invokes it. The cron rescan path now advances the schedule as a side effect of `claimRescan` (the claim's conditional `updateMany` sets `nextScanAt` to the next cadence up front) and uses `advanceScheduleAfterFailure` for the failure backoff, so the success-path advance this function used to perform is dead — its own doc-comment on `claimRescan` even notes "a separate post-success advance is unnecessary."
- **Root cause**: Left behind by the claim-before-work refactor (cron now claims+advances atomically); the old post-success advance helper was never removed and is still carried through the barrels.
- **Impact**: Misleading: a maintainer reading `org-watch.ts` sees two schedule-advance helpers and may wire the redundant one back in, reintroducing a double-advance race the claim refactor was designed to remove. It also widens the `@/lib/db` public surface with a function nothing calls.
- **Fix sketch**: Delete the `advanceSchedule` function (org-watch.ts:177-181) and remove its line from the `org.ts` (:13) and `index.ts` (:127) re-export lists. No callers to update. Confirm with `rg -n "advanceSchedule\b" src` showing only the now-removed lines before deleting. (Keep `advanceScheduleAfterFailure` — it is live in cron/rescan.)
