# Code Refactor — GitHub OAuth & Session
> Context group: Identity & GitHub Connectivity
> Total: 4 findings (Critical: 0, High: 1, Medium: 2, Low: 1)

The context is, overall, in good shape: the `auth.ts` helpers (`safeNext`, `readableOrgForOwner`, `getActiveOrg`, `orgOptionsForSession`, `isSameOrigin`, `getSession`/`getSessionState`) are all live and reused across many routes; the custom-OAuth `GitHubSignInButton`/login/callback flow is described as "dormant" but is genuinely wired into `connect/page.tsx` (the re-sync flow), so it is NOT dead code; and the `SignInNotice` `provider` prop drives both a live `github` branch (default callers) and a live `supabase` branch (`org/[slug]/layout.tsx`). No dead exports were found. The findings below are duplication and one stale comment.

## 1. Sign-in button chrome duplicated verbatim across two components (and already drifting)
- **Severity**: High
- **Category**: duplication
- **File**: src/components/GitHubSignInButton.tsx:15-54 and src/components/SupabaseAuthButtons.tsx:15-46
- **Scenario**: `GitHubSignInButton.tsx` and `SupabaseAuthButtons.tsx` each define their own private `GitHubMark` (the full Octocat `<path d="M8 0C3.58 0…">`), `Spinner`, and the `VARIANTS: Record<Variant, …>` style map. The SVG path string, the spinner markup/classes, and the entire `VARIANTS` object (`primary`/`nav` box classes, icon sizes, idle/busy copy) are byte-for-byte identical between the two files. The two button bodies are also near-identical: the same `relative inline-flex` icon cross-fade, the same `role="status" aria-live="polite"` sr-only region, the same `focus-ring … gap-2 transition ${v.box}` shell and `cursor-wait opacity-70` pending styling.
- **Root cause**: `SupabaseAuthButtons.tsx` was created later to drive the now-active Supabase login wall and was deliberately "styled to match" the older custom-OAuth `GitHubSignInButton` (its header comment says so) by copy-pasting the chrome rather than extracting it. Two affordances that are required to look identical were implemented as two copies.
- **Impact**: Every visual/a11y tweak to the GitHub CTA must be made in two places or the buttons silently diverge — which has **already begun**: the Supabase button renders `aria-busy`/copy but omits `GitHubSignInButton`'s `aria-disabled={pending || undefined}` semantics and uses `v.busy` directly while the GitHub one funnels through `busyLabel`/`pendingLabel`. The duplicated Octocat path (~700 chars) also ships twice in the client bundle.
- **Fix sketch**: Extract the shared pieces into one module (e.g. `src/components/auth/signInButtonChrome.tsx` or co-locate in `GitHubSignInButton.tsx` and export): a `GitHubMark`, a `Spinner`, the `Variant` type, the `VARIANTS` map, and ideally a presentational `SignInButtonInner({ variant, pending, idleLabel, busyLabel })` that renders the icon cross-fade + label + status region. Have both `GitHubSignInButton` (anchor + `onClick` pending) and `SupabaseSignInButton` (button + `signIn()` pending) import them and supply only their distinct wrapper element and click handler. Behavior-preserving as long as the extracted markup matches the current `GitHubSignInButton` version (the canonical one); update `SupabaseAuthButtons.tsx` to consume it (this also closes the `aria-disabled` drift). No external callers change — both public exports keep their signatures.

## 2. Session-cookie attributes hand-rolled in the callback instead of the existing `sessionCookieAttrs` helper
- **Severity**: Medium
- **Category**: duplication
- **File**: src/app/api/auth/callback/route.ts:136-146 (vs src/lib/auth.ts:156-164)
- **Scenario**: `auth.ts` already defines `sessionCookieAttrs(secure)` — the single source for the session cookie's `{ httpOnly, sameSite: "lax", secure, path: "/", maxAge: sessionMaxAgeSeconds }` shape — and uses it in `getSessionState`'s silent-refresh re-mint and in `revokeOtherSessions`. The callback route, however, re-types that same attribute object inline when it sets the initial `SESSION_COOKIE` (`httpOnly: true, sameSite: "lax", secure: await secureCookieForRequest(), path: "/", maxAge: sessionMaxAgeSeconds`). So the cookie's defining attributes live in two places, one of which (`sessionMaxAgeSeconds`, the long TTL) is imported into the callback solely to re-spell the helper's body.
- **Root cause**: `sessionCookieAttrs` is a private (un-exported) helper in `auth.ts`; the callback predates it (or couldn't reach it) and so the "mint the session cookie" attribute set was written out longhand at the one place outside `auth.ts` that mints it.
- **Impact**: A future change to session-cookie policy (e.g. adding `priority`, flipping `sameSite`, changing the TTL) must be made in `sessionCookieAttrs` AND in the callback or the initial cookie diverges from every re-minted one — a subtle, security-relevant drift (the three sites are deliberately kept in lockstep today, per the inline comments about `Secure`). It is exactly the class of cookie-attribute mismatch the `delete`-with-matching-attrs comment in the login route warns about.
- **Fix sketch**: Export `sessionCookieAttrs` from `auth.ts` and replace the callback's inline options object with `sessionCookieAttrs(await secureCookieForRequest())`. Drop the now-unused `sessionMaxAgeSeconds` import from the callback (it's only referenced to build that object). Behavior-preserving: the helper produces the identical attribute set. (Leave the login route's STATE/NEXT/RESYNC cookies alone — those are short-lived `maxAge: 600` OAuth cookies, a different shape, not session cookies.)

## 3. `proxy.ts` re-implements the access-gate env logic instead of sharing a pure predicate
- **Severity**: Medium
- **Category**: duplication
- **File**: src/proxy.ts:13-23 (vs src/lib/access.ts:27-46)
- **Scenario**: `proxy.ts`'s `gateInactive()` re-derives, by hand, the exact boolean logic that already lives in `access.ts`: `Boolean(NEXT_PUBLIC_SUPABASE_URL && NEXT_PUBLIC_SUPABASE_ANON_KEY)` duplicates `supabaseAuthConfigured()`, and the `NODE_ENV !== "production" && (ASCENT_AUTH_BYPASS === "1" || === "true")` block duplicates `authBypassEnabled()`. `gateInactive()` is effectively `!authGateEnabled()`. The file's own comment acknowledges the duplication ("The env checks are inlined to match authGateEnabled()").
- **Root cause**: The proxy must use the request/response cookie adapter and cannot import `access.ts` (server-only, uses `next/headers`), so the *cookie* plumbing legitimately can't be shared — but that constraint was over-applied to the pure, header-free *env predicates*, which were copied rather than factored into a dependency-free module both sides can import.
- **Impact**: The bypass/configured rules now live in three places (`access.ts`, `proxy.ts`, and conceptually the inlined constants), so a change to how the bypass is recognized (e.g. accepting `"yes"`, or a new env var name) silently desyncs the cookie-refresh gate from the real auth gate — the proxy could keep refreshing cookies on a surface the gate considers closed, or vice-versa. The `auth.test.ts` matrix only pins `access.ts`, not the proxy copy.
- **Fix sketch**: Move the two pure predicates into a `next/headers`-free module (e.g. `src/lib/authEnv.ts`): `supabaseAuthConfigured()` and `authBypassEnabled()` reading only `process.env`. Have `access.ts` re-export / build `authGateEnabled` from them, and have `proxy.ts` import them and define `gateInactive = () => !(supabaseAuthConfigured() && !authBypassEnabled())`. This keeps the cookie-adapter code in the proxy untouched while collapsing the env logic to one definition the existing tests already cover.

## 4. Stale `BUG (github-oauth-session #1)` comment marking an already-applied fix
- **Severity**: Low
- **Category**: cleanup
- **File**: src/proxy.ts:50-54
- **Scenario**: A multi-line comment block opens with `// BUG (github-oauth-session #1): this best-effort cookie refresh must NOT be request-fatal…` and describes a defect (un-guarded `getUser()` turning a transient Supabase hiccup into a site-wide 500). The very next lines (55-59) already implement the fix — `getUser()` is wrapped in `try { … } catch { /* degrade */ }`. So the `BUG (…)` label now annotates resolved behavior, reading as an open defect.
- **Root cause**: The fix was applied in place but the originating bug-scan annotation was left as the explanatory comment rather than being rewritten into a plain rationale.
- **Impact**: Minor maintainer confusion — a `BUG (…)` tag with a scan id implies an outstanding issue and invites a redundant "is this still broken?" investigation; it also couples the source to a transient harness finding id.
- **Fix sketch**: Reword the comment to a neutral rationale, e.g. "// This best-effort cookie refresh must not be request-fatal: the proxy runs on a broad matcher incl. public paths, so a thrown getUser() would turn a transient Supabase hiccup into a site-wide 500. Mirror getViewer()'s tolerance — treat a thrown/error result as 'no user, cookie not refreshed this request' and proceed." Drop the `BUG (github-oauth-session #1)` prefix. Pure comment edit; no behavior change.
