## 1. Refund uses "newest hit" not "the hit this request charged" â€” concurrent refunds on a coalesced scan double-refund and under-count the bucket
- **Severity**: Critical
- **Lens**: bug-hunter
- **Category**: TOCTOU / refund race / quota bypass
- **File**: src/lib/public-scan-quota.ts:259 (`removeNewestHit`) consumed by :290-293 (`refundPublicScanQuota`)
- **Scenario**: Two anonymous clients on the same IP bucket (or a StrictMode double-mount + a second tab) start an uncached scan of the same commit in the same instant. Each independently runs `consumePublicScanQuota` and records its own hit, so the window holds two fresh timestamps `[t1, t2]`. Both requests then coalesce onto ONE shared scan run (`coalesceScan`). That run degrades to mock (transient LLM 429), so BOTH requests see `degradedToMock` and call `refundQuota()` â†’ `refundPublicScanQuota`. Each refund calls `removeNewestHit`, which drops `Math.max(...hits)`. Refund A drops `t2`; refund B (racing or after) drops `t1`. Two hits removed, leaving the bucket emptier than before either scan started â€” even though only the still-live request's own slot should have been returned. The serialized transaction prevents a lost UPDATE but does NOT prevent each refund from legitimately removing a different newest hit.
- **Root cause**: The refund is identity-by-recency ("drop the newest"), not identity-by-value ("drop the timestamp THIS request recorded"). `consumePublicScanQuota` returns no handle to the exact hit it appended, so two refunds against the same bucket cannot tell each other's hit apart and each peels off one.
- **Impact**: Quota under-counts â†’ free-scan budget bypass. A scripted client that forces degrade-to-mock (or just races duplicate requests) can refund more slots than it consumed, grazing the expensive LLM funnel indefinitely. This is the same double-refund class the scan-pipeline scan already flagged, still unfixed on the quota side; the cross-module note in scan-pipeline.md:11-14 names this exact bucket.
- **Fix sketch**: Have `consumePublicScanQuota` return the exact `now` it recorded; pass it to `refundPublicScanQuota`, which removes the entry whose timestamp `=== chargedAt` (and tolerates "already absent" as success). Idempotent + value-keyed, so a second refund of the same charge is a no-op instead of stealing a sibling's hit.

## 2. `clientIp` trusts the right-most X-Forwarded-For hop, which is attacker-controlled on a direct/single-hop connection â€” per-request bucket minting bypasses both limiters
- **Severity**: High
- **Lens**: bug-hunter
- **Category**: key derivation / IP spoofing / rate-limit + quota bypass
- **File**: src/lib/rate-limit.ts:18-27 (`clientIp`), used at :67-69 and public-scan-quota.ts:164
- **Scenario**: On any deployment where the platform does NOT inject `x-real-ip` and does NOT guarantee it appends its own trusted XFF hop (e.g. a direct-to-Node connection, a misconfigured proxy, or a CDN that passes the client XFF through verbatim), an attacker sends `X-Forwarded-For: <random>` on every request. `x-real-ip` is absent, so the code falls to the XFF branch and trusts `hops[hops.length - 1]` â€” the right-most hop. If nothing trusted appended a hop, the right-most entry IS the attacker's value. A fresh random per request mints a fresh per-IP rate-limit bucket AND a fresh weekly-quota bucket every time, defeating both the burst limiter (per-IP window never fills) and the weekly soft gate (every scan is "the first" for a new IP hash).
- **Root cause**: "Right-most hop is trusted" holds only when a trusted proxy is guaranteed to have appended exactly one hop. The code has no allowlist of trusted proxy hops / no configured hop count, so it cannot distinguish "trusted proxy appended this" from "client supplied this." The module comment asserts the right-most is "trusted-proxy-appended," but that invariant is unenforced.
- **Impact**: Full bypass of the per-IP rate limit and the per-IP weekly quota on any non-`x-real-ip` platform â€” the two controls guarding the expensive free LLM funnel. The global per-instance ceiling (rate-limit.ts:68) still caps total spend, but per-IP abuse attribution is gone.
- **Fix sketch**: Require an explicit trusted-proxy configuration: a `TRUSTED_PROXY_HOPS` count (strip exactly N right-most hops, take the next) or a platform-header allowlist (`cf-connecting-ip`/`true-client-ip` when behind that CDN). When the platform's trusted client header is absent AND no hop count is configured, fall through to the shared `"unknown"` bucket (fail closed) rather than trusting any client-settable XFF value.

## 3. Per-instance in-memory rate limiter resets on every cold start â€” serverless/multi-instance makes the limiter effectively non-binding
- **Severity**: High
- **Lens**: bug-hunter
- **Category**: in-memory vs persistent state / quota bypass on deploy
- **File**: src/lib/rate-limit.ts:29 (`const windows = new Map`), :32-44 (`hit`)
- **Scenario**: `windows` is a module-global Map living only in one server instance's heap. On Next.js serverless / autoscaled deploys, (a) every cold start begins with an empty map, so the per-IP and global windows reset to zero on each new lambda/instance, and (b) concurrent requests fan out across N instances, so the effective limit is `N Ă— perIp` and `N Ă— global`. An attacker who spreads requests (or simply benefits from autoscaling under load) sails past `SCAN_RATE_LIMIT.global` (120/min/instance) because each instance counts independently. Unlike the weekly quota â€” which is deliberately Prisma-backed precisely to survive restarts (public-scan-quota.ts:7-9) â€” the burst limiter has no shared store.
- **Root cause**: The cost-control backstop for the most expensive endpoint (a GitHub ingest + an LLM completion per scan) keeps its state in volatile per-instance memory. The module comment acknowledges this is a "backstop, not a precise distributed quota," but on a fully serverless deploy the backstop can degrade to near-zero binding because windows almost never accumulate within one warm instance.
- **Impact**: On serverless, the global spend ceiling intended to cap LLM cost per minute is multiplied by the live instance count and reset by every scale event â€” real-dollar exposure on the free surface. The weekly quota (finding #1/#2 notwithstanding) is the only durable cap left.
- **Fix sketch**: Back `hit()` with a shared store (Upstash Redis `INCR`/sliding-window, already flagged in PRODUCTION_READINESS Wave 2) behind an env flag, keeping the in-memory Map as the dev/no-Redis default. The window logic ports directly; make `rateLimitRequest` async at its 4 call sites.

## 4. Refund/peek/consume compute `now`/window with `>` cutoff but a slot that ages out exactly at the boundary is treated inconsistently across consume vs peek timing
- **Severity**: Medium
- **Lens**: bug-hunter
- **Category**: window-reset edge case / header staleness
- **File**: src/lib/public-scan-quota.ts:118 (`remaining` from consume) and the refund at :276-302
- **Scenario**: `consumePublicScanQuota` returns `remaining` computed at consume time and the route bakes it into the `x-ascent-quota-remaining` header (scan/route.ts:121, stream/route.ts:77). When the scan then delivers nothing chargeable and `refundPublicScanQuota` runs, the slot is given back in the DB but the already-sent header still says the lower post-consume `remaining`. The module comment at :273-275 explicitly accepts this ("headers may overstate usage by the one refunded slot"), but combined with finding #1 a refund can return MORE than one slot's worth of accuracy drift, so the meter shown to the user (QuotaBanner, finding #5) can be off by more than one and never self-corrects until the next `peek`/scan.
- **Root cause**: `remaining` is a snapshot captured before the refund decision exists; there is no post-refund recompute and no re-emission path (headers are already flushed, especially on the streaming route). The drift is bounded to 1 only if refunds are single-slot â€” which finding #1 breaks.
- **Impact**: User-visible "scans left this week" can understate the true allowance (showing fewer than they actually have), suppressing legitimate scans. Lower severity because it never grants MORE than entitled, but it directly undermines trust in the meter.
- **Fix sketch**: After a refund, don't trust the consume-time `remaining`; have the client meter re-fetch from `GET /api/quota` (`peekPublicScanQuota`) rather than the consume header, or return the refunded `remaining` on the SSE `done` event so the banner reconciles. Fixing #1 also bounds this back to â‰¤1.
