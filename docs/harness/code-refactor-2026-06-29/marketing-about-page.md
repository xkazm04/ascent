## 2. Two parallel Remotion-diagram subsystems duplicate their wrapper and canvas constants
- **Severity**: Medium
- **Category**: duplication
- **File**: src/components/about/ChampionNetwork.tsx:1-33 vs src/components/about/RiskRadar.tsx:1-33; src/components/about/champion/graph.ts:7-10 vs src/components/about/risk/radar.ts:7-10
- **Scenario**: `ChampionNetwork` and `RiskRadar` are near-identical ~30-line wrappers: each imports `RemotionStage`, its composition, and `{ W, H, FPS, DURATION }` from its data module, then renders `<RemotionStage component={â€¦} durationInFrames={DURATION} fps={FPS} width={W} height={H} legend={<>â€¦2 swatchesâ€¦</>} />`. The only real differences are the composition, the import path, and the two-item legend. Separately, `champion/graph.ts` and `risk/radar.ts` each declare `W=960, H=540, FPS=30, DURATION=330` â€” byte-for-byte identical canvas constants.
- **Root cause**: The two diagrams were built as independent vertical slices; the shared playback envelope (canvas size, fps, single-shot duration) and the wrapper skeleton were never hoisted, even though `compositionShared.tsx` already exists precisely to single-source cross-diagram values.
- **Impact**: Adding a third diagram means copying the wrapper again; changing the canvas/fps means editing two constant blocks that must stay in lockstep (a drift would silently mis-scale one Player). The legend-swatch markup is also re-typed per diagram.
- **Fix sketch**: Move the shared `W/H/FPS/DURATION` into `compositionShared.tsx` (or a `compositionFrame.ts`) and import them in both data modules. A tiny `<LegendSwatch color line?>` helper removes the repeated swatch JSX.

## 3. Reduced-motion gating idiom is re-implemented element-by-element across three diagrams
- **Severity**: Medium
- **Category**: duplication
- **File**: src/components/about/AboutAscentSteps.tsx:26,54-58,64-68,70-76,93-99; src/components/about/FleetGrid.tsx:31,90-100,118-128; src/components/about/RoiSimulator.tsx:35,75-82
- **Scenario**: Because `MotionConfig reducedMotion="user"` only degrades transform/layout props (not `pathLength`, `cx/cy`, `left`, `flexGrow`, `width`), each interactive diagram re-codes the same manual gate: `const reduced = useReducedMotion()` then, per animated element, the four-prop ternary scaffold `initial={reduced ? false : {â€¦}} animate={reduced ? {â€¦final} : undefined} whileInView={reduced ? undefined : {â€¦}} transition={reduced ? { duration: 0 } : {â€¦}}`. AboutAscentSteps repeats this ~4Ă— (riser line, platform line, node circle, climber); FleetGrid and RoiSimulator each repeat their own variant.
- **Root cause**: No shared helper encapsulates "play this entrance unless reduced motion, in which case jump to the final state." Each author re-derived the ternary set inline, with explanatory comments duplicated nearly verbatim ("â€¦is non-transform, so reducedMotion=\"user\" doesn't degrade itâ€¦").
- **Impact**: The pattern is verbose and error-prone â€” a single missed branch silently animates for reduced-motion users (exactly the bug class the comments warn about). Maintenance means re-checking the same idiom in three files.
- **Fix sketch**: Add a helper in `components/deck` (or `compositionShared` neighbour), e.g. `gatedReveal(reduced, { initial, final, transition })` returning the correct `{ initial, animate, whileInView, transition }` set, and a `viewport` const.
